# Reviewer 口令集

> 让 AI 当审查者，而不是打字员

## 原版三个核心口令

| # | 口令 | 用途 |
|---|------|------|
| 1 | 对我这些代码变更进行严厉拷问。在我通过你的测试之前，不要创建 PR。 | 找问题 |
| 2 | 向我证明这行得通。请对比 main 分支和当前 feature 分支的行为差异，列出你验证了什么。 | 验证正确性 |
| 3 | 基于你现在掌握的所有信息，废掉目前的方案，去实现一个更优雅的解法。 | 优化方案 |

## 扩充口令集

### 安全审查类

| # | 口令 | 用途 |
|---|------|------|
| 4 | **以安全专家视角审查这段代码。列出所有可能的安全漏洞：注入攻击、越权访问、敏感数据泄露、认证绕过。** | 安全审计 |
| 5 | **假设我是攻击者，我会如何利用这段代码做坏事？请列出攻击向量和防护建议。** | 红队视角 |
| 6 | **检查所有用户输入是否经过验证和清理。找出未经验证就直接使用的输入。** | 输入验证 |
| 7 | **这段代码如果部署到公网，会有什么风险？列出最严重的 5 个。** | 部署风险 |

### 边界情况类

| # | 口令 | 用途 |
|---|------|------|
| 8 | **列出这段代码所有可能的边界情况：空值、空数组、极大值、极小值、并发、网络超时。对每种情况说明当前是否处理。** | 边界检查 |
| 9 | **如果传入非法参数会怎样？请追踪每条可能的执行路径。** | 异常路径 |
| 10 | **当依赖服务全部挂掉时，这段代码会怎么崩溃？如何让它优雅降级？** | 容错分析 |

### 性能与资源类

| # | 口令 | 用途 |
|---|------|------|
| 11 | **分析这段代码的时间复杂度和空间复杂度。有优化空间吗？** | 复杂度分析 |
| 12 | **如果数据量增长 100 倍，哪里会先崩溃？** | 扩展性检查 |
| 13 | **找出所有 N+1 查询、循环内调用、不必要的重复计算。** | 性能瓶颈 |
| 14 | **这段代码会泄漏内存或连接吗？追踪所有资源的创建和释放路径。** | 资源泄漏 |

### 可读性与维护类

| # | 口令 | 用途 |
|---|------|------|
| 15 | **假设 6 个月后一个新人来读这段代码，他会困惑什么？请列出需要解释的点。** | 可读性检查 |
| 16 | **这段代码违反了哪些 SOLID 原则？如何重构让它符合？** | 架构设计 |
| 17 | **找出所有"魔法数字"和"魔法字符串"，它们应该被命名常量替代。** | 代码规范 |
| 18 | **如果要把这段代码写成单元测试，哪些逻辑难以测试？说明原因。** | 可测试性 |

### 测试完整性类

| # | 口令 | 用途 |
|---|------|------|
| 19 | **列出这段代码需要的所有测试用例：正常路径、边界、异常、并发。** | 测试覆盖 |
| 20 | **当前的测试覆盖了哪些场景？漏掉了哪些？** | 测试缺口 |
| 21 | **给我 5 个可能让这段代码失败的测试输入。** | 失败用例 |

### 数据库与持久化类

| # | 口令 | 用途 |
|---|------|------|
| 22 | **检查所有数据库查询。找出可能的 SQL 注入、N+1 查询、缺少索引的查询。** | 数据库审查 |
| 23 | **如果有事务，列出事务边界。是否有死锁或长时间持锁的风险？** | 事务审查 |
| 24 | **数据迁移脚本是否可回滚？如果执行一半失败会发生什么？** | 迁移安全 |

### 业务逻辑类

| # | 口令 | 用途 |
|---|------|------|
| 25 | **这段代码解决了什么业务问题？用一句话概括。如果你做不到，说明业务意图不清晰。** | 业务理解 |
| 26 | **列出这段代码的所有假设（关于数据、环境、用户行为）。哪些假设可能不成立？** | 隐含假设 |
| 27 | **如果用户恶意使用这个功能，会造成什么损失？如何防御？** | 业务风控 |

### AI 特有的优势类

| # | 口令 | 用途 |
|---|------|------|
| 28 | **在代码库中搜索类似功能的实现。当前的实现与已有模式一致吗？如果不一致，说明理由。** | 模式一致性 |
| 29 | **列出这段代码引用的所有其他文件。画出调用关系图。指出可能的循环依赖。** | 依赖分析 |
| 30 | **在过去 30 天的 Git 历史 中，这个文件被改动过吗？改动原因是什么？当前改动会冲突吗？** | 历史上下文 |

## 组合使用场景

### 场景 1：核心功能上线前

```
1. 以安全专家视角审查这段代码
2. 列出所有可能的边界情况
3. 给我 5 个可能让这段代码失败的测试输入
4. 向我证明这行得通
```

### 场景 2：接手老代码

```
1. 在代码库中搜索类似功能的实现，当前与已有模式一致吗
2. 列出这段代码的所有假设
3. 如果数据量增长 100 倍，哪里会先崩溃
4. 基于你现在掌握的所有信息，实现一个更优雅的解法
```

### 场景 3：性能优化

```
1. 分析时间复杂度和空间复杂度
2. 找出所有 N+1 查询和循环内调用
3. 列出可能的性能瓶颈
4. 给出优化方案并验证
```

### 场景 4：PR 审查（最严格）

```
1. 对我这些代码变更进行严厉拷问
2. 以安全专家视角审查
3. 列出所有边界情况
4. 检查业务逻辑是否有漏洞
5. 向我证明这行得通

在我通过以上测试之前，不要创建 PR。
```

---

## 使用技巧

1. **选择对口令**：根据代码类型选择 2-3 个相关口令
2. **组合使用**：不同角度的口令组合效果更好
3. **迭代审查**：修复后再次用相同口令检查
4. **建立模板**：把常用组合存成 Skill，一键调用
