# 项目建设规范深度审查与优化建议（advice_v2）

> 审查范围：`AGENTS.md.template`、`ai_workflow.md`、`manu.md`、`STRUCTURE.md`、`phases/` 全部文件、`templates/` 全部文件。  
> 审查视角：**市场营销高手、资深产品专家、资深架构师、资深开发工程师、资深运维工程师**。

---

## 0. 总体结论

当前规范体系在“研发过程可追溯、CR 管控、阶段门禁、文档闭环”方面已经很强，属于偏工程治理型框架。  
主要不周全点集中在三类：

1. **商业与增长闭环不足**：从提案到部署流程里，缺少市场验证、GTM（Go-To-Market）、发布传播、增长指标复盘的强制链路。
2. **可执行自动化不足**：大量门禁以“文档规则+示例命令”存在，缺少统一脚本化入口，跨项目落地依赖人工水平。
3. **生产运维安全细节不足**：密钥处理、上线准备评审（ORR）、应急热修流程、SLO/值班机制等还未形成硬性规范。

---

## 1. 市场营销高手视角

### 1.1 考虑不周全点

1. **缺少市场验证前置机制**  
   现有 Proposal 更偏“需求合理性”，但缺少“市场空间、竞品对比、差异化定位、付费意愿验证”的结构化要求。

2. **缺少 GTM 交付物**  
   目前没有“发布对象、渠道、节奏、素材、传播口径”的模板化要求，部署完成后容易出现“功能上线但没人知道/不会用”。

3. **缺少增长指标闭环**  
   `status.md`/`test_report.md` 强调技术完成度，但没有“激活率、留存率、转化率、复购率、NPS”等增长指标跟踪框架。

4. **缺少发布后市场反馈回流机制**  
   规范里有 lessons learned，但没有把用户反馈、工单、运营数据强制回流到下一轮 Proposal 的机制。

### 1.2 优化建议

1. 在 `templates/proposal_template.md` 增加章节：
   - 目标市场（ICP）与典型客户画像
   - 竞品对比矩阵（至少 2 个替代方案）
   - 差异化价值主张（为什么用户会选我们）
   - 定价/商业模式假设（如适用）

2. 新增 `templates/launch_plan_template.md`（建议）
   - 发布目标与人群
   - 渠道策略（站内、邮件、社媒、私域、BD）
   - 内容素材清单（公告、FAQ、演示、销售话术）
   - 发布节奏（T-7/T-1/T+1/T+7）

3. 在 `templates/status_template.md` 增加“业务指标看板”区块（可选但推荐）
   - 北极星指标 + 2~4 个护栏指标
   - 基线值、目标值、观察周期、负责人

4. 在 `phases/07-deployment.md` 增加“发布后增长复盘触发”
   - T+7/T+30 复盘结论写入 `docs/lessons_learned.md`
   - 若关键指标未达阈值，自动触发 CR（优化/回滚/迭代）

## 2. 资深产品专家视角

### 2.1 考虑不周全点

1. **需求优先级机制不够强制**  
   `requirements_template.md` 虽有优先级概念，但缺少“版本归属、取舍依据、延期策略”的硬约束。

2. **用户研究证据链偏弱**  
   Proposal 的“背景与现状”没有强制要求证据来源（访谈、工单、行为数据、客服反馈）。

3. **体验与可用性验收不够明确**  
   当前偏功能验收，缺少可用性指标（学习成本、任务完成率、错误恢复成本）的量化条目。

4. **Minor 流程承载过于依赖 status.md**  
   简化流程下把大量内容内联到 `status.md`，长期会导致信息拥挤、追溯成本升高。

### 2.2 优化建议

1. 在 `templates/requirements_template.md` 的需求列表新增列：
   - 商业价值
   - 用户价值
   - 版本归属（本版/下版）
   - 延期策略（可延期条件）

2. 在 `templates/proposal_template.md` 增加“问题证据”区块：
   - 数据证据（指标截图/查询口径）
   - 访谈证据（用户原话摘要）
   - 业务方证据（投诉/工单/损失）

3. 在 `templates/test_report_template.md` 增加“用户体验验收”区块（如适用）
   - 关键任务完成时间
   - 关键路径错误率
   - 空状态/错误态可理解性

4. 为 Minor 流程新增轻量文档：`docs/<版本号>/mini_spec.md`
   - 变更摘要
   - 验收标准（GWT）
   - 影响面（功能/API/文档）
   - 回滚要点
   - 验证证据

## 3. 资深架构师视角

### 3.1 考虑不周全点

1. **架构决策留痕尚未制度化到 ADR**  
   `design_template.md` 有决策记录，但缺少标准 ADR（Architecture Decision Record）目录与生命周期管理。

2. **版本策略过粗，不利于补丁与热修治理**  
   `STRUCTURE.md` 仅定义 `vMAJOR.MINOR`，未覆盖常见 `PATCH` 和 `-rc`、热修分支策略。

3. **架构“可验证性”指标不足**  
   已强调可观测/安全，但缺少架构层可量化阈值（容量上限、错误预算、降级 SLA）。

4. **跨系统兼容矩阵不足**  
   API/数据迁移提到兼容，但缺少“客户端版本—API版本—数据版本”兼容矩阵模板。

### 3.2 优化建议

1. 新增 `docs/adr/ADR-XXXX-*.md` 规范（建议写入 `STRUCTURE.md`）
   - 状态：Proposed/Accepted/Superseded
   - 决策背景、备选方案、后果、回滚条件

2. 升级版本号规范为：`vMAJOR.MINOR.PATCH[-rcN]`
   - 在 `STRUCTURE.md`、`phases/00-change-management.md` 保持一致
   - 明确热修发布路径（hotfix/*）和回合并规则

3. 在 `templates/design_template.md` 增加“架构验收阈值”
   - 可用性（SLO）
   - 性能预算（P95、吞吐）
   - 错误预算（Error Budget）
   - 降级生效时间

4. 新增“兼容矩阵”模板（可并入 `design_template.md`）
   - 客户端版本 ↔ API 版本 ↔ 数据 schema 版本
   - 升级顺序与回滚影响

## 4. 资深开发工程师视角

### 4.1 考虑不周全点

1. **门禁执行偏人工，工程自动化不足**  
   多处为示例命令，缺少统一脚本入口，导致不同项目执行标准不一致。

2. **协作术语存在工具绑定**  
   部分内容出现 `Claude`、`TodoWrite` 等绑定词，跨 Agent/跨平台复用性不足。

3. **质量阈值未量化**  
   有“测试通过/自测通过”要求，但缺少最低阈值（变更覆盖率、静态检查等级、关键路径必测）。

4. **测试稳定性治理不足**  
   缺少 flaky test 处理策略、测试数据管理规范、可重复执行保障。

### 4.2 优化建议

1. 在 `.aicoding` 增加脚本目录（建议）
   - `scripts/check_refs.sh`（R6）
   - `scripts/check_diff_scope.sh`（AC-05）
   - `scripts/check_master_docs_trace.sh`（Deployment 规则2）
   - `scripts/check_cr_trace.sh`（CR-ID 提交追溯）

2. 统一术语
   - 将 `Claude` 统一为 `Agent`
   - 将 `TodoWrite` 改为“任务追踪工具（实现自选）”

3. 在 `templates/implementation_checklist_template.md` 增加量化门槛
   - 变更代码行覆盖率阈值（建议按项目自定义）
   - 关键路径必须有集成测试
   - 安全扫描（SAST/依赖漏洞）通过

4. 在 `templates/test_report_template.md` 增加“测试稳定性”区块
   - flaky 用例清单
   - 重跑策略与收敛标准
   - 测试数据构造/清理规范

## 5. 资深运维工程师视角

### 5.1 考虑不周全点

1. **敏感信息处理存在现实风险**  
   `manu.md` 中“可填写密码/密钥”虽强调写入 `.env`，但未定义“对话通道是否允许传真实密钥”的安全边界。

2. **缺少 ORR（上线准备评审）硬门禁**  
   目前有部署检查，但缺少值班、告警接收人、应急通讯录、变更窗口、回滚演练记录等完整 ORR。

3. **缺少 IaC 与环境漂移治理**  
   对基础设施变更（网络、权限、资源）缺少“基础设施即代码+漂移检测”的要求。

4. **缺少灾备演练节奏化机制**  
   有回滚条目，但缺少“RTO/RPO 演练周期、演练证据、负责人”的持续治理。

### 5.2 优化建议

1. 在 `AGENTS.md.template` 与 `manu.md` 明确敏感信息策略
   - 禁止在聊天正文直接粘贴生产密钥
   - 只允许引用密钥名/Secret ID
   - 通过 Vault/KMS/CI Secret 注入真实值

2. 在 `templates/deployment_template.md` 增加 ORR 清单
   - 值班负责人、回滚拍板人
   - 告警路由与升级路径
   - 变更窗口与冻结期检查
   - 依赖系统联动确认

3. 在 `templates/design_template.md` 增加“运维基线”
   - IaC 路径与版本
   - 配置来源分层
   - 漂移检测方式
   - Secret 轮换策略

4. 新增 `templates/postmortem_template.md`（建议）
   - 触发条件、时间线、根因、遏制、长期修复
   - 与 `docs/lessons_learned.md` 双向链接

## 6. 跨角色优先级落地建议（建议按两周节奏）

### P0（本周应落地）

1. **敏感信息输入规范**：禁止在会话中传真实生产密钥（`manu.md`、`AGENTS.md.template`）。
2. **发布后业务复盘机制**：新增 T+7/T+30 业务指标复盘触发（`phases/07-deployment.md`、`status_template.md`）。
3. **最小脚本化门禁**：把 R6/AC-05/主文档追溯从“示例命令”升级成脚本。

### P1（两周内）

1. 引入 `mini_spec.md` 承载 Minor 流程，降低 `status.md` 过载。
2. 引入 ADR 目录与模板，补强架构长期演进可追溯。
3. 完善 ORR 清单和应急热修流程（含版本命名与审批矩阵）。

### P2（持续优化）

1. 引入 Launch Plan、Postmortem 模板，形成“市场-产品-研发-运维”闭环。
2. 建立指标看板与自动回填机制，减少手工维护负担。

## 7. 预期收益

按上述改进后，这套规范将从“强工程交付框架”升级为“商业成功导向的全链路框架”，实现：

- 研发交付可控（继续保持）
- 商业目标可验证（新增）
- 上线运营可持续（新增）
- 安全与合规更稳健（增强）
