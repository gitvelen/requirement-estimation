# 项目建设规范审查与改进建议（advice.md）

> 范围：基于 `AGENTS.md.template`、`ai_workflow.md`、`manu.md`、`STRUCTURE.md`、`phases/` 全部文件、`templates/` 全部文件的规定与模板，分别从 **资深产品专家 / 资深架构师 / 资深开发工程师 / 资深运维工程师** 视角做“考虑不周全点”的审查，并给出可落地的优化建议。

---

## 0. 总体结论（先修的 P0 / 建议的 P1）

### P0（建议优先修复，否则流程容易“卡死/漂移/误判”）
1. **文档引用存在明显断链**  
   `phases/03-design.md`、`phases/04-planning.md`、`phases/05-implementation.md`、`phases/06-testing.md` 多处写“见本文档 §3.5.2 解决方案”，但这些阶段文件本身并无该章节；当前真正可执行入口在 `templates/review_template.md` 的 AC-05（diff-only 审查增强）。  
   - 建议：阶段文件中删除/替换“§3.5.2”引用，统一指向 `templates/review_template.md` 的 AC-05，避免执行时找不到规则导致门禁形同虚设。

2. **`status.md` 的 `_baseline/_current` 机读字段与 shell 命令假设不一致（潜在致命）**  
   `templates/status_template.md` 的 YAML front matter 用了带引号占位（`"<基线版本>"`），而多个阶段 SOP 用 `grep + awk '{print $2}'` 直接取值；一旦填入时保留引号（如 `_baseline: "v1.0"`），`git rev-parse` / `git diff` 会直接失败。  
   - 建议：明确并强制 `_baseline/_current` **不加引号**（或统一用 YAML 解析器/剥离引号的命令），并在模板与 SOP 中保持一致。

3. **“人工确认/验收”与“AI 自动期无需人工介入”存在语义冲突**  
   例如：`phases/03-design.md` 的质量门禁含“与用户确认通过”，`phases/06-testing.md` 含“用户验收通过”，但 `ai_workflow.md` / `manu.md` 对 03–06 阶段强调“AI 全自动、你无需操作”。  
   - 建议：统一定义“验收/确认”的含义（AI 自验收 vs 人工签署），并在自动期将“用户确认”改为 **可选** 或 **仅在触发条件下强制**（见后续建议）。

4. **版本号/Tag 规范存在冲突：RC Tag 与 `vMAJOR.MINOR` 约束不兼容**  
   `STRUCTURE.md` 规定版本号形如 `vMAJOR.MINOR`，但 `phases/00-change-management.md` 推荐 RC tag（如 `v2.0-rc1`）。  
   - 建议：明确采用（类）SemVer：`vMAJOR.MINOR[.PATCH][-rcN]`，或明确“RC tag 属于例外命名规则”，并补充“何时允许/如何回收”。

### P1（不改也能跑，但会降低产出质量/可维护性/可运营性）
1. **缺少统一的严重度（P0/P1/P2）判定口径**：目前“收敛判定”依赖 P0/P1(open)=0，但未给出跨阶段一致的定义/示例。  
2. **缺少“证据存放与脱敏规则”**：模板大量要求“日志/截图/输出”，但核心原则又要求敏感信息不落盘，二者需要可操作的折中（脱敏、路径、保留期）。  
3. **自动化门禁多数是“示例命令”，缺少最小可用的脚本化落地**：建议提供 `scripts/` 或 Makefile 统一入口（哪怕只覆盖 R5/R6/AC-05/主文档同步）。  
4. **Minor 简化流程落地细节不足**：只写“记录到 status.md”，但缺少 REQ/验收/测试证据的结构化承载，容易让 `status.md` 过载且追溯断裂。

## 1. 资深产品专家视角：缺口与建议

### 1.1 主要缺口（产品层面）
1. **价值证据链不够硬**：提案模板有“成功指标”，但缺少“证据来源”（数据、访谈、工单、竞品对比）与“指标采集方案”。只写目标容易变成口号，后续验收也缺口径与埋点。
2. **优先级与取舍机制薄弱**：需求模板没有强制的“优先级/版本归属”，计划模板虽有任务优先级，但容易出现“需求全要做、任务再排序”的反向失真。
3. **上线沟通/培训/运营动作缺位**：Deployment 强调技术发布，但对用户可感知变更的公告、培训、FAQ、客服话术、运营指标跟踪没有统一承载。
4. **验收角色与签署不清**：测试报告模板有“验收签署”，但工作流又说测试阶段无需人工介入；实际项目常需要明确“谁有最终验收权、何时验收、验收证据形式”。

### 1.2 建议改进（可落地）
1. **在 `templates/proposal_template.md` 增加“证据与洞察”与“度量/埋点计划”**（可选但推荐）
   - 证据：现网数据截图/SQL、用户访谈要点、工单Top问题、竞品差异点。
   - 度量：为每个成功指标补齐“数据源/埋点事件/口径/采样窗口/负责人”。
2. **在 `templates/requirements_template.md` 的需求列表中加入“优先级/版本”列，并强制 MoSCoW（Must/Should/Could/Won’t）或 P0/P1/P2**  
   - 让“范围锁定”真正落到需求层，而不是只在提案的 Non-goals 口头约束。
3. **补一个轻量 `Release Notes` 机制（建议加到 `templates/deployment_template.md` 或新增模板）**  
   - 结构：面向用户的变更点、影响人群、已知限制、回滚说明、客服/运营注意事项。
4. **定义“验收”的两种模式并固化到工作流**  
   - AI 自动期：默认“AI 自验收”（以 test_report 证据为准）。  
   - 强制人工验收触发条件：涉及收费/合规/核心路径/对外契约变更/重大交互变更时，在 `status.md` 标记 `wait_confirm` 并要求人工签署。
5. **Minor 简化流程增加一个“Mini Spec”承载**  
   - 建议：对 Minor 流程，至少落一个 `docs/<版本号>/mini_spec.md`（包含：变更摘要、验收标准、影响面、回滚要点、验证命令），避免把结构化信息全塞进 `status.md`。

## 2. 资深架构师视角：缺口与建议

### 2.1 主要缺口（架构治理）
1. **关键决策缺少“强制沉淀点”**  
   主文档模板里有 ADR 摘要表，但流程并未要求在关键分岔处产出 ADR（何时必须写、写到哪里、谁批准）。
2. **高风险变更的“提前停机点”偏晚**  
   当前对 API/数据迁移/权限等强调 Deployment 例外暂停，但架构上很多风险在 Design/Implementation 就应该触发“强制审查/强制人工确认”，否则后期返工成本高。
3. **契约治理（API/事件）缺少“兼容性判定标准”**  
   模板有 API 列表与兼容性字段，但缺少明确规则：字段新增/删除/语义变化如何判定破坏性；客户端如何迁移；废弃策略如何执行。
4. **数据迁移治理缺少“在线迁移”基线策略**  
   设计模板要求写迁移/回滚，但缺少更工程化的“向后兼容窗口/双写/回填/校验/切换/清理”的标准步骤与最小门禁。

### 2.2 建议改进（可落地）
1. **补齐 ADR 机制（建议新增目录与模板）**
   - 目录：`docs/adr/ADR-XXXX.md`（或 `docs/<版本号>/adr/`）。  
   - 触发：新增依赖、存储选型变更、对外 API 破坏性变更、迁移策略选择、权限模型调整、引入异步/队列、可用性目标变更等。  
   - 在 `templates/design_template.md` 增加“关联 ADR”章节/表格索引。
2. **把“高风险触发”从 Deployment 前移到 Design/Planning 的门禁**  
   - 当 CR 勾选了“强制清单触发”（API/数据迁移/权限/不可逆配置/性能）时：  
     - Design 阶段：强制 `@review`（或强制人工确认）后才能进入 Planning。  
     - Implementation 阶段：要求附带“失败路径 + 回滚演练方案”的证据占位。
3. **明确 API 兼容性规则并模板化**（建议加到 `templates/master_api_doc_template.md` 的“版本策略”）
   - 规则示例：字段只增不删、删除需双版本、错误码语义不变、默认值策略、分页/排序约定等。
4. **给数据迁移提供最小 SOP（建议以清单形式写入 `templates/design_template.md` 与 `templates/deployment_template.md`）**
   - 迁移步骤：新增字段→双写→回填→一致性校验→读切换→清理旧字段（含回滚路径）。  
   - 门禁：必须有备份点、回填校验脚本、灰度策略、回滚演练记录（与 `templates/test_report_template.md` 的回滚验证联动）。
5. **统一“P0/P1 严重度口径”**（建议在 `templates/review_template.md` 增加附录）
   - P0：安全/数据丢失/不可回滚/对外契约破坏/无法部署/无法验证。  
   - P1：可用性风险/监控缺失/高概率缺陷/主文档追溯缺失等。  
   让“收敛判定”更可解释、可复用。

## 3. 资深开发工程师视角：缺口与建议

### 3.1 主要缺口（研发落地）
1. **缺少 CI/门禁的统一执行入口**  
   当前门禁散落在阶段文件与模板附录里（命令示例），但没有“一个命令跑完必需检查”的约定，容易出现“写了规范但没人执行”。
2. **Commit/PR 追溯规则与 Squash merge 可能冲突**  
   规范要求 commit 消息包含 CR-ID；但推荐 Squash and merge 后，如果 squash message 未包含 CR-ID，会导致追溯门禁或审计链路断裂。
3. **多 Agent 协作的目录约定未进入 `STRUCTURE.md`**  
   `phases/05-implementation.md` 约定 `docs/<版本号>/tasks/`，但结构文档未声明该目录的地位与约束（是否必须、是否入库、如何清理）。
4. **“证据驱动”与“敏感信息不落盘”的冲突缺少工程手段**  
   开发/测试证据（日志/截图/命令输出）很容易带 token、cookie、真实数据；没有脱敏规范会导致要么不留证据、要么留了违规证据。

### 3.2 建议改进（可落地）
1. **增加最小脚本化门禁（建议新增 `scripts/`）**
   - `check_refs.sh`：覆盖 R6（plan 引用存在性）  
   - `check_diff_only.sh`：覆盖 AC-05（CR Impact vs diff 的一致性）  
   - `check_docs_trace.sh`：覆盖 Deployment 的主文档追溯（规则2）  
   目标不是“全自动完美”，而是把关键门禁从“文档约定”变成“可执行工具”。
2. **明确 Squash merge 的提交信息模板**
   - 约定：Squash commit message 必须包含全部 CR-ID；PR 标题同样必须包含。  
   - 或：改为 `rebase and merge` 保留每个 commit 的 CR-ID（看团队偏好，但要统一）。
3. **把 `docs/<版本号>/tasks/` 纳入结构约定**
   - 在 `STRUCTURE.md` 增加“可选：tasks 工作区”的说明（何时使用、清理规则、是否影响主代码目录）。
4. **补一份“证据脱敏与存放规则”**
   - 建议约定：证据默认写到 `docs/<版本号>/evidence/`；文件名含日期与 CR-ID；必须脱敏（token/手机号/邮箱/订单号等）。  
   - 对命令输出：建议用 `sed`/`rg` 做关键字段打码示例，避免“不能落盘”被理解成“什么都不记录”。
5. **把“依赖与安全”从原则变成门禁**
   - 增加：新增依赖必须写入 `design.md` 的“新增依赖评估表”，并在 review 时作为 P1 检查点；可选增加 license/漏洞扫描的执行入口（由项目工具链决定）。

## 4. 资深运维工程师视角：缺口与建议

### 4.1 主要缺口（可运维/可支持）
1. **缺少 ORR（Operational Readiness Review）门禁**  
   Deployment 强调步骤与回滚，但缺少“上线前可运维性准备”的最小清单：监控告警是否齐、SLO/阈值是否定、Runbook 是否可用、值班/升级路径是否明确。
2. **缺少“上线后观察窗口与回滚触发条件”的硬约定**  
   模板有“监控跟进”，但缺少“必须监控哪些指标、观察多久、触发回滚阈值是什么”的可执行要求。
3. **备份/恢复演练不够刚性**  
   数据迁移检测很强，但没有强制要求“恢复演练/校验脚本/恢复时间目标”与证据。
4. **配置与密钥治理过于原则化**  
   说了“禁止写真实值”，但缺少：配置来源层级（环境变量/配置中心/Secret 管理）、权限申请流程、轮换频率、最小权限落地方式。

### 4.2 建议改进（可落地）
1. **在 `templates/deployment_template.md` 增加 ORR 清单（建议 P1，触发条件可 P0）**
   - SLO/告警：关键 SLIs、阈值、告警去噪、通知对象。  
   - Runbook：常见故障处理步骤、回滚脚本、数据修复手册。  
   - 观测：日志字段/trace id、关键看板链接。  
   - 权限：谁可操作、审计记录位置。
2. **强制“上线后观察窗口”与“回滚触发”字段**
   - 例如：上线后 30/60 分钟观察；若错误率>阈值或 P95>阈值则回滚；并记录观察证据（截图/查询）。
3. **把“备份与恢复演练”纳入测试报告或部署门禁**
   - 建议与 `templates/test_report_template.md` 的“回滚验证”联动：当 CR 勾选数据迁移/不可逆配置时，必须给出备份点与恢复演练证据。
4. **补齐配置/密钥的最小规范**
   - 在 `templates/design_template.md` 的“配置、密钥与开关”中增加：配置来源、Secret 管理方案（Vault/KMS 等）、轮换策略、最小权限的操作步骤（不写真实值，但写流程）。
5. **增加“变更沟通与值班”字段（尤其是对外影响）**
   - 在 Deployment/Release Notes 中写清：公告对象、时间窗口、回滚联系人、升级路径（谁拍板回滚）。

## 5. 建议的“文本级修订清单”（便于你直接改规范文件）

> 以下是“改了就更稳”的具体修订点（不改变框架思想，只修断链与语义冲突）：

1. 将 `phases/03-design.md`、`phases/04-planning.md`、`phases/05-implementation.md`、`phases/06-testing.md` 中的“见本文档 §3.5.2”统一改为“见 `templates/review_template.md` 的 AC-05”。
2. 在 `templates/status_template.md` 明确 `_baseline/_current` **禁止加引号**（或同步更新所有 grep/awk 命令以剥离引号）。
3. 在 `ai_workflow.md`/`manu.md` 补一句：自动期如触发高风险项（API/迁移/权限/不可逆配置/合规），自动进入 `wait_confirm` 并建议 `@review`。
4. 在 `STRUCTURE.md` 补充：允许 RC tag 命名（或升级为 `vMAJOR.MINOR.PATCH[-rcN]`）。
5. 在 `templates/review_template.md` 增补 P0/P1/P2 判定示例与“Accept/Defer”的定义与落盘位置（建议落到 `status.md` 的 Deferred Items 表）。
6. 在 `STRUCTURE.md` 增补 `docs/<版本号>/tasks/`（可选）与 `docs/<版本号>/evidence/`（建议）目录约定。
7. 在 `templates/deployment_template.md` 增补 ORR 与“上线后观察窗口/回滚触发阈值”字段。
8. 在 `templates/requirements_template.md` 的需求列表加入“优先级/版本归属”列，避免范围漂移。
9. 在 `templates/proposal_template.md` 增补“证据与洞察 / 埋点与度量方案 / 发布沟通要点”。
10. 明确“验收签署”的触发条件：默认 AI 自验收；触发条件下才要求人工签署（并在 `status.md` 记录）。
