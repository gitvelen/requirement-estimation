# 阶段1：起草提案 (Proposal)

## 目标
将模糊的创意转化为清晰的产品提案

## 输入
- 用户的初步想法、创意或问题
- `docs/系统功能说明书.md`（全新项目没有，存量项目才具备）

## 输出
- `docs/<版本号>/proposal.md` — 产品提案文档

## 阶段开始时检查
- [ ] 确认变更目录已创建
- [ ] 确认与用户沟通了初步想法
- [ ] 如为存量系统：在 `docs/<版本号>/status.md` 中填写"基线版本（对比口径）"（例如 `v1.0`）

## 行为准则
```
┌─────────────────────────────────────────────────────────────┐
│  1. 倾听用户意图，不急于给出方案                             │
│  2. 引导用户深入思考：                                       │
│     - 这个功能的核心价值是什么？                             │
│     - 目标用户是谁？他们有什么痛点？                         │
│     - 成功的衡量标准是什么？                                 │
│  3. 充分讨论，厘清痛点与价值                                 │
│     - 与用户反复确认痛点的真实性和优先级                     │
│     - 讨论现有解决方案的不足之处                             │
│     - 帮助用户区分"想要"和"需要"                             │
│  4. 运用极致的产品思维，帮助用户发现思维盲区                 │
│  5. 反复确认，确保理解无误                                   │
│  6. 整理成结构化的提案文档                                   │
│  7. 如果修改了proposal.md,应形成修改记录，写明修改要点和修改章节    │
│  8. 已完成/已测试后出现新意图：优先写 CR（见 `phases/00-change-management.md`） │
└─────────────────────────────────────────────────────────────┘
```

## 读取模板
编写提案时读取 `.aicoding/templates/proposal_template.md`。

## 质量门禁
- [ ] 用户确认提案内容准确反映其意图
- [ ] 目标清晰可衡量
- [ ] 范围边界明确

## 完成条件（🔴 MUST）

### 审查要求
- 人工指定审查者：`@review Claude` 或 `@review Codex`
- 审查结果追加到 `review_proposal.md` 文件末尾
- 可多次指定不同审查者，直至问题收敛

### 人工确认
- [ ] 人工阅读 `review_proposal.md`
- [ ] 人工确认问题可接受
- [ ] 人工确认进入 Requirements 阶段
- [ ] 更新 `status.md`：当前阶段 = Requirements

## 完成后
1. 确认质量门禁通过
2. 询问用户："提案已完成，是否进入 Requirements 阶段？"
3. 用户确认后，更新 `docs/<版本号>/status.md` 中的"当前阶段"为 Requirements（如未创建则先创建）
