## 决策与配置收集（🔴 MUST，Design 前置步骤）

> **目标**：一次性收集所有需要用户拍板的技术决策和部署配置，确保 Design → Testing 全程无需人工介入。

### 触发时机
- Design 阶段开始后、正式编写 design.md 之前

### AI 执行步骤
1. **扫描 `requirements.md`**，提取需要用户决策的技术问题
2. **生成"决策与配置清单"**，一次性向用户提问，包含两部分：

#### Part 1：技术决策（必答）
AI 根据需求自动识别需要决策的问题，常见类型：
- 后端语言/框架（如已有代码库则跳过）
- 数据库类型与版本
- 缓存/消息队列选型（如需要）
- 部署形态（Docker / K8s / 裸机 / 云服务）
- 架构模式（如需要选择）
- 第三方服务/SDK（如需要引入）

**用户回答方式**（每项三选一）：
- 填写具体值（如"PostgreSQL 15"）
- 标注"AI 自行决定"
- 标注"给我两个方案对比"（AI 给出对比后用户选择）

#### Part 2：环境与部署配置（能填就填，不确定的留空）
- 目标环境数量与用途（如：开发 + 生产）
- 服务器/云平台（如：阿里云 ECS / AWS EC2）
- 域名与端口规划
- 资源规格（CPU/内存/存储预算）
- 运行时版本（如：Python 3.12 / Node 20）
- 数据库连接信息（地址、端口、库名、账号密码）
- 缓存/消息队列连接信息
- 第三方服务密钥（API Key 等）
- 其他环境变量

**敏感信息处理规则（🔴 MUST）**：
- 用户提供的敏感值（密码、密钥、连接串等）**不写入 design.md**
- AI 在 design.md 中只记录变量名（如 `DATABASE_URL=<见 .env>`）
- AI 生成 `.env.example`（列出所有变量名和说明）和 `.env`（填入实际值）
- `.env` 必须加入 `.gitignore`，禁止提交仓库

**留空项处理**：
- 用户留空的配置项，AI 在 design.md 中标记"待部署阶段确认"
- Deployment 阶段 AI 会提醒用户补填这些项（仅限留空项，不重复已填项）

### 收集完成后
- 用户回答完毕 → AI 将决策结果写入 `design.md` 的"决策记录"章节
- 环境配置写入 `design.md` 第 6 章"环境与部署"
- 敏感信息写入 `.env`（不进 git）
- **此后 Design 阶段全自动完成，不再暂停**

### 存量项目/CR 场景的简化
- 如技术栈已确定（存量项目增量迭代），Part 1 中已确定的项自动跳过
- 如环境配置无变化，Part 2 可整体跳过（沿用上一版本的 `.env`）
- AI 只问"本次迭代新增的决策点"

## 设计原则
```
┌─────────────────────────────────────────────────────────────┐
│  1. 架构简洁优先                                             │
│     - 避免过度设计                                           │
│     - 优先选择成熟方案                                       │
│     - 便于运维                                               │
│                                                              │
│  2. 依赖审慎引入                                             │
│     - 不引入非主流库                                         │
│     - 优先使用语言生态标准库                                 │
│     - 评估依赖的安全性和维护状态                             │
│                                                              │
│  3. 设计必须可落地                                           │
│     - 包含足够的细节                                         │
│     - 可直接转化为代码                                       │
│                                                              │
│  4. 考虑向后兼容                                             │
│     - 不破坏现有功能                                         │
│     - 数据迁移方案                                           │
│                                                              │
│  5. 如果修改了 design.md，应形成修改记录，写明修改要点和修改章节 │
└─────────────────────────────────────────────────────────────┘
```
