# AI 工作流控制规则

## 变更分级与流程选择

> 目标：根据变更规模选择合适的流程，在效率与可追溯性之间取得平衡。

### 分级标准

| 级别 | 判断条件（满足任一即为该级别） | 流程 |
|------|-------------------------------|------|
| **Major（完整流程）** | 新功能开发；涉及 API 契约变更；涉及数据库 schema 变更；涉及权限/安全变更；跨模块影响；用户明确要求完整流程 | 完整 8 阶段流程 |
| **Minor（简化流程）** | 单模块内的小功能增强；Bug 修复（非紧急）；UI 微调；配置变更；文档修正 | 简化流程（见下方） |

### Minor 简化流程

> 核心原则：**简化文档，不简化理解**。AI 必须确认理解了需求再动手。

```
1. 需求确认（合并 Proposal + Requirements）
   - AI 与用户充分讨论，确认：做什么、不做什么、验收标准
   - 输出：在 status.md 中记录变更摘要 + 验收标准（无需单独的 proposal.md / requirements.md）
   - 用户确认后进入下一步

2. 设计+计划+实现（合并 Phase 03-05）
   - AI 直接编码，但必须：
     - 先读后写（理解现有代码）
     - 遵循实现检查清单
     - commit 消息包含变更摘要
   - 如实现中发现复杂度超出预期（如需跨模块修改），AI 应暂停并建议升级为 Major 流程

3. 测试（Phase 06，不简化）
   - 仍需执行完整测试（单元+集成+回归）
   - 输出简化版测试报告（可内联在 status.md 中）

4. 部署确认（Phase 07，不简化）
   - 仍需用户确认后部署
   - 更新 status.md 为 Done
```

### 分级决策由谁做？
- **用户指定**：用户可直接说"这是个小改动，走简化流程"
- **AI 建议**：AI 评估后可建议分级，但必须经用户确认
- **升级机制**：Minor 流程中发现复杂度超预期时，AI 必须暂停并建议升级为 Major

## 收敛判定标准

### 人工介入期（Phase 00-02）
- **触发方式**：人工指定审查者（`@review Claude` 或 `@review Codex`）
- **判定方式**：人工判定
- **收敛信号**：审查者输出"建议人工确认进入下一阶段"

### AI 自动期（Phase 03-06）
- **触发方式**：AI 主动自我审查
- **判定方式**：AI 自动判定
- **收敛条件**：
  - ✅ P0(open) = 0
  - ✅ P1(open) = 0
  - 允许存在 P1(accept/defer)，但需在 RVW 记录中写清理由+缓解
  - **单轮自我审查满足即收敛**
- **自动推进**：满足条件后自动更新 status.md，进入下一阶段

### Deployment 阶段（Phase 07）
- **默认行为**：自我审查后，满足条件则**暂停**，输出："请人工确认后部署"
- **例外情况（强制 wait_confirm）**：
  - 涉及 API 契约变更
  - 涉及数据迁移
  - 涉及权限/安全变更
  - 涉及不可逆配置
- **例外处理**：AI 自动暂停，输出："涉及 [例外类型]，请人工指定审查或确认"

## 人工介入期规则（Phase 00-02）

### 审查流程
1. AI 完成阶段产出
2. 人工指定审查者：`@review Claude` 或 `@review Codex`
3. 审查者执行审查，结果**追加到文件末尾**
4. 人工可继续指定其他审查者
5. 重复 2-4，直至收敛
6. 人工确认后进入下一阶段

### 强制等待机制（🔴 MUST）

- AI 完成阶段产出后，**必须**将 `status.md` 的运行状态设为 `wait_confirm`
- 🚫 **禁止** AI 在人工介入期（Phase 00-02）自行更新"当前阶段"字段
- 仅当用户在对话中**明确确认**（如"确认"、"进入下一阶段"）后，AI 才可更新"当前阶段"并将运行状态恢复为 `running`

### 审查记录格式
每轮审查追加到同一文件末尾，格式：
```
## YYYY-MM-DD HH:MM | 第 N 轮 | 审查者：<Name>

### 审查角度
...

### 上一轮问题处理状态
...

### 本轮新发现问题
...

### 收敛判定
...
```

## AI 自动期规则（Phase 03-06）

### 轮次定义（🔴 MUST）

> **一轮** = AI 完成阶段工作 → 自我审查 → 输出审查报告

**轮次计数规则**：
- 一轮 = AI 完成本阶段工作 → 执行自我审查 → 输出审查报告（含 P0/P1 问题列表）
- AI 修复问题后再次审查 = 新的一轮（轮次计数 +1）
- 人工中断并给出反馈 = 不计入轮次（轮次计数不变）
- 人工直接修改文件 = 不计入轮次（轮次计数不变）

**收敛判定**：
- P0(open) = 0 且 P1(open) = 0 → 收敛
- 允许 P1(accept/defer) 存在，但需在 RVW 记录中写清理由+缓解

**轮次记录位置**：
- 记录在 `status.md` 的"阶段转换记录"表中，或阶段产出文件末尾

### AI 行为
1. 完成阶段产出
2. 执行自我审查（按 `.aicoding/templates/review_template.md` 对应清单）
3. 如满足 P0(open)=0 且 P1(open)=0，自动判定收敛
4. 自动更新 status.md，进入下一阶段

### 连续3轮不收敛触发机制（🔴 MUST）

> **目标**：避免 AI 陷入无法收敛的死循环，及时请求人工介入

**触发条件**：
- 同一阶段连续自我审查 3 轮后仍不收敛（P0(open) > 0 或 P1(open) > 0）

**触发动作**：
1. AI 自动暂停，输出：
   ```text
   ⚠️⚠️⚠️ 连续3轮自我审查未收敛，请求人工确认 ⚠️⚠️⚠️

   当前阶段：[阶段名]
   轮次计数：3
   未收敛问题：
   - P0(open): X 个
   - P1(open): Y 个

   请选择：
   1. 人工确认后继续（将部分 P1 标记为 accept/defer）
   2. 人工给出反馈，AI 继续修复
   3. 跳过当前阶段，进入下一阶段（需人工确认风险）
   ```
2. 在 `status.md` 中记录："AI 自我审查 3 轮未收敛"
3. 更新 `status.md`：运行状态=wait_confirm

**人工处理选项**：
- **选项1**：将部分 P1 问题标记为 accept/defer，然后 AI 继续验证
- **选项2**：人工给出具体反馈，AI 根据反馈继续修复
- **选项3**：人工确认跳过，记录风险后进入下一阶段

**重置规则**：
- 人工确认后，轮次计数重置为 0
- 最多再给 2 轮机会（即第 5 轮仍不收敛时强制停止）

### 收敛后输出
```
✅ [阶段] AI 自动审查已收敛（第 N 轮）
▶️ 自动进入下一阶段：[阶段名]
```

## Deployment 阶段规则（Phase 07）

### AI 行为
1. 完成阶段产出
2. 执行自我审查
3. 如满足 P0(open)=0 且 P1(open)=0：
   - 检查是否涉及例外情况（API/数据迁移/权限/不可逆配置）
   - 输出 "请人工确认后部署"，暂停
4. 人工确认后才继续

---

## 紧急中断机制

### 触发条件
- P0 Blockers 无法自动修复
- 连续 3 轮不收敛
- 发现涉及安全/合规的重大风险

### AI 行为
1. 立即停止当前工作
2. 更新 `status.md`：运行状态=paused 或 wait_confirm
3. 输出醒目提示：
   ```
   ⚠️⚠️⚠️ 工作已暂停 ⚠️⚠️⚠️
   原因：[原因描述]
   当前阶段：[阶段名]
   请人工确认后恢复
   ```

### 恢复机制
- 人工在 `status.md` 中设置：运行状态=running
- 工作流模式 保持不变（由人工根据情况决定是否修改）
- AI 检测到状态恢复，继续工作
- 或人工直接给出"继续"指令
