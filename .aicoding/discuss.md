# 变更管理流程讨论记录

> 讨论时间：2026-02-08
> 讨论主题：现有变更管理流程在变更场景下的问题分析

---

## 一、主文档同步问题讨论

### 1.1 问题描述

用户提出：如果变更情况下不及时更新文档，那么 Deployment 阶段主文档的更新依据是什么？新增的一些功能和设计，岂不是就遗漏了？

### 1.2 问题本质

```
变更流程：
┌─────────────────────────────────────────────────────────────────┐
│  CR 创建                                                          │
│  - 填写 Impact（影响文档：proposal/requirements/design/...）     │
│  - 但这些是"阶段文档"，不是"主文档"                               │
└─────────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────────┐
│  实现阶段                                                         │
│  - 修改代码                                                       │
│  - "回填最少必要文档"（只更新受影响的阶段文档）                   │
│  - 阶段文档 ≠ 主文档                                              │
└─────────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────────┐
│  Deployment 阶段                                                 │
│  - 要求"按影响面同步更新主文档"                                   │
│  - 但影响面在哪里？？？                                           │
│  - CR 的 Impact 字段只列出了阶段文档！                            │
└─────────────────────────────────────────────────────────────────┘
```

### 1.3 核心矛盾

| 维度 | 阶段文档 | 主文档 |
|------|---------|--------|
| 维护方式 | 追加式（每次变更新建目录） | 截面式（覆盖当前状态） |
| 详细程度 | 详细（包含决策过程） | 精炼（只描述最终状态） |
| ID 系统 | REQ-xxx, API-xxx, TEST-xxx | FUNC-xxx（功能说明书） |
| 同步机制 | **无** | **无** |

### 1.4 关键发现

主文档模板中已经有关联字段，说明设计者意识到了追溯问题，但只留了字段，没有强制机制：

| 主文档模板 | 关联字段 |
|-----------|---------|
| 系统功能说明书.md | `关联需求文档: docs/<版本号>/requirements.md` |
| 技术方案设计（版本文档） | `关联主文档: docs/技术方案设计.md` |
| design.md | `关联主文档: docs/技术方案设计.md` |

## 三、解决方案讨论

### 3.1 核心思路

不增加新文档，不增加新流程，只增强 Deployment 阶段的门禁检查。

### 3.2 CR 模板增强

修改 `.aicoding/templates/cr_template.md` 的 Impact 字段：

```markdown
## 3. 影响面

### 3.1 阶段文档影响
| 影响文档 | 影响章节/内容 |
|---------|-------------|
| [ ] proposal | |
| [ ] requirements | 影响 REQ: ___ |
| [ ] design | 影响章节: ___ |
| [ ] plan | |
| [ ] test_report | |
| [ ] deployment | |

### 3.2 主文档影响（🔴 MUST 勾选）
| 主文档 | 是否影响 | 影响说明 | 关联 ID |
|--------|---------|---------|---------|
| [ ] 系统功能说明书.md |  | 新增/修改 FUNC-xxx | |
| [ ] 技术方案设计.md |  | 新增/修改章节 | |
| [ ] 接口文档.md |  | 新增/修改 API-xxx | |
| [ ] 用户手册.md |  | 新增/修改操作 | |
| [ ] 部署记录.md |  | 追加部署记录 | |

### 3.3 代码影响
| 影响模块/文件 | 变更类型 |
|--------------|---------|
| | |
```

### 3.3 Deployment 阶段强制追溯检查

在 Deployment 模板中增加强制检查清单：

```markdown
## 主文档同步验证（🔴 MUST，零容忍）

### 验证步骤（AI 自动执行）
1. 读取 `docs/<版本号>/status.md` 获取当前版本的 CR 列表
2. 对每个 Active CR：
   - 读取 CR 文件，提取 "影响文档" 和 "影响 ID"
   - 根据影响 ID，检查对应主文档是否已更新
3. 输出验证报告

### 验证规则（AI 自动判断）

| 影响类型 | 必须检查的主文档 | 检查方法 |
|---------|-----------------|---------|
| 新增/修改 REQ-xxx（功能性） | 系统功能说明书.md | 主文档中是否存在对应 FUNC-xxx 或内容覆盖 |
| 新增/修改 API-xxx | 接口文档.md | 主文档中是否存在该 API-xxx 的说明 |
| 架构/选型/关键设计变更 | 技术方案设计.md | 主文档中对应章节是否已更新 |
| 用户可见交互/流程变更 | 用户手册.md | 主文档中操作步骤是否已更新 |

### 质量门禁（🔴 MUST）
- [ ] 所有 CR 中标记的影响文档，主文档均已更新
- [ ] 如有遗漏，AI 必须拒绝进入 Deployment，并报告具体遗漏项
- [ ] 修复后重新验证，直至零遗漏
```

### 3.4 方案优势

| 特点 | 说明 |
|------|------|
| **成本可控** | 只在 Deployment 阶段增加检查，不影响其他流程 |
| **零容忍缺失** | 门禁强制验证，有遗漏就不让通过 |
| **自动执行** | AI 自动检查，不依赖人工记忆 |
| **追溯完整** | CR → 主文档 → 代码，全程可追溯 |
| **不改架构** | 不新增文档类型，不改变现有结构 |

## 四、用户提出的四个关键问题及分析

### 问题 1：变更单的需求是否强制澄清并需用户同意？

**结论：无明确规定**

- 澄清机制依赖 CR 创建者的主动性
- 没有像 Proposal/Requirements 阶段那样的强审查机制
- "人工确认"只是确认问题可接受，不等同于确认需求澄清无误

### 问题 2：是否强制在变更单中把影响填写清楚？谁填？

**结论：无强制验证**

- CR 模板中有 `## 3. 影响面` 字段，但全是空的
- 没有填写示例、验证规则、责任人分配
- 如果不填或填错，后续的"diff-only"审查就会遗漏

### 问题 3：创建变更单时是否要先把代码 push 到仓库？

**结论：未明确强制**

- 流程要求建立 Baseline，但没有强制在创建 CR 前先 push 代码
- 如果代码没有 push，无法做真正的 diff 对比
- 可追溯性断裂，CR 引用的 commit/tag 可能不存在

### 问题 4：如何保证改了代码就不会忘记更新对应文档？

**结论：无强制关联**

- 代码提交和文档更新是独立的操作
- "回填最少必要文档"依赖人工记住哪些文档受影响
- 审查无法自动发现遗漏

## 五、变更场景下的 18 个问题清单

### 一、CR 创建与入口阶段

#### 问题 1：CR 需求澄清不充分
| 现象 | 根因 | 后果 |
|------|------|------|
| CR 只写"改什么"，没写清楚"为什么要改" | 模板中"变更意图"只有一行，没有强制澄清机制 | 实现时可能理解偏差，返工 |
| CR 中验收标准模糊 | 模板没有强制要求可验收的 GWT 格式 | Deployment 时无法验证是否成功 |

#### 问题 2：CR 优先级与影响评估缺失
| 现象 | 根因 | 后果 |
|------|------|------|
| 多个 CR 同时 Active 时没有优先级 | `status.md` 的 Active CR 列表没有优先级字段 | 资源冲突时无法决策先做哪个 |
| 影响评估依赖人工判断 | 没有"影响分析 → 强制清单触发"的映射规则 | 应该 full 审查的只做了 diff-only |

---

### 二、CR → 阶段文档同步阶段

#### 问题 3：CR 与阶段文档的双向追溯不完整
| 现象 | 根因 | 后果 |
|------|------|------|
| CR 提到了影响 REQ-xxx，但 requirements.md 没更新 | 没有"CR 阶段文档变更记录"字段 | 追溯时发现不一致，但不知道谁对 |
| requirements.md 变更了，但 CR 没更新 | 没有"阶段文档变更 → CR 回填"的强制流程 | CR 和实际实现脱节 |

#### 问题 4：diff-only 审查的盲区
| 现象 | 根因 | 后果 |
|------|------|------|
| diff-only 只看 CR 列出的影响面 | CR 的 Impact 字段填写不完整时，审查也跟着漏 | 隐性影响被遗漏 |
| 没有验证"CR 以外的代码是否受影响" | diff-only 默认不重读全部阶段文档 | 级联影响未被发现 |

---

### 三、实现阶段

#### 问题 5：代码变更与 CR 的追溯断裂
| 现象 | 根因 | 后果 |
|------|------|------|
| 代码改了，但 commit/PR 没有引用 CR-ID | 流程只"建议"包含 CR-ID，非强制 | 无法通过代码追溯到 CR |
| 多个 CR 合并实现时，代码层面无法区分 | 没有"CR 代码边界"的概念 | 一个出问题，不知道是哪个 CR 导致的 |

#### 问题 6：实现阶段的文档更新时机不明确
| 现象 | 根因 | 后果 |
|------|------|------|
| 代码改完忘记更新 design.md | 流程说"回填最少必要文档"，但没说什么时候回填 | 可能永远不更新 |
| requirements.md 需要微调，但不确定该不该改 | 没有澄清流程：小的理解偏差算不算 CR | 要么不敢改，要么乱改 |

---

### 四、测试阶段

#### 问题 7：回归范围的确定机制不清晰
| 现象 | 根因 | 后果 |
|------|------|------|
| CR 说"影响模块 A"，但模块 B 的测试也失败了 | diff-only 没有自动分析级联影响 | 回归不充分，上线后出问题 |
| 全量回归成本太高，但不知道能否缩小 | 没有"回归范围 = CR Impact + 自动分析"的规则 | 要么过度测试，要么测试不足 |

#### 问题 8：CR 验收标准与测试用例的映射缺失
| 现象 | 根因 | 后果 |
|------|------|------|
| CR 写了验收标准，但 test_report.md 没对应测试用例 | 没有"CR 验收 → TEST 用例"的强制映射 | CR 无法闭环验证 |
| CR 的"回滚条件"在测试中没有验证 | 模板有回滚字段，但测试阶段没有强制验证 | 真要回滚时发现不可行 |

---

### 五、Deployment 阶段

#### 问题 9：主文档同步验证缺失（已讨论）
| 现象 | 根因 | 后果 |
|------|------|------|
| 功能改了但主文档没更 | 没有"CR Impact → 主文档验证"的门禁 | 主文档永久过时 |

#### 问题 10：多个 CR 同时上线的追溯混乱
| 现象 | 根因 | 后果 |
|------|------|------|
| 一次上线包含 3 个 CR，但部署记录只写"v2.1" | 没有"本次上线包含的 CR 列表"的结构化记录 | 出问题时不知道是哪个 CR 导致的 |
| 回滚时不知道要回滚哪些 CR | CR 没有标记"是否已上线" | 回滚不彻底或回滚过度 |

#### 问题 11：CR 闭环机制不完整
| 现象 | 根因 | 后果 |
|------|------|------|
| CR 状态一直是 Accepted，从未改为 Implemented | 没有强制在 Deployment 后更新 CR 状态 | Active CR 列表永远不清理 |
| Dropped 的 CR 没有记录原因 | 模板有决策记录表，但不是必填 | 同样的错误反复出现 |

### 六、跨阶段问题

#### 问题 12：基线管理的强制力不足
| 现象 | 根因 | 后果 |
|------|------|------|
| CR 没有填写基线版本 | 模板有字段但没验证 | 无法确定 diff 的对比对象 |
| 基线版本打 tag 的时机不明确 | 流程说"建议打 tag"，但没说什么时候必须打 | 要么忘记打，要么打错时机 |

#### 问题 13：CR 的合并与拆分机制缺失
| 现象 | 根因 | 后果 |
|------|------|------|
| 两个相关的小 CR 分别实现，造成冲突 | 没有"CR 依赖关系"字段 | 重复工作或互相覆盖 |
| 一个大 CR 应该拆成多个小的 | 没有"CR 粒度"的指导原则 | 审查和测试成本过高 |

#### 问题 14：异常场景的处理流程不清晰
| 现象 | 根因 | 后果 |
|------|------|------|
| 实现时发现 CR 写错了怎么办？ | 没有"CR 修订流程" | 只能废弃重写，历史丢失 |
| CR 实现一半发现做不完怎么办？ | 没有"CR 暂停/挂起"机制 | 半成品混入代码库 |

### 七、AI 自动期的问题

#### 问题 15：AI 自动期对 CR 的感知能力不足
| 现象 | 根因 | 后果 |
|------|------|------|
| AI 自动推进阶段时，没有检查 Active CR | Design → Planning → Implementation 自动推进，不读 CR | 实现了错误的版本 |
| AI 自我审查只看阶段文档，不看 CR | 审查模板没有强制读取 CR | 审查通过但不符合 CR 要求 |

#### 问题 16：连续 3 轮不收敛的触发条件模糊
| 现象 | 根因 | 后果 |
|------|------|------|
| 什么算"一轮"？ | `ai_workflow.md` 没有明确定义 | 计数方式不一致 |
| 3 轮后自动暂停，但 CR 怎么办？ | 暂停后没有恢复 CR 的机制 | CR 被遗忘 |

## 七、最核心的 3 个问题

1. **CR → 阶段文档 → 代码 → 主文档的追溯链不完整**

3. **CR 闭环机制不完整**
   - CR 状态更新缺失（问题 11）
   - 验收标准与测试用例的映射缺失（问题 8）

## 八、基于当前项目规定的可解决性复核（结论追加）

> 复核范围：以当前 `.aicoding/` 目录内的**流程规定与模板**为准（`manual.md` / `ai_workflow.md` / `STRUCTURE.md` / `phases/*` / `templates/*` / `enhance_v2.md`）。

### 8.1 总体结论

1. **“主文档同步问题”在当前流程下可以解决**：已经形成可执行的 **Deployment 零容忍门禁**，其输入来自 `status.md`（Active CR 真相源）+ `deployment.md`（本次上线 CR 决策）+ `CR` 的 Impact（主文档影响声明）+ 主文档的“变更记录（含 CR-ID）”。  
2. **本文件列出的 18 个问题大多已被“模板字段 + 门禁检查 + 真相源定义”覆盖**；但仍有少量点属于“依赖人工判定/尚未写入最终规则文件/模板与门禁表述不完全一致”，严格意义下会影响“零容忍缺失”的确定性，需要继续补齐（见 8.4）。

### 8.2 关键闭环链路（人机协同如何让问题可解）

结合 `manual.md` 与 `ai_workflow.md` 的分工边界，当前流程能够把“记忆型/容易漏”的工作转成“门禁型/必须过”的检查：

- **真相源层**：`docs/<版本号>/status.md` 作为单一真相源，提供 `_baseline/_current`、`diff-only/full` 口径与 Active CR 列表（模板：`.aicoding/templates/status_template.md`）。
- **意图层**：CR 强制写清 What/Impact/验收/回滚，并显式声明“主文档是否影响”（模板：`.aicoding/templates/cr_template.md`）。
- **实现层**：Implementation 阶段门禁要求 commit/PR 追溯到 CR-ID，并验证 baseline 合法（规定：`.aicoding/phases/05-implementation.md`）。
- **验证层**：Testing 阶段要求 REQ→TEST 追溯矩阵，并为每个上线 CR 提供验证证据与回滚证据（模板：`.aicoding/templates/test_report_template.md`）。
- **发布层**：Deployment 阶段要求：
  - `deployment.md` 的“本次上线CR列表”必须是 `status.md` Active CR 的子集；
  - 对每个上线 CR，按其 Impact 中勾选的主文档执行“变更记录含 CR-ID（必过）+ ID 存在性（增强）”验证；
  - 上线后强制更新 CR 状态为 Implemented 并清理 Active CR（规定：`.aicoding/phases/07-deployment.md`）。

上述链路的关键点在于：**人工负责“确认/决策/例外处理”，AI 负责“执行检查/拒绝放行/给出遗漏清单”**，从机制上降低“忘记同步文档/忘记更新状态/忘记补证据”的概率。

### 8.3 对本文件“四个关键问题”结论的复核

> 注：本文件第“四、用户提出的四个关键问题及分析”的结论属于讨论当时的“旧状态”；以当前 `.aicoding/` 规定复核如下。

1. **问题1（CR 需求是否强制澄清并需用户同意）**：  
   - 当前流程将变更管理（Phase 00）明确为“人工介入期”，要求人工指定审查并确认后进入下一阶段（见 `.aicoding/phases/00-change-management.md` 与 `ai_workflow.md`）。  
   - 但“澄清到什么程度算通过”仍主要依赖审查质量与人工确认口径；属于**流程上可要求、但难以用纯机械门禁完全证明**的项（见 8.4 的剩余风险）。

2. **问题2（Impact 是否强制填清、谁填）**：  
   - CR 模板已将 Impact 标注为 MUST，并拆分为“阶段文档影响/主文档影响/代码影响/强制清单触发”（`.aicoding/templates/cr_template.md`）。  
   - 同时 Deployment 门禁直接依赖“主文档影响”做同步验证（`.aicoding/phases/07-deployment.md`），因此在机制上 **Impact 不完整会直接导致无法通过发布门禁**（前提是门禁把“未明确是/否”视为失败，见 8.4）。

3. **问题3（创建 CR 是否要先 push 代码）**：  
   - 现行规定把 baseline 作为 `status.md` 的唯一真相源，并要求 AI 在 CR 创建时验证 baseline 存在（`.aicoding/phases/00-change-management.md`）。  
   - 对于“代码是否 push”本身并未写成硬性条款，但 Implementation 门禁要求 baseline..current 范围可验证（`.aicoding/phases/05-implementation.md`），实践上会把“可对比的代码边界”强制出来。

4. **问题4（如何保证改代码不忘更新对应文档）**：  
   - 主文档侧：已由 Deployment “主文档同步验证（零容忍门禁）”强制（`.aicoding/phases/07-deployment.md` + 各主文档模板的“变更记录含 CR-ID”要求）。  
   - 阶段文档/影响面侧：通过 diff-only 审查增强（AC-05）对“CR 影响面 vs 实际变更文件”做差异检测，并要求 P1 差异必须修复才能收敛（`.aicoding/templates/review_template.md` + `.aicoding/phases/03~06-*.md`）。

### 8.5 仍需补齐的关键点（否则“零容忍缺失”可能被削弱）

1. **CR 的主文档影响表目前存在“门禁表述 vs 模板写法”的不一致风险**：`phases/07-deployment.md` 按“勾选是/否”表述抽取影响项，但 `templates/cr_template.md` 的主文档影响表目前更像“勾选主文档名 + 填说明”，且未显式强制每行是/否二选一。若执行实现按“只取勾选项”而不检查“是否留空”，会产生漏检空间。  
2. **ISSUE-16 的“一轮/三轮不收敛”尚未写入 `ai_workflow.md`**：当前仍可能出现不同执行者对“轮次”的理解不一致，影响自动暂停的确定性。  
3. **“零容忍”仍有一个不可完全机械化的前提：Impact 声明必须诚实且完整**。现行机制通过 AC-05（CR vs diff）降低漏填概率，但对“应该影响主文档却被错误标为不影响”的情况，仍需要审查质量与规则进一步加固（例如：对 API/数据迁移/权限类变更做更强的自动触发与兜底策略）。
