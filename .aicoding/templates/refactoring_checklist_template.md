# 版本重构功能对照清单

> 适用场景：Major 变更且涉及架构/路由/数据源重写时使用
> 路径：`docs/<版本号>/refactoring_checklist.md`
> 用途：确保旧版本功能在新版本中全部覆盖，避免功能遗漏

## 基线版本
- 旧版本：`v1.0`
- 新版本：`v2.0`
- 对比时间：YYYY-MM-DD

## 功能对照表

| 旧版功能 ID | 旧版功能描述 | 新版覆盖状态 | 新版对应 REQ-ID | 验证方式 | 验证结果 | 备注 |
|------------|-------------|-------------|----------------|---------|---------|------|
| FUNC-001 | 用户登录 | ✅ 已覆盖 | REQ-001 | 手动测试 + 自动化测试 | ✅ 通过 | - |
| FUNC-002 | 数据导出 | ✅ 已覆盖 | REQ-005 | 手动测试 | ✅ 通过 | 导出格式从 CSV 改为 JSON |
| FUNC-003 | 旧版报表 | ⚠️ 部分覆盖 | REQ-010 | 手动测试 | ⚠️ 部分通过 | 仅保留核心报表，次要报表延期到 v2.1 |
| FUNC-004 | 管理员面板 | ❌ 不覆盖 | - | - | - | 已废弃，用户量 < 1% |

**覆盖状态说明**：
- ✅ 已覆盖：新版本完全实现该功能
- ⚠️ 部分覆盖：新版本实现了核心功能，但部分细节有差异（需在备注中说明）
- ❌ 不覆盖：新版本不实现该功能（需在备注中说明原因）

## 验证方式说明

### 运行时行为验证（🔴 MUST）
- **禁止**仅使用静态字符串匹配（如 grep 搜索函数名）
- **必须**执行实际操作并验证输入输出：
  - UI 功能：打开页面 → 执行操作 → 确认结果
  - API 功能：发送请求 → 检查响应 → 验证副作用
  - 后台任务：触发任务 → 等待完成 → 检查结果

### 验证记录要求
- 每个功能必须记录验证方式和验证结果
- 验证结果必须是明确的"通过/失败/部分通过"
- 失败或部分通过的功能必须在备注中说明原因

## 风险评估

| 风险项 | 风险等级 | 缓解措施 | 责任人 |
|-------|---------|---------|--------|
| 旧版功能遗漏 | 高 | 逐项验证 + 用户验收 | AI + 用户 |
| 行为差异未发现 | 中 | 运行时验证 + 回归测试 | AI |
| 用户习惯改变 | 低 | 用户手册 + 培训 | 用户 |

## 完成条件

- [ ] 所有"已覆盖"功能验证通过
- [ ] 所有"部分覆盖"功能的差异已明确说明并获得用户确认
- [ ] 所有"不覆盖"功能的废弃原因已明确说明并获得用户确认
- [ ] 用户对功能对照清单进行了审查并确认
