# 人类顾虑汇总（worry.md）

> 目的：把“人类真正担心什么”显式列出，作为后续流程/门禁设计与验真的基线。  
> 主要来源：`prop.md`（已归档合并）（人类/AI关于交付偏差的复盘与讨论）+ 本次对话补充口径。

## 1) 交付偏差：明明说清楚了，交付仍不一致

- 担心点：提案/需求阶段已经讲清楚，但实现/交付仍偏离要求，导致返工与信任受损。
- 典型后果：过早宣称完成；上线后暴露“和要求不一致”的硬偏差。
- 追溯：`prop.md`（已归档合并）开头"根因分析（按阶段）"。

## 2) 负向需求最容易丢（不要/禁止/不允许）

- 担心点：对话里说过的“不要/禁止/不显示/不出现”没有被强制固化为可验收条目，后续阶段最容易被忽略或被“合理化”。
- 典型后果：页面出现不该出现的卡片/文案/字段；角色差异展示规则被破坏。
- 追溯：`prop.md`（已归档合并）"断点 1""负向需求是否完整/是否覆盖""描述卡片"案例。

## 3) 追溯链断裂：proposal → requirements → design → plan → code → test 逐阶段衰减

- 担心点：阶段交接时信息会被“翻译/概括/遗忘”，尤其在 Implementation 按任务推进时，“修一处漏一处”。
- 典型后果：设计与需求无法逐条对齐；实现只满足局部任务而非整页需求基线。
- 追溯：`prop.md`（已归档合并）"根因分析（按阶段）""断点 2/3"。

## 4) 测试通过 ≠ 需求满足（缺少用户视角/UI/角色验收）

- 担心点：构建/单测/技术测试通过，但最在意的“展示规则/文案/禁止项/角色差异”未被逐条验收。
- 典型后果：大量技术测试全绿，但仍存在需求偏差；测试报告看似完备但实际漏项。
- 追溯：`prop.md`（已归档合并）"测试阶段偏重构建和单测""342 个测试通过但仍偏差"的讨论。

## 5) 门禁“查证据存在”而非“覆盖完整”（遗漏无法被拦截）

- 担心点：交付关口只验证 test_report 里有没有证据/结论，而不验证 requirements 里的每条需求/验收是否都被覆盖与判定。
- 典型后果：test_report 覆盖矩阵漏条目也能通过门禁，最终带着遗漏交付。
- 追溯：`prop.md`（已归档合并）"断点 4""W16 不检查 REQ 覆盖率"。

## 6) 自审盲区：同一模型“自己实现自己审”，难以质疑自己的决策

- 担心点：即使强调“要苛刻”，也可能只在技术维度更苛刻，而不会质疑“这个页面元素/文案该不该出现”这种需求维度问题。
- 典型后果：多轮 self-review 仍漏掉需求偏差；生成与审查共享同一认知模型导致结构性盲区。
- 追溯：`prop.md`（已归档合并）关于"描述卡片"与"7 轮 self-review"的复盘。

## 7) 流程成本与上下文开销：不希望为“严谨”付出不可控的负担

- 担心点：如果机制过重（例如一张四列大矩阵、或每个任务重读全量需求），AI 容易敷衍，人类也难以持续执行；上下文还会被无关信息挤爆。
- 典型后果：表格“形式正确但内容空洞”；因上下文爆炸导致实现质量下降。
- 追溯：`prop.md`（已归档合并）关于"矩阵太重易敷衍""每个任务重读 500 行 requirements 上下文爆炸"的讨论。

## 8) “复制”导致不一致：验收清单/追溯表一旦分叉就会过时

- 担心点：把 GWT/验收点复制到 plan/test 等多个位置，会引入“requirements 变了但副本没变”的结构性不一致风险。
- 典型后果：实现/测试按过时清单执行，最终交付偏离最新需求。
- 追溯：`prop.md`（已归档合并）关于"验收清单过时""单一真相源"的讨论。

## 9) 不想每次手工指定审查参数；希望工具无关、自动固化、可复核

- 担心点：人工指定范围/覆盖参数不一定准确，且增加操作负担；同时希望 Claude/Codex 都能一致触发与执行，不用纠结“谁做的审查”。
- 典型后果：因为参数不准导致审查漏项；跨工具执行口径不一致导致争议与返工。
- 追溯：本次对话补充口径（"显示覆盖参数/不想每次指定/能否自动生成固化"）+ `prop.md`（已归档合并）末尾"兼容 codex/claude 的 @review"诉求。

## 10) 口径强硬：说了“不许做/不允许出现”，就必须做到（以最终用户可见为准）

- 担心点：负向约束被解释为“建议/风格偏好”；或被实现者用“设计意图/有用信息”合理化，从而在 UI 上“冒出来”。
- 典型后果：页面出现明确禁止的卡片/字段/文案；即使功能可用也被判定为交付失败。
- 追溯：`prop.md`（已归档合并）的禁止项案例 + 本次对话补充口径（"说了不要做就是不要做""最终用户看不到即可"）。

## 约束偏好（方案必须满足）

- 不新增阶段；复用既有阶段骨架与 `@review`。
- 追溯要“最轻但可验真”：可追溯即可，避免额外负担导致执行走形。
- 质量优先（成本其次）：宁可多一次对账/门禁，也不接受遗漏交付。
