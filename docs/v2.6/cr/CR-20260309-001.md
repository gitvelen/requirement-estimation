## 问题描述

### 背景
用户在内网环境导入技术方案文档时遇到 token 超限错误：
- 内网模型：`Qwen3-32B`，最大上下文长度为 **32,000 tokens**
- 内网 Embedding 模型：`Qwen3-Embedding-8B`
- 当前请求：输入 29,748 tokens + 输出 2,500 = 32,248 tokens，超出之前误判的 32,000 限制
- 错误发生在 `profile_summary_service.py` 的 Stage2 LLM 调用（行 597-602）

### 用户需求
- **不希望文档内容有损失**
- 建议将文档拆分成多块，确保每块在 32,000 tokens 限制以内
- Token 估算使用保守策略（字符数 / 2.5）
- 分块重叠采用按段落边界智能重叠
- **本地开发环境也需配置相同模型**（Qwen3-32B + Qwen3-Embedding-8B）以确保部署一致性

### 当前问题
1. 系统画像 AI 总结服务使用字符级限制（120,000 字符），无 token 计数
2. 超长文档（如 525 段落 + 47 表格）会导致 token 超限
3. 缺少分块调用与结果合并机制

## 技术方案概要

### 核心策略
轻量级 Token 估算 + 段落级分块 + 多轮调用合并

### 分块阈值
- 模型最大上下文：32,000 tokens
- 单次输入上限：25,000 tokens（为输出预留 7,000 tokens）
- Stage1 输出：600 tokens
- Stage2 输出：动态计算（32,000 - 输入 tokens - 1,000 安全边界）

### 分块与降级约束
- 任意 chunk（含重叠段落）必须满足 `chunk_tokens <= 25,000`
- 如果重叠段落会导致超限，必须缩减到 1 个或 0 个重叠段落
- 禁止在 token 估算和分块前先做字符截断
- `ENABLE_LLM_CHUNKING=false` 时，普通文档走单次调用；超长文档显式失败并提示开启分块，不做静默截断

### 合并策略
- **Stage1**：合并 `relevant_domains` 和 `related_systems`（集合去重）
- **Stage2**：深度合并 JSON 对象
  - 列表字段：追加并去重（保持顺序）
  - 字典字段：递归合并
  - 字符串字段：优先保留非空值，如果都非空则拼接（用分号分隔）
  - 特殊处理：`module_structure`、`integration_points`、`key_constraints` 按特定字段去重

### 回滚方案
配置开关 `ENABLE_LLM_CHUNKING`，默认启用；关闭后回退到单次调用模式，超长文档显式失败并提示开启分块，不做字符截断

- 需监控分块数和总调用时间

## 风险评估

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| Token 估算不准确 | 仍可能超限 | 中 | 保守估算（字符数 / 2.5），预留 1,000 tokens 安全边界 |
| 分块导致语义丢失 | 结果不完整 | 低 | 按段落边界智能重叠，保持完整段落 |
| 多轮调用成本增加 | 响应变慢 | 高 | 仅在超限时分块，记录日志监控 |
| JSON 合并逻辑错误 | 数据损坏 | 中 | 充分测试，保留原始响应日志 |

## 验收标准

### 功能验收
1. 使用超长文档样本（覆盖 525 段落 + 47 表格量级）进行后端单元/服务级验证，不出现 token 超限错误
2. 分块后的结果与单次调用结果在语义上一致（允许细节差异）
3. 配置 `ENABLE_LLM_CHUNKING=false` 时，普通文档回退到单次调用；超长文档显式失败并提示开启分块

### 性能验收
1. 正常文档（< 25,000 tokens）不触发分块，响应时间无变化
2. 超长文档分块调用总耗时 < 单次调用超时时间 × 分块数
3. 单次调用 token 数不超过 31,000（接近上限时告警），且任意 chunk 输入不超过 25,000

### 质量验收
1. 单元测试覆盖率 > 80%
2. 后端单元测试与服务级回归通过（覆盖分块上限、降级路径、结果合并）
3. 日志完整记录分块数、token 数、调用时间
