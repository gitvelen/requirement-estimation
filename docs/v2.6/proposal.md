## 一句话总结
通过轻量级 Token 估算 + 段落级分块 + 多轮调用合并，解决内网环境导入超长技术方案文档时的 Token 超限问题，确保文档内容无损失。

---

## 背景与现状

### 问题描述
用户在内网环境导入技术方案文档时遇到 Token 超限错误：
- **内网模型**：`Qwen3-32B`，最大上下文长度为 **32,000 tokens**
- **内网 Embedding 模型**：`Qwen3-Embedding-8B`
- **当前请求**：输入 29,748 tokens + 输出 2,500 = 32,248 tokens，超出之前误判的 32,000 限制
- **错误位置**：`profile_summary_service.py` 的 Stage2 LLM 调用（行 597-602）

### 当前系统限制
1. 系统画像 AI 总结服务使用字符级限制（120,000 字符），无 Token 计数
2. 超长文档（如 525 段落 + 47 表格）会导致 Token 超限
3. 缺少分块调用与结果合并机制
4. 本地开发环境与内网环境模型不一致，可能导致部署后才发现问题

### 用户核心诉求
- **不希望文档内容有损失**（最高优先级）
- 建议将文档拆分成多块，确保每块在 32,000 tokens 限制以内
- Token 估算使用保守策略（字符数 / 2.5）
- 分块重叠采用按段落边界智能重叠
- **本地开发环境也需配置相同模型**（Qwen3-32B + Qwen3-Embedding-8B）以确保部署一致性

## 解决方案概述

### 核心策略
**轻量级 Token 估算 + 段落级分块 + 多轮调用合并**

### 技术方案
1. **新增 Token 计数工具**（`backend/utils/token_counter.py`）
   - 轻量级 token 估算（基于字符启发式：字符数 / 2.5）
   - 从 LLM 响应提取实际 token 数
   - 按 token 限制分块文本（保持段落完整性）

2. **扩展 LLM 客户端**（`backend/utils/llm_client.py`）
   - 新增 `chat_with_chunking()` 方法支持分块调用
   - 实现深度合并逻辑（`_deep_merge()`）
   - 支持 JSON 结果合并和文本拼接两种策略

3. **改进系统画像服务**（`backend/service/profile_summary_service.py`）
   - 重构 `_call_llm()` 方法，引入 token 计数
   - Stage1（域识别）：分块调用并合并域和系统列表
   - Stage2（建议生成）：分块调用并深度合并 JSON 结果
   - 动态调整 `max_tokens`，确保总和不超限

4. **配置扩展**（`backend/config/config.py`）
   - 新增 `LLM_MAX_CONTEXT_TOKENS`（默认 32000）
   - 新增 `LLM_INPUT_MAX_TOKENS`（默认 25000）
   - 新增 `LLM_CHUNK_PRESERVE_PARAGRAPHS`（默认 True）
   - 新增 `ENABLE_LLM_CHUNKING`（默认 True）

5. **环境配置更新**
   - 更新 `.env.backend` 本地开发环境配置：`LLM_MODEL=Qwen3-32B`
   - 更新 `.env.backend.internal` 内网部署配置：补充 token 限制配置说明

### 分块阈值
- 模型最大上下文：32,000 tokens
- 单次输入上限：25,000 tokens（为输出预留 7,000 tokens）
- Stage1 输出：600 tokens
- Stage2 输出：动态计算（32,000 - 输入 tokens - 1,000 安全边界）

### 分块与降级规则
- **任何单块（含重叠段落）都必须满足输入上限**：`chunk_tokens <= 25,000`
- **重叠不能突破预算**：若加入 1-2 个重叠段落后会超限，必须自动缩减为 1 个或 0 个重叠段落
- **禁止先截断再分块**：上下文必须基于完整文档做 token 估算和切块
- **关闭分块开关时不允许静默截断**：超长文档应显式失败并返回可操作提示

### 合并策略
- **Stage1**：合并 `relevant_domains` 和 `related_systems`（集合去重）
- **Stage2**：深度合并 JSON 对象
  - 列表字段：追加并去重（保持顺序）
  - 字典字段：递归合并
  - 字符串字段：优先保留非空值，如果都非空则拼接（用分号分隔）
  - 特殊处理：`module_structure`、`integration_points`、`key_constraints` 按特定字段去重

## 关键验收锚点

### P-DO（必须做到）
1. **超长文档 100% 成功导入**
   - 使用超长文档样本（覆盖 525 段落 + 47 表格量级）进行后端单元/服务级验证，不出现 token 超限错误
   - 验收方式：相关单元测试通过，且断言每个 chunk 不超过输入上限

2. **AI 结果完整性**
   - 分块后的结果与单次调用结果在语义上一致（允许细节差异）
   - 验收方式：人工对比分块前后的系统画像建议，确认关键信息无遗漏

3. **正常文档性能不变**
   - 正常文档（< 25,000 tokens）不触发分块，响应时间无变化
   - 验收方式：性能测试对比 v2.5 和 v2.6 的响应时间

4. **回滚能力**
   - 配置 `ENABLE_LLM_CHUNKING=false` 时，普通文档回退到单次调用；超长文档显式失败并提示开启分块
   - 验收方式：功能测试验证降级逻辑不发生静默截断

### P-DONT（明确禁止）
1. ❌ **不允许文档内容丢失**
   - 分块和降级路径都不能截断或丢弃任何段落
   - 验收方式：对比分块前后的文本长度和段落数

2. ❌ **不允许引入外部依赖**
   - 不引入 tiktoken、transformers 等外部库
   - 验收方式：检查 requirements.txt 无新增依赖

3. ❌ **不允许修改 API 契约**
   - 系统画像 API 的输入输出格式不变
   - 验收方式：API 测试通过

4. ❌ **不允许生成超限 chunk**
   - 任意切块（含重叠段落）都不能超过 `LLM_INPUT_MAX_TOKENS`
   - 验收方式：单元测试校验所有 chunk 的 token 数均不超限

### P-METRIC（关键指标）
1. **Token 超限错误率 = 0**
   - 测试阶段所有超长文档导入成功
   - 数据源：测试日志

2. **单元测试覆盖率 > 80%**
   - 新增代码的单元测试覆盖率达标
   - 数据源：pytest-cov 报告

3. **分块调用总耗时 < 单次调用超时时间 × 分块数**
   - 确保分块不会导致总耗时过长
   - 数据源：性能日志
