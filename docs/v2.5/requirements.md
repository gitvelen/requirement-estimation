## 1. 概述

### 1.1 目的与范围

**目的**：为项目经理提供更清晰、结构化的系统画像五域展示（D1-D5），支持多层级模块结构、实时推送和模板下载，提升知识导入与画像查看体验。

**范围**：
1. **五域展示重构（D1-D5）**：D1 基础信息摘要式展示、D2 多层级模块结构（最大 3 层）、D3 集成点表格展示、D4 技术架构紧凑展示、D5 约束与风险表格展示、AI 建议混合 diff 展示
2. **知识导入模板下载**：提供历史评估报告和 ESB 服务治理文档的 XLSX 模板下载
3. **WebSocket 实时推送**：知识导入完成后前端自动感知并刷新画像数据，连接失败时自动降级为轮询（5 秒间隔，最多 10 次）

**不包含**：
- v2.4 Idea 池中的其他功能（系统识别修正、功能拆分偏好、类型归纳等，延后到 v2.6+）
- 旧数据自动迁移脚本（v2.5 上线后项目经理手动重新导入）
- 功能开关/灰度发布（新功能直接生效）
- D2 虚拟滚动阈值能力（本期延期，待后续版本/CR 评估）

### 1.2 背景、约束与关键假设

**现状与痛点**：
1. D2 模块结构展示不清晰，用户看不出层级关系（模块→子模块→函数）
2. D3 集成点展示松散，用户难以看出字段关系
3. D4 技术架构、D5 约束与风险展示缺乏层次感
4. 知识导入缺少模板下载，用户手动准备文件容易出错
5. 知识导入完成后需手动刷新页面才能看到 AI 建议

**约束**：
- 技术栈：FastAPI（后端）+ React + Ant Design（前端）
- 数据存储：JSON 文件（`data/system_profiles.json`）
- 基线版本：v2.4（已完成系统画像五域 12 子字段重构）
- 兼容性：API 接口保持兼容，仅扩展数据结构（`module_structure` 增加 `children` 字段）；新增模板/任务状态接口同时提供 `/api/v1/system-profiles/*` 主路径与 `/api/system-profile/*` 兼容别名；服务端兼容读取历史 `functions` 结构并归一化为 `children`
- 导航入口：沿用 v2.4 菜单结构，系统画像菜单仅保留 `知识导入` 与 `信息展示` 两个入口，不新增独立 `code-scan` / `workbench` 菜单
- 数据迁移：不提供自动迁移脚本，v2.5 上线后项目经理手动重新导入

**关键假设**：
1. 项目经理熟悉历史评估报告和 ESB 服务治理文档的格式
2. 用户浏览器支持 WebSocket（不支持时自动降级为轮询）
3. 系统画像的模块结构深度不超过 3 层（模块→子模块→函数）

#### REQ-007：WebSocket 实时推送

**测试等级**：L1（核心功能）

**目标/价值**：
- 知识导入完成后，前端自动感知并刷新画像数据
- 提升用户体验，避免手动刷新页面
- 减少等待焦虑

**入口/触发**：
- 后端知识导入任务完成（LLM 处理完成，AI 建议生成完成）

**前置条件**：
- 项目经理已登录
- 项目经理已上传知识文档
- WebSocket 连接已建立

**主流程**：
1. 项目经理在知识导入页触发知识导入任务
2. 系统画像展示页按 `system_name` 建立 WebSocket 连接（`ws://localhost:8000/ws/system-profile/{system_name}`）
3. 后端开始处理知识文档（LLM 提取信息、生成 AI 建议）
4. 后端通过 WebSocket 推送任务状态：
   - `extraction_started`：开始提取
   - `extraction_completed`：提取完成
5. 前端接收 WebSocket 推送，自动刷新系统画像数据
6. 项目经理看到最新的 AI 建议，无需手动刷新页面

**输入/输出**：
- 输入：WebSocket 推送消息（任务状态）
- 输出：更新后的系统画像数据和 AI 建议

**页面与交互**：
- 页面：系统画像展示页（`SystemProfileBoardPage.js`）
- 交互：
  - WebSocket 连接：页面加载时按 `system_name` 自动建立
  - 任务状态推送：后端推送 `extraction_started` / `extraction_completed` / `extraction_failed`
  - 自动刷新：接收到 `extraction_completed` 后自动刷新画像数据

**业务规则**：
- WebSocket 端点：`ws://localhost:8000/ws/system-profile/{system_name}`
- 任务状态枚举值：extraction_started、extraction_completed、extraction_failed
- 心跳保活间隔：30 秒（业界标准，见 REQ-C005）
- 消息延迟要求：< 500ms（见 REQ-102）

**异常与边界处理**：
- 如果 WebSocket 连接失败，自动降级为轮询模式（见 REQ-008）
- 如果后端处理失败，推送 `extraction_failed` 状态，前端显示错误提示
- 如果心跳超时（> 30 秒无响应），前端自动重连

**验收标准（GWT）**：

**GWT-REQ-007-01**：WebSocket 实时推送
- **Given** 项目经理已打开系统画像展示页，且 WebSocket 连接已建立
- **When** 知识导入完成（后端推送 `extraction_completed`）
- **Then** 前端自动刷新系统画像数据，项目经理看到最新的 AI 建议

**关联**：
- 关联场景：SCN-008
- 关联 Proposal 锚点：P-DO-06
- 关联 CR：CR-20260305-001

#### REQ-008：WebSocket 连接失败降级为轮询

**测试等级**：L1（核心功能）

**目标/价值**：
- WebSocket 连接失败时，自动降级为轮询模式
- 保证功能可用性，避免用户无法感知任务完成
- 提升系统健壮性

**入口/触发**：
- WebSocket 连接失败（网络环境不支持或连接超时）

**前置条件**：
- 项目经理已登录
- 项目经理已上传知识文档
- WebSocket 连接失败（超时或网络错误）

**主流程**：
1. 项目经理在知识导入页面触发知识导入任务，前端尝试建立 WebSocket 连接
2. 连接失败（超时或网络错误）
3. 前端自动降级为轮询模式
4. 前端每 5 秒调用一次后端接口 `GET /api/v1/system-profiles/task-status/{task_id}`（兼容别名：`GET /api/system-profile/task-status/{task_id}`）
5. 后端返回任务状态（extraction_started/extraction_completed/extraction_failed）
6. 如果任务完成，前端自动刷新系统画像数据
7. 如果任务失败，前端显示错误提示并停止轮询

**输入/输出**：
- 输入：任务 ID（task_id）
- 输出：任务状态（extraction_started/extraction_completed/extraction_failed）

**页面与交互**：
- 页面：知识导入页面（`SystemProfileImportPage.js`）
- 交互：
  - 轮询间隔：5 秒
  - 轮询超时：10 次（50 秒）
  - 任务完成：自动刷新画像数据
  - 任务失败：显示错误提示并停止轮询

**业务规则**：
- 轮询间隔：5 秒
- 轮询超时：10 次（50 秒）
- 任务状态枚举值：extraction_started、extraction_completed、extraction_failed
- 轮询超时后显示"任务处理超时，请稍后手动刷新"提示并停止轮询

**异常与边界处理**：
- 如果轮询超过 10 次（50 秒）仍未完成，前端显示"任务处理超时，请稍后手动刷新"提示并停止轮询
- 如果后端接口返回 404（任务不存在），前端显示"任务不存在"提示并停止轮询
- 如果后端接口调用失败，前端显示"查询失败，请稍后重试"提示

**验收标准（GWT）**：

**GWT-REQ-008-01**：WebSocket 连接失败降级为轮询
- **Given** 项目经理在知识导入页面，WebSocket 连接失败
- **When** 前端自动降级为轮询模式（5 秒间隔）
- **Then** 前端每 5 秒调用一次任务状态接口，任务完成后自动刷新画像数据

**关联**：
- 关联场景：SCN-009
- 关联 Proposal 锚点：P-DO-07
- 关联 CR：CR-20260305-001

### 4A.4 技术约束明细

#### REQ-C005：WebSocket 心跳保活间隔

**约束类型**：技术约束

**约束内容**：
- WebSocket 心跳保活间隔：30 秒（业界标准）
- 心跳超时（> 30 秒无响应）时，前端自动重连

**理由**：
- 保证 WebSocket 连接稳定性
- 避免长时间无数据传输导致连接断开
- 30 秒是业界标准，平衡连接稳定性和网络开销

**验收标准（GWT）**：

**GWT-REQ-C005-01**：WebSocket 心跳保活间隔
- **Given** WebSocket 连接已建立
- **When** 30 秒内无数据传输
- **Then** 前端发送心跳消息（ping），后端响应（pong），连接保持

**关联**：
- 关联 Proposal 开放问题：Proposal 开放问题 #1
- 关联 CR：CR-20260305-001

## 6. 数据与接口

### 6.1 数据结构变更

#### 6.1.1 module_structure 数据结构（新增 children 字段）

**v2.4 数据结构**：
```json
{
  "module_structure": [

```

**v2.5 数据结构**：
```json
{
  "module_structure": [
    {
      "module_name": "用户管理模块",
      "description": "负责用户注册、登录、权限管理",
      "children": [
        {
          "module_name": "用户注册子模块",
          "description": "处理用户注册流程",
          "children": [
            {
              "module_name": "注册表单验证",

              "module_name": "发送验证邮件",

          ]
        },
        {
          "module_name": "用户登录子模块",
          "description": "处理用户登录流程",
          "children": [
            {
              "module_name": "登录凭证验证",

```

**变更说明**：
- 新增 `children` 字段，支持递归（最大 3 层）
- 新写入口径统一使用 `children`；历史 `functions` 结构在服务端读取时归一化为 `children`
- 兼容性：API 接口保持兼容，既支持新结构，也兼容读取历史模块结构数据

### 6.2 API 接口变更

#### 6.2.1 新增接口

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 模板下载 | GET | 主路径：`/api/v1/system-profiles/template/{template_type}`；兼容别名：`/api/system-profile/template/{template_type}` | 下载知识导入模板（history_report / esb_document） |
| 任务状态查询 | GET | 主路径：`/api/v1/system-profiles/task-status/{task_id}`；兼容别名：`/api/system-profile/task-status/{task_id}` | 查询知识导入任务状态（轮询降级使用） |
| WebSocket 连接 | WS | `/ws/system-profile/{system_name}` | 建立 WebSocket 连接，接收任务状态推送 |

#### 6.2.2 接口详细说明

**接口 1：模板下载**

- **路径**：主路径 `GET /api/v1/system-profiles/template/{template_type}`；兼容别名 `GET /api/system-profile/template/{template_type}`
- **请求参数**：
  - `template_type`（路径参数）：模板类型，枚举值 `history_report` / `esb_document`
- **鉴权**：Bearer Token（JWT），仅 `manager` 可访问
- **响应**：
  - 成功（200）：返回 XLSX 文件流
  - 失败（400）：模板类型无效
  - 失败（404）：模板文件不存在
- **示例**：
  ```bash
  curl -X GET "http://localhost:8000/api/v1/system-profiles/template/history_report" \
    -H "Authorization: Bearer <token>" \
    --output 工作量评估模板.xlsx
  ```

**接口 2：任务状态查询**

- **路径**：主路径 `GET /api/v1/system-profiles/task-status/{task_id}`；兼容别名 `GET /api/system-profile/task-status/{task_id}`
- **请求参数**：
  - `task_id`（路径参数）：任务 ID
- **鉴权**：Bearer Token（JWT），仅 `manager` 可访问
- **响应**：
  - 成功（200）：返回任务状态
    ```json
    {
      "task_id": "task-123",
      "system_id": "sys_001",
      "status": "extraction_completed",
      "system_name": "CRM系统",
      "updated_at": "2026-03-06T10:00:00Z",
      "error": null
    }
    ```
  - 失败（404）：任务不存在
- **示例**：
  ```bash
  curl -X GET "http://localhost:8000/api/v1/system-profiles/task-status/task-123" \
    -H "Authorization: Bearer <token>"
  ```

**接口 3：WebSocket 连接**

- **路径**：`WS /ws/system-profile/{system_name}`
- **请求参数**：
  - `system_name`（路径参数）：系统名称
- **连接要求**：沿用现有登录态/JWT；未登录或角色不满足时拒绝连接
- **消息格式**：
  ```json
  {
    "event": "extraction_started",
    "system_name": "CRM系统",
    "timestamp": "2026-03-06T10:00:00Z"
  }
  ```
- **事件类型**：
  - `extraction_started`：开始提取
  - `extraction_completed`：提取完成
  - `extraction_failed`：提取失败
  - `ping`：客户端心跳保活
  - `pong`：服务端心跳应答
- **示例**：
  ```javascript
  const ws = new WebSocket('ws://localhost:8000/ws/system-profile/CRM系统?token=<token>');
  ws.onmessage = (event) => {
    const data = JSON.parse(event.data);
    console.log('收到推送:', data);
  };
  ```

### 6.3 数据迁移策略

**迁移方式**：手动重新导入

**迁移步骤**：
1. v2.5 上线后，项目经理访问知识导入页面
2. 下载模板（历史评估报告 / ESB 服务治理文档）
3. 按照模板格式填写数据
4. 上传文件，系统自动提取信息并生成 AI 建议
5. 项目经理查看 AI 建议，采纳或忽略
6. 保存系统画像

**注意事项**：
- v2.4 旧数据不会自动迁移到 v2.5
- 项目经理需手动重新导入历史评估报告和 ESB 文档
- 建议在 v2.5 上线前备份 v2.4 数据（`data/system_profiles.json`）
