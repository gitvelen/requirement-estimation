## 2. 变更点（What）
### 2.1 本次逆向走查确认的实现缺口（按 REQ / Design 对齐）
- **本轮澄清结论（2026-03-06）**：
  - `REQ-006`：模板下载按钮放在对应文档类型卡片内，与导入文档操作保持同一行。
  - `REQ-C006`：虚拟滚动阈值本期不做；相关 requirements / design / plan / test_report 口径需在后续阶段统一调整，不再作为本 CR 的实现缺口。

1. **D2 递归树与折叠交互未落地（REQ-002 / REQ-003 / REQ-C003）**
   - 需求/设计目标：`module_structure.children` 三层递归展示、缩进 20px、层级线 4px、默认展开第 1 层、第 2/3 层支持折叠。
   - 当前实现：前端仍按旧 `functions` 二层结构渲染与保存，未消费 `children`，也未实现折叠/展开、超深度提示。
   - 现状证据：`frontend/src/pages/SystemProfileBoardPage.js` 仍使用 `normalizeModuleStructureDraft()` / `sanitizeModuleStructureForPersist()` 的 `functions` 草稿模型；全文无 `children` / `expand` / `collapse` 相关实现。

2. **D3 集成点表格未落地（REQ-004）**
   - 需求/设计目标：以表格展示 `peer_system`、`protocol`、`direction`、`description`，并对方向做 `→ / ← / ⇄` 映射。
   - 当前实现：仍是卡片+输入框编辑形态，没有表格列结构，也没有方向图标/枚举映射。

3. **D1 / D4 / D5 结构化展示未完整落地（REQ-001）**
   - 需求/设计目标：D1 摘要式分组、D4 紧凑卡片+关键指标、D5 约束与风险表格（至少包含约束类型、内容、风险等级、应对措施）。
   - 当前实现：页面仍以通用表单控件为主；D5 仅有 `category + description` 与 `known_risks` 列表，缺少设计要求的表格列结构与清晰化呈现。

4. **AI 建议混合 diff 未落地（REQ-005）**
   - 需求/设计目标：列表字段行内 `+/-/~` 标记、文本字段左右双栏、表格字段行级标记、`module_structure` 树形 diff。
   - 当前实现：统一使用“当前值 / AI 建议”通用预览卡片，仅能做泛化对象预览，未提供列表差异标记、表格行级高亮与树形 diff。

5. **导入页模板下载未落地（REQ-006 / REQ-103）**
   - 需求/设计目标：模板下载按钮放在对应文档类型卡片内，并与文档选择/导入操作保持同一行；调用 API-001 / API-001A 下载历史评估报告与 ESB 服务治理文档模板。
   - 当前实现：导入页仅支持上传导入；页面内不存在模板下载按钮，也没有 `/template/{template_type}` 调用。
   - 说明：后端模板下载接口已存在，本缺口主要在前端接入与验证闭环。

6. **WebSocket 实时推送链路未落地（REQ-007 / REQ-102 / REQ-C005）**
   - 需求/设计目标：前端按 `system_name` 建立 WS，收到 `extraction_completed` 自动刷新画像；客户端按 30 秒心跳保活。
   - 当前实现：前端页面未建立任何 WebSocket 连接，也没有 `ping/pong` 心跳与消息延迟验证逻辑。
   - 说明：后端 `/ws/system-profile/{system_name}`、`ping/pong` 与广播入口已存在，本缺口主要在前端接入与验收证据。

7. **轮询降级语义未按设计落地（REQ-008）**
   - 需求/设计目标：WS 失败时自动降级到 API-002 `task-status/{task_id}`，5 秒一次，最多 10 次，终态即停止。
   - 当前实现：前端仍调用旧 `/profile/extraction-status`，轮询间隔为 3 秒，未限制最大 10 次，也不是由 WS 失败触发。

## 5. 风险与回滚
- **风险**：
  1. BoardPage 结构化改造可能影响现有保存/发布链路。
  2. 导入页接入 WS/轮询双链路后，若状态机设计不当可能导致重复刷新或提示抖动。
  3. 若不先修复测试与 `test_report` 漂移，Deployment 会形成“实现与证据双缺口”的假收敛。
- **监控/告警**：前端单测 + 构建 + STAGING 手工链路；重点观察模板下载成功率、WS 建连成功率、轮询终止条件，以及 D2 递归树交互稳定性。
- **回滚条件**：
  1. BoardPage 首屏崩溃或保存/发布能力异常。
  2. ImportPage 模板下载不可用，或 WS/轮询双链路均不可用。
  3. 补齐后回归导致 `v2.4` 基线功能（导航、导入、查看）受损。
- **回滚步骤**：
  1. 优先回退本 CR 的前端页面与测试改动，保留已验证的后端模板/WS/任务状态接口。
  2. 如问题不可快速收敛，则维持 `v2.5` 在 `wait_confirm` / 不准出状态，必要时退回 `v2.4` 页面行为。
  3. 回滚后验证：导入页、信息展示页、导航回归、基础导入/查看链路恢复可用。

## 6. 验收与验证
### 验收标准（GWT 格式，🔴 MUST 至少一条）
1. **D2 递归树闭环**
   - Given 系统画像包含三层 `module_structure.children`，When 用户查看 D2，Then 页面按树形递归渲染并支持折叠/展开。
2. **D3 / D5 结构化展示闭环**
   - Given 系统画像包含集成点与约束/风险数据，When 用户查看 D3 / D5，Then 页面按设计定义的表格列与图标/字段语义展示。
3. **AI diff 闭环**
   - Given AI 建议包含列表、文本、表格、树形结构变化，When 用户查看 diff，Then 对应字段按混合 diff 方案展示，不再退化为通用对象预览。
4. **导入页链路闭环**
   - Given 用户在知识导入页，When 用户在对应文档类型卡片中点击下载模板或等待导入完成，Then 可下载两类模板，并可通过 WS 自动刷新；若 WS 失败则改用 5 秒一次、最多 10 次的 `task-status` 轮询。
5. **证据闭环**
   - Given `test_report.md` 声明某项 GWT 通过，When 运行对应测试命令，Then 能在真实测试文件与输出中定位到一一对应的证据。
