### RVW-010（P1）系统画像工作台重构未同步到既有场景/需求入口描述，导致“按钮在哪个页面”口径不一致
- **证据**：
  - `SCN-001/SCN-002` 仍描述“在系统画像页面触发/上传”（示例：`rg -n "在系统画像页面|系统画像页面 →" docs/v2.0/requirements.md` 可命中多处）
  - `REQ-001` 入口仍为“系统画像页面 → 代码扫描按钮”，`REQ-003` 入口仍为“系统画像页面 → ESB导入按钮”
- **风险**：
  - 实现阶段可能继续把“导入/扫描”放在单页或放错页，造成与 REQ-022 的“知识导入/信息看板分离”不一致
  - 后续测试用例与验收口径难对齐（到底在哪个路由/菜单验证）
- **建议修改**（需求层面最小修复）：
  1. 将 `SCN-001/SCN-002/REQ-001/REQ-003` 的入口/触发统一改为“配置管理 → 系统画像 → 知识导入”
  2. 将 `REQ-004` 中“手动：项目经理在系统画像管理页面保存草稿/点击发布”统一明确为“信息看板页面”执行（保持功能不变，仅调整入口表述）
  3. 若仍保留“系统画像管理页面”这一称呼，建议在术语处定义它由“知识导入+信息看板”组成，避免误解为单页
- **验证方式（可复现）**：
  - [ ] `rg -n "系统画像页面 →|在系统画像页面" docs/v2.0/requirements.md` 仅在“历史/说明”处出现，不再作为入口口径出现
  - [ ] `rg -n "知识导入" docs/v2.0/requirements.md` 可覆盖到 SCN-001/002 与 REQ-001/003 的入口描述

### RVW-011（P1）REQ-022/SCN-015 对“导入记录/任务列表”的要求自相矛盾，且术语不精确
- **证据**：
  - `REQ-022` 主流程写“每个区块显示最近扫描任务列表（仅显示最近3条）”，但业务规则又写“**不再显示导入历史记录**（保持页面简洁）”
  - 同一段落里，“最近扫描任务列表”被用于 ESB 导入与知识导入区块，语义不准确
- **风险**：
  - 交互实现会在“完全不展示历史”与“展示最近3条”之间摇摆，导致 UI 来回返工
  - “扫描/导入/入库”三类动作的反馈口径不清，验收难以判定
- **建议修改**（二选一明确口径）：
  - 方案A（更贴近“保持简洁”）：每个区块仅展示“最近一次任务状态/结果摘要”（不展示列表）；必要时仅提供“查看详情”跳转到任务详情页
  - 方案B（保留轻量历史）：允许展示“最近N条任务/导入记录”（N=3），但需统一命名为“最近任务/最近导入记录”，并明确每个区块展示的记录类型（代码扫描job / ESB导入批次 / 文档导入批次）
- **验证方式（可复现）**：
  - [ ] `rg -n "不再显示导入历史记录|最近扫描任务" docs/v2.0/requirements.md` 仅保留一种口径，且术语与区块匹配

## 关键发现（按优先级）

### RVW-017（P1）API-016 文件格式白名单与 REQ-027 不完全对齐
- **证据**：API-016（[requirements.md:3295](docs/v2.0/requirements.md#L3295)）请求参数描述为"支持：`.docx/.doc/.xls`"，但 REQ-027 的入口/触发明确包含"知识导入（系统画像）：ESB导入、知识库文档导入"。API-016 仅覆盖"发起评估"场景的任务创建上传，而 `.xls` 在发起评估场景中的用途不明确（评估任务通常上传需求文档，而非 Excel 表格）。同时 proposal 附录B B.4.10 原文为"发起评估（UploadPage）：新增支持 .doc（Word 97-2003）和 .xls（Excel 97-2003）格式"，但从业务语义看，发起评估上传 `.xls` 的场景需要进一步澄清。
- **风险**：实现时可能对 API-016 的 `.xls` 支持产生歧义（是否需要解析 Excel 为需求文本？还是仅作为附件？），导致前后端对接不一致。
- **建议修改**：在 API-016 说明中补充 `.xls` 在"发起评估"场景下的解析策略（如：作为需求附件上传，按行/列拼接为文本输入AI解析），或在 REQ-027 中明确 `.xls` 仅适用于知识导入/ESB导入场景而非发起评估。
- **验证方式**：`rg -n "\.xls" docs/v2.0/requirements.md` 检查所有 `.xls` 出现位置的语义一致性。

### RVW-018（P1）REQ-024 AI总结触发时机与 ProfileSummaryService 的异步机制未定义错误重试策略
- **证据**：REQ-024（[requirements.md:1935](docs/v2.0/requirements.md#L1935)）主流程第1步描述"系统创建并异步执行'画像AI总结'任务"，失败处理仅提到"AI总结失败不影响导入与画像草稿更新；应发送失败通知并允许重试（可选）"。但"重试"标记为"可选"，且未定义：(a) 重试是自动还是手动？(b) 重试次数上限？(c) 失败通知的通知类型（是否复用 API-017 的 notification.type？）。CR-20260209-001 第3.3节提到后端新增 `ProfileSummaryService`，但 requirements 中未给出该服务的错误码或失败场景的 API 契约。
- **风险**：Design/Implementation 阶段可能对"AI总结失败"的处理方式产生分歧（静默失败 vs 通知+手动重试 vs 自动重试），影响用户体验一致性。
- **建议修改**：(1) 将"允许重试（可选）"改为明确策略，建议"失败后发送失败通知（type=`system_profile_summary_failed`），PM可在信息看板手动触发重试"；(2) 在错误码表中补充 AI 总结失败的错误码（如 `SUMMARY_001`）或明确复用 `EMB_001`。
- **验证方式**：检查 REQ-024 失败处理段落是否包含明确的重试策略与错误码引用。

### RVW-019（P2）SCN-008 追溯矩阵中 REQ-026 关联 API 缺失
- **证据**：追溯矩阵（[requirements.md:3346](docs/v2.0/requirements.md#L3346)）中 SCN-008 关联 `REQ-010, REQ-023, REQ-026`，关联 API 仅为 `API-010`。但 REQ-026（管理员任务详情布局重设计）的页面数据来源应包含 API-008（专家偏离度）、API-013（评估详情）、API-014（报告下载），这些 API 在 REQ-026 的主流程中隐含使用（偏离度统计、评估进度、报告下载按钮）。
- **风险**：Design 阶段可能遗漏 REQ-026 对 API-008/013/014 的依赖，导致页面数据接口不完整。
- **建议修改**：在追溯矩阵 SCN-008 行的"关联API"列补充 `API-008, API-013, API-014`（或在 REQ-026 的"关联"字段中显式列出）。
- **验证方式**：检查 REQ-026 关联字段与追溯矩阵 SCN-008 行的 API 列表是否一致。

### RVW-020（P2）API-017 通知清理策略未定义留存期
- **证据**：API-017（[requirements.md:3311](docs/v2.0/requirements.md#L3311)）定义了 `DELETE /api/v1/notifications/clear-read`（清理已读）和 `DELETE /api/v1/notifications/{notification_id}`（删除单条），但数据字典中 notification.* 字段的留存期标记为"永久"（[requirements.md:2408-2413](docs/v2.0/requirements.md#L2408-L2413)）。通知数据如果永久留存且无自动清理机制，长期运行后可能产生存储膨胀。
- **风险**：低。当前为 JSON 文件存储，数据量有限；但作为需求完整性建议补充。
- **建议修改**：在 API-017 说明或 REQ-024 业务规则中补充通知留存策略（如"已读通知默认保留90天后自动清理"或"由用户手动清理"），并更新数据字典留存期。
- **验证方式**：检查 notification.* 数据字典的留存期字段。

### RVW-021（P2）REQ-022 知识导入"仅展示负责系统"的判定口径未显式定义
- **证据**：CR-002 要求"只显示当前用户作为负责人的系统在顶部 TAB 页"，REQ-022 更新后应包含此约束。但需确认"负责人"的判定口径：是系统清单中的"系统负责人（主责）"还是包含B角？根据 proposal 和术语定义，统计口径仅使用主责，但知识导入的"可见系统"是否也仅限主责需要明确。
- **风险**：如果B角也需要导入知识（如主责不在时），当前口径可能导致B角无法操作。
- **建议修改**：在 REQ-022 的业务规则中显式说明"系统Tab仅展示当前用户为**主责**的系统（不含B角）"，或根据业务需要扩展为"主责或B角"。
- **验证方式**：`rg -n "负责系统|负责人" docs/v2.0/requirements.md` 检查口径一致性。
