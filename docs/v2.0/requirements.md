## 1. 概述

### 1.1 目的与范围

本文档定义需求分析与评估系统 v2.0 版本的功能性与非功能性需求。

**目的**：
- 提升功能点拆分和工作量估算的准确性
- 建立系统画像机制，让AI全面了解系统现状
- 实现学习闭环，持续优化评估效果
- 优化管理层/项目经理/专家的效能展示与交互体验（含效能看板）

**范围（In Scope）**：
- 系统画像数据结构与管理（代码扫描、ESB集成、人工维护；文档作为知识库导入供AI检索）
- AI评估时检索系统画像
- 项目经理修改轨迹记录与分析
- 复杂度三维度评估模型（业务规则35% + 集成点35% + 技术难度30%）
- 效能看板（效能展示）：材料完整度、角色处理量、系统改造、人天与AI准确率等指标的趋势/排行/下钻
- 界面体验优化（时间格式、COSMIC简化、任务管理交互统一、知识导入/信息看板体验优化等）
- **增量（CR-20260209-001）**：多角色切换、任务管理Tab统一、知识导入仅展示负责系统、信息看板AI总结+通知+采纳/忽略、编辑功能点页面优化、管理员任务详情布局重设计、系统清单双sheet导入、规则管理平铺Tab、旧格式文件（.doc/.xls）支持、专家查看权限缺陷修复

**不包含（Out of Scope）**：
- COSMIC算法逻辑重构
- 系统画像可视化展示（能力树/关系图/架构图等）
- AI使用提示（标注AI评估时“参考了哪些信息”）
- 优化建议生效治理机制（版本号/一键回滚/管理员审批）
- B角（代理负责人）纳入统计口径/排行榜（统计仍仅使用主责）
- 文档技术特征结构化自动抽取（文档仅用于知识库检索与证据引用）
- 新的AI模型训练
- 实际工时收集与校准
- 数据库迁移（复用现有JSON文件存储）
- 移动端适配
- .ppt 旧格式支持（使用频率低，Non-goal）
- JWT token 中嵌入角色信息（保持按请求时读取角色列表）
- 通用“消息通知中心”平台化能力（本次仅新增“画像AI总结完成/失败通知”场景）

### 1.2 背景、约束与关键假设

**现状与痛点**：
1. **功能点拆分没有方法论**：拆分粒度凭感觉，虚增比遗漏更严重
2. **对系统了解不全面**：AI不知道系统能力、集成关系、技术债务
3. **复杂度评估粗糙**：仅用"高/中/低"三个标签，缺乏多维度评估
4. **需求文档质量差**：信息缺失、隐含复杂度、边界模糊

**约束**：
- **技术栈**：React + FastAPI + Ant Design
- **部署**：Docker Compose
- **时间**：本期一次性完成所有功能（不分期），通过并行开发交付
- **人员**：现有团队，不增加人力
- **数据存储**：复用现有JSON文件存储

**关键假设**：
- ESB治理文档Excel样例可在 Planning 阶段提供
- 代码扫描环境可访问（默认服务端可访问本地 `repo_path`；`repo_archive` 上传压缩包作为备选）
- 无历史实际工时数据可用于校准

### 1.3 术语与口径

| 术语 | 定义 |
|---|---|
| **系统画像（System Profile）** | 系统的能力清单、集成关系、技术特征的结构化描述，来源于代码扫描、ESB清单与人工维护（文档作为证据与知识库检索输入） |
| **学习闭环（Learning Loop）** | 记录PM修改 → 分析模式 → AI优化建议的闭环机制 |
| **功能点拆分准确性** | 通过PM修正率衡量（见下条PM修正率定义） |
| **PM修正率** | 发布后30天内统计：PM对AI初始功能点的（删除+调整）操作数/AI初始功能点数（不含新增） |
| **PM新增数（AI遗漏）** | 发布后30天内统计：新增操作数（不纳入PM修正率；用于衡量AI遗漏） |
| **复杂度三维度评估** | 业务规则复杂度35% + 集成点数量35% + 技术难度30% |
| **L0层级** | 历史评估文档的最低参考层级，质量最低，AI检索时降低权重 |
| **COSMIC混合方式** | 后台保持COSMIC方法论，前端用业务语言展示 |
| **效能看板（Efficiency Dashboard）** | 面向管理层/PM/专家的效能展示页，统一用同一套指标口径展示趋势、TOP榜单与任务明细下钻 |
| **活跃角色（activeRole）** | 多角色用户在前端选择的当前角色；用于决定菜单展示与默认数据范围（不改变后端JWT/资源级授权） |
| **字段来源（field_source）** | 系统画像字段当前值的来源标记：`manual`（人工编辑）/`ai`（采纳AI总结）；用于避免后台任务自动覆盖人工编辑结果 |
| **系统改造（评估口径）** | 以“评估结论里要改的系统”为准（不依赖实际落地改造系统） |
| **系统改造次数** | 在统计窗口内，某系统被纳入“最终评估系统拆分人天”的评估任务数量（去重按任务） |
| **系统改造工作量（人天）** | 在统计窗口内，某系统在“最终评估系统拆分人天”中的人天汇总 |
| **AI评估准确率（对齐最终评估）** | 以“AI初稿人天/结论”对齐“最终评估人天/结论”计算的误差（不涉及实际消耗） |
| **命中率（Hit Rate）** | `abs(AI-最终)/最终 ≤ 阈值` 的占比（默认阈值=20%）；当最终<1人天时按绝对误差阈值（详见6.3） |
| **MAE（人天）** | `mean(abs(AI-最终))`（以人天为单位） |
| **Bias（偏差）** | `mean(AI-最终)`，>0表示系统性高估，<0表示系统性低估 |
| **评估快照（Owner Snapshot）** | 在评估冻结节点写入“系统负责人/涉及系统/系统拆分人天”等快照；统计默认使用快照，避免系统清单变更导致历史漂移 |
| **B角（代理负责人）** | 系统清单中为主责提供代理操作权限的负责人；用于知识导入/草稿编辑/AI总结重试等“系统画像草稿侧”操作。判定口径：`user_id==owner_id` 或 `user_id ∈ backup_owner_ids`；统计/排行榜仍仅使用主责（owner_id） |

**ID 前缀规则**：
- 统一定义见 `.aicoding/STRUCTURE.md`（本文件不重复定义，仅补充本项目用到的前缀）：
  - 场景：SCN-xxx
  - 功能需求：REQ-xxx（编号不强制按“核心/体验”分段；以“需求分类”列为准）
  - 非功能需求：REQ-NF-xxx
  - API需求：API-xxx

## 2. 业务场景说明

### 2.1 角色与对象

**角色**：
- **管理员**（Admin）：系统配置、数据管理、查看效能看板
- **管理层/只读查看者**（Viewer）：查看效能看板与排行榜（只读，不做配置）
- **项目经理**（PM）：创建评估任务、管理知识库、修改AI评估结果
- **专家**（Expert）：参与评估、提供估算意见、下载报告
- **系统**（System）：AI评估、代码扫描、文档解析
> 说明：系统清单中维护“系统负责人（主责）”与B角（代理负责人）。本期统计口径与排行榜**仅使用主责**；B角不纳入统计（但允许作为代理负责人执行知识导入/草稿编辑/重试AI总结等操作）。
> 系统画像相关分工：管理员仅负责系统清单导入（REQ-019）；项目经理负责开启代码扫描（REQ-001/REQ-002）、导入ESB治理文档（REQ-003）、导入技术文档与历史评估文档L0（REQ-011）以丰富系统画像。

**对象**：
- **系统画像**（System Profile）：系统能力、集成关系、技术特征的结构化描述
- **评估任务**（Task）：需求评估的完整流程记录
- **功能点**（Feature）：需求拆分后的最小可估算单元
- **修改轨迹**（Modification Trace）：PM对AI评估结果的修改记录
- **知识条目**（Knowledge）：代码扫描、文档、ESB导入的知识片段
- **效能看板**（Dashboard）：按统一口径聚合展示趋势/排行/下钻的效能数据集合
- **消息通知**（Notification）：系统向用户推送的通知消息（如画像AI总结完成/失败）

### 2.3 场景明细

#### SCN-001：代码扫描构建系统画像
**场景分类**：系统画像
**主要角色**：项目经理
**相关对象**：系统画像、代码仓库
**关联需求ID**：REQ-001, REQ-002
**前置条件**：
- 用户已登录且具有PM权限
- 当前用户为目标系统主责或B角（系统清单 owner_id=当前用户 或 `backup_owner_ids` 包含当前用户）
- 目标代码仓库可访问（服务端本地路径或Git URL）；或可上传代码仓库压缩包（备选）

**触发条件**：
- 项目经理在“配置管理 → 系统画像 → 知识导入”触发代码扫描，或调用 API-001 发起扫描任务

**流程步骤**：
1. 项目经理输入系统名称，并提供代码仓库来源（二选一）：
   - `repo_path`：本地目录或 Git URL
   - `repo_archive`：上传代码仓库压缩包（备选）
2. 项目经理选择扫描选项（扫描路径、排除目录）
3. 后端系统创建扫描任务（job），返回 `job_id`
4. 后台扫描代码，提取HTTP API、定时任务、消息监听器、外部调用等元数据
5. 扫描完成后生成结果文件，并将能力条目入库到向量库（knowledge_type=capability_item）
6. 更新系统画像草稿（字段/证据引用/完整度评分），供后续发布

**输出产物**：
- 代码扫描结果JSON文件
- 能力清单（HTTP API、定时任务、消息监听器）
- 集成关系提示（Feign、Dubbo、HTTP Client、MQ）
- 更新后的系统画像（后端数据）

**异常与边界处理**：
- 扫描失败：返回错误信息，任务状态标记为failed
- 仓库路径无效：提示用户检查路径
- 扫描超时：任务状态标记为timeout，可重新扫描

**说明**：扫描为后台异步处理流程；前端仅负责触发与展示任务状态/结果摘要。

#### SCN-003：系统画像发布与管理（含AI总结）
**场景分类**：系统画像
**主要角色**：项目经理/系统
**相关对象**：系统画像、证据引用
**关联需求ID**：REQ-004, REQ-024
**前置条件**：
- 已存在至少一种系统材料来源（代码扫描/ESB/知识库文档）
- 项目经理已登录且具有PM权限，且为目标系统主责或B角（系统清单 owner_id=当前用户 或 `backup_owner_ids` 包含当前用户）

**触发条件**：
- 数据变更触发系统画像草稿更新（代码扫描完成/ESB导入完成/文档导入完成）
- 项目经理在“配置管理 → 系统画像 → 信息看板”查看/编辑系统画像草稿并发布

**流程步骤**：
1. 系统汇总代码扫描、ESB、文档等多源数据，生成/更新系统画像**草稿**（`status=draft`）
2. 系统异步触发“画像字段AI总结”任务（不阻塞导入/更新流程）：
   - 以导入的材料为输入生成7个画像字段的**候选建议**
   - 不自动覆盖 `field_source=manual` 的字段当前值
   - 总结完成后向主责PM发送“画像AI总结完成”通知（可跳转到信息看板）；失败则发送“画像AI总结失败”通知并提供手动重试入口（见 REQ-024 / API-018）
3. 计算完整度评分（代码30% + 文档40% + ESB30%），并给出 `pending_fields`
4. 项目经理在“信息看板”查看草稿（字段表单 + 证据引用列表 + AI建议），可补充/纠错并保存
5. 项目经理点击发布（或系统在草稿满足发布必填字段时自动发布）
6. 发布时生成画像文本embedding并写入向量库（knowledge_type=system_profile），更新 `status=published` 与 `published_at`
7. 系统每周执行过时检查：超过阈值（默认30天）标记 `is_stale=true`

**输出产物**：
- 草稿/已发布的系统画像（含字段、证据引用、完整度评分）
- AI字段建议与来源标记（`field_source`）
- “画像AI总结完成/失败”通知
- 画像完整度评分
- 画像元数据（更新时间、发布时间、状态、是否可能过时）

**异常与边界处理**：
- 草稿缺少发布必填字段：禁止发布并提示缺失字段（`PROFILE_003`）
- embedding服务不可用：发布失败，记录错误可重试（`EMB_001`）
- AI总结失败：不影响系统画像草稿更新与发布；通知中心提示失败原因（`notification.type=system_profile_summary_failed`，含 `error_code/error_reason`），PM可在信息看板手动触发重试（API-018）

**说明**：系统画像主要用于AI检索；前端仅提供“列表/详情/字段表单 + 证据引用”的管理能力，不做能力树/关系图等可视化展示。

   - 三个导入区域采用横向三栏卡片布局：代码扫描、ESB导入、知识导入（每个卡片包含图标、标题、上传区域与状态指示）
   - 不展示导入历史/最近任务列表；仅在操作后展示当前任务状态/导入统计作为反馈
3. 项目经理进入"信息看板"页面：
   - 顶部系统TAB与知识导入页保持一致并同步
   - 显示系统概览（系统名称、ID、状态、完整度、最后更新时间）
   - 显示可编辑的画像字段（与REQ-004字段一致）：系统边界（做什么）、系统不做什么、核心功能、业务目标、业务对象、主要集成点、关键约束
   - 若存在AI总结建议，则逐字段展示建议并提供“采纳/忽略”（不自动覆盖人工编辑字段）
   - 显示完整度分析（代码扫描、文档导入、ESB数据）
   - 提供保存草稿、发布画像操作

**输出产物**：
- 分离后的知识导入页面
- 分离后的信息看板页面

**异常与边界处理**：
- 页面加载失败：提示错误并提供重试
- 系统列表为空：显示空状态提示

**说明**：
- 两个页面通过系统TAB同步
- 不展示导入历史/最近任务列表（保持页面简洁）
- 移除"待办事项"维度（信息已融入完整度分析）

### 3.2 功能性需求明细

#### REQ-001：代码扫描能力提取
**目标/价值**：
从Spring Boot代码中自动提取系统能力清单，补充系统画像

**入口/触发**：
配置管理 → 系统画像 → 知识导入 → "代码扫描"按钮

**前置条件**：
- 用户已登录且具有PM权限
- 当前用户为目标系统主责或B角（系统清单 owner_id=当前用户 或 `backup_owner_ids` 包含当前用户）
- 代码仓库可访问

**主流程**：
1. 用户输入系统名称，并提供代码仓库来源（二选一）：
   - `repo_path`：本地目录或 Git URL
   - `repo_archive`：上传代码仓库压缩包（备选）
2. 选择扫描选项（可选）：扫描路径、排除目录
3. 系统创建扫描任务，返回job_id
4. 后台线程执行扫描：
   - 遍历Java文件
   - 提取HTTP API（@GetMapping、@PostMapping等）
   - 提取定时任务（@Scheduled）
   - 提取消息监听器（@KafkaListener、@RabbitListener）
   - 提取外部调用（Feign、Dubbo、RestTemplate、MQ）
5. 扫描结果写入JSON文件
6. 更新任务状态为completed

**输入/输出**：
- 输入：
  - `system_name`: string（必填）
  - `system_id`: string（可选）
  - `repo_path`: string（可选，本地路径或Git URL；与 `repo_archive` 二选一）
  - `repo_archive`: file（可选，zip/tar.gz；与 `repo_path` 二选一）
  - `options.paths`: array[string]（可选，默认src/main/java）
  - `options.exclude_dirs`: array[string]（可选，默认排除.git、target、build）
- 输出：
  - `job_id`: string
  - `status`: queued/running/completed/failed/timeout
  - `result_path`: string（扫描结果文件路径）
  - `items`: array[object]（扫描结果项）

**业务规则**：
- 支持的代码类型：Spring Boot Java代码
- `repo_path` 为本地路径时：
  - 表示**服务端可访问的路径**（非客户端路径）
  - 必须位于配置的 allowlist 根目录内（拒绝 `..`、软链越界等），否则拒绝并返回错误
- `repo_archive` 上传时：
  - 服务端需进行安全解压（防 ZipSlip/路径穿越），解压到隔离工作目录；扫描完成后清理临时目录
  - 解压后总大小不超过 500MB，文件数量不超过 10,000 个；超限直接失败并清理临时目录
- 自动排除健康检查端点（actuator、health、metrics）
- HTTP API路径合并类级别@RequestMapping和方法级别@XxxMapping
- 外部调用包含目标系统提示

**异常与边界**：
- 仓库路径无效：返回400错误（`SCAN_001`）
- 本地仓库路径不在允许范围：返回400错误（`SCAN_004`）
- 仓库压缩包不合法/解压失败：返回400错误（`SCAN_005`）
- 仓库压缩包解压后超限：返回400错误（`SCAN_006`）
- 扫描超时（30分钟）：任务状态标记为timeout
- 文件编码异常：跳过该文件，记录警告

**验收标准（GWT）**：
- [ ] Given 用户输入有效的代码仓库路径（本地allowlist内路径或上传的zip压缩包），When 执行扫描，Then 返回job_id且任务状态为queued
- [ ] Given 扫描任务正在执行，When 查询任务状态，Then 返回running状态和进度百分比
- [ ] Given 代码包含@RestController，When 扫描完成，Then 提取HTTP API并包含完整路径和方法名
- [ ] Given 代码包含@FeignClient，When 扫描完成，Then 记录外部调用且target为FeignClient的name属性
- [ ] Given 仓库路径无效，When 执行扫描，Then 返回400错误且不创建任务

**关联**：SCN-001；API-001

#### REQ-002：代码扫描结果入库
**目标/价值**：
将代码扫描结果持久化到向量库，供AI检索使用

**入口/触发**：
配置管理 → 系统画像 → 知识导入 → 代码扫描区块（扫描完成后）→ "提交入库"按钮

**前置条件**：
- 代码扫描已完成且状态为completed
- 向量库服务可用

**主流程**：
1. 用户确认提交扫描结果
2. 系统读取扫描结果JSON文件
3. 为每个能力项生成文本描述
4. 批量生成embedding向量
5. 写入向量库（knowledge_type=capability_item）
6. 更新系统画像完整度评分（代码扫描部分30%）

**输入/输出**：
- 输入：
  - `job_id`: string（必填）
- 输出：
  - `success`: int（成功入库数量）
  - `failed`: int（失败数量）
  - `errors`: array[string]（可选，失败明细）

**业务规则**：
- 每个能力项生成包含summary、entry_id、keywords、related_calls的文本
- 向量库元数据包含entry_type、entry_id、location、system_id
- 入库按能力项粒度处理：单条失败不影响其他项，计入 `failed` 并记录失败明细
- embedding服务不可用：本次入库整体失败并返回错误（可重试），不更新完整度评分

**异常与边界**：
- 任务不存在：返回404错误
- 任务状态非completed：返回400错误
- embedding服务不可用：本次入库整体失败并返回错误信息（可重试，`EMB_001`）

**验收标准（GWT）**：
- [ ] Given 扫描任务已完成，When 提交入库，Then 扫描结果写入向量库且返回成功数量
- [ ] Given 任务状态为failed，When 提交入库，Then 返回400错误
- [ ] Given 入库完成，When 查询系统画像，Then 完整度评分代码扫描部分增加

**关联**：SCN-001；API-002

#### REQ-003：ESB文档导入
**目标/价值**：
从ESB治理文档中提取系统集成关系，补充系统画像

**入口/触发**：
配置管理 → 系统画像 → 知识导入 → "ESB导入"按钮

**前置条件**：
- 用户已登录且具有PM权限
- 当前用户为目标系统主责或B角（系统清单 owner_id=当前用户 或 `backup_owner_ids` 包含当前用户）
- ESB文档已准备（Excel（.xlsx/.xls）/CSV格式）

**主流程**：
1. 用户选择目标系统并选择ESB文档文件
2. 系统解析文件，识别列名（支持字段别名）
3. 验证必填字段：provider_system_id、consumer_system_id、service_name、status
4. 为每个条目生成embedding
5. 数据入库到ESB索引JSON文件
6. 返回导入统计（总数、成功、跳过、错误）

**输入/输出**：
- 输入：
  - `system_id`: string（必填）
  - `file`: file（必填）
  - `mapping_json`: object（可选，字段映射配置）
- 输出：
  - `total`: int（总行数）
  - `imported`: int（成功导入数）
  - `skipped`: int（跳过数）
  - `errors`: array[string]（错误列表）

**业务规则**：
- 支持文件类型：`.xlsx`、`.xls`、`.csv`
- 支持字段别名：交易码/service_code、提供方系统/provider_system_id等
- 仅导入与目标系统相关的条目：`provider_system_id=system_id` 或 `consumer_system_id=system_id`；其他条目计入skipped
- 必填字段缺失：跳过该行，记录错误
- 默认有效状态："正常使用"
- 生成文本格式："{服务名称} 提供方:{提供方} 调用方:{调用方} 场景:{场景码} 交易码:{交易码}"

**异常与边界**：
- 文件格式不支持：返回400错误
- 文件为空：返回错误信息
- 必填列全部缺失：返回详细错误提示
- embedding服务不可用：返回503错误（可重试，`EMB_001`），不写入任何条目

**验收标准（GWT）**：
- [ ] Given 上传有效的ESB Excel文件（.xlsx/.xls），When 执行导入，Then 返回导入统计且数据写入ESB索引
- [ ] Given 文件包含必填字段缺失行，When 执行导入，Then 跳过该行且在errors中记录
- [ ] Given 导入成功，When 查询ESB统计，Then 返回active_entry_count

**关联**：SCN-002；API-003

#### REQ-004：系统画像管理（草稿/发布/查看）
**目标/价值**：
- 汇总代码扫描/ESB/知识库文档生成系统画像草稿，并计算完整度评分
- 支持项目经理补充/纠错画像字段与证据引用，并发布到向量库供AI检索
- 支持管理员/PM/专家按权限查看系统画像（写权限仅限PM；专家只读）
- 支持知识导入后异步生成“字段AI建议”，供PM采纳/忽略，并用 `field_source` 标记字段来源（见 REQ-024）

**入口/触发**：
- 自动：代码扫描完成、ESB导入完成、知识库文档导入完成时，触发生成/更新系统画像草稿
- 手动：项目经理在“配置管理 → 系统画像 → 信息看板”保存草稿/点击发布

**前置条件**：
- 目标系统已识别（system_id/system_name）
- 至少存在一种材料来源（代码扫描/ESB/文档）或项目经理手动创建草稿
- 当前用户为目标系统主责或B角（系统清单 owner_id=当前用户 或 `backup_owner_ids` 包含当前用户；草稿写/重试AI总结允许主责或B角；发布仅主责）

**主流程**：
1. 系统汇总多源数据（代码扫描、ESB、知识库文档），生成/更新系统画像**草稿**（`status=draft`）
2. （异步，见 REQ-024）系统可基于导入材料生成字段AI建议（不自动覆盖人工编辑字段），并通知主责PM在信息看板采纳/忽略
3. 计算完整度评分：代码扫描30% + 文档40% + ESB30%
4. 项目经理可查看/编辑草稿字段与 `evidence_refs` 并保存
5. 发布前校验必填字段（`in_scope`、`core_functions`）；缺失则提示 `pending_fields` 并拒绝发布
6. 发布：生成画像文本embedding并写入向量库（knowledge_type=system_profile），更新 `status=published` 并写入 `published_at`
7. 过时检查与提醒：
   - 系统每周检查一次已发布画像的 `published_at`；超过阈值（默认30天）则标记 `is_stale=true`
   - 系统画像列表页过时条目显示警告图标，PM可点击查看发布时间并选择更新

**输入/输出**：
- 输入：
  - 自动：代码扫描结果、ESB数据、知识库文档
  - 手动：`fields`、`evidence_refs`（项目经理）
- 输出：
  - `system_id`: string
  - `system_name`: string
  - `status`: draft/published
  - `fields`: object（画像字段）
    - `in_scope` string，必填：系统边界（做什么）
    - `out_of_scope` string，可选：系统不做什么
    - `core_functions` string，必填：核心功能
    - `business_goals` string，可选：业务目标
    - `business_objects` string，可选：业务对象
    - `integration_points` string，可选：主要集成点
    - `key_constraints` string，可选：关键约束
  - `field_sources`: object（字段来源标记，manual/ai；可选）
    - `in_scope` string（manual/ai）
    - `out_of_scope` string（manual/ai）
    - `core_functions` string（manual/ai）
    - `business_goals` string（manual/ai）
    - `business_objects` string（manual/ai）
    - `integration_points` string（manual/ai）
    - `key_constraints` string（manual/ai）
  - `ai_suggestions`: object（AI字段建议，可选；由 REQ-024 产出）
  - `ai_suggestions_updated_at`: string（ISO时间，可选）
  - `evidence_refs`: array[object]
  - `pending_fields`: array[string]
  - `completeness_score`: number（完整度评分0-100）
  - `updated_at`: string（ISO时间）
  - `published_at`: string（ISO时间；published时返回）
  - `is_stale`: boolean（是否可能过时）

**业务规则**：
- 完整度评分 = 代码扫描30% + 文档40% + ESB30%
- 文档计数仅统计 `knowledge_type=document` 且 `level=normal`（不包含历史评估文档L0）
- 发布必填字段：`in_scope`、`core_functions`；其他字段允许为空
- `business_goals` 为canonical字段；兼容历史 `business_goal`：
  - 读取：若仅存在 `business_goal` 则映射为 `business_goals` 展示
  - 写入：统一写入 `business_goals`
- 画像文本格式：系统名称 + 画像字段（`in_scope/out_of_scope/core_functions/business_goals/business_objects/integration_points/key_constraints`）拼接为一段文本，用于 embedding
- 权限：
  - 草稿写操作（创建/保存/采纳AI建议/忽略/重试AI总结）：目标系统主责或B角PM
  - 发布：仅目标系统主责PM

#### REQ-006：复杂度三维度评估
**目标/价值**：
使用业务规则复杂度、集成点数量、技术难度三维度评估功能点复杂度

**入口/触发**：
AI评估Agent执行功能点拆分时

**前置条件**：
- 已完成功能点识别

**主流程**：
1. AI分析功能点的业务规则：
   - 规则数量（条件分支、特殊场景）
   - 规则复杂度（嵌套层级、组合逻辑）
2. AI分析集成点：
   - 对接系统数量
   - 接口复杂度（同步/异步、数据量）
3. AI分析技术难度：
   - 技术栈成熟度（现有框架/新框架/全新技术栈）
   - 数据量级（小/中/大）
   - 性能要求（普通/高/严苛）
   - 重构规模（无重构/局部重构/大面积重构）
4. 计算复杂度评分：
   - 业务规则复杂度分 = 规则评分 × 35%
   - 集成点数量分 = 集成点评分 × 35%
   - 技术难度分 = 技术难度评分 × 30%
   - 总分 = 业务规则分 + 集成点分 + 技术难度分
5. 映射到高/中/低标签
6. 生成复杂度说明

**输入/输出**：
- 输入：
  - `feature_description`: string（功能点描述）
  - `system_context`: object（系统上下文）
- 输出：
  - `complexity_level`: string（high/medium/low）
  - `complexity_score`: number（0-100）
  - `business_rules_score`: number（0-35）
  - `integration_score`: number（0-35）
  - `technical_difficulty_score`: number（0-30）
  - `reasoning`: string（评估理由）

**业务规则**：
- 业务规则评分（0-35）：
  - 简单规则（1-3条）：7-14分
  - 中等规则（4-10条）：14-24分
  - 复杂规则（10+条或有复杂组合）：24-35分
- 集成点评分（0-35）：
  - 无集成：3分
  - 单系统简单集成：7-14分
  - 多系统集成：14-24分
  - 复杂集成（异步/大数据量）：24-35分
- 技术难度评分（0-30）：
  - 技术栈成熟度：0-10分（现有框架0-3分、新框架4-7分、全新技术栈8-10分）
  - 数据量级：0-8分（小数据量0-2分、中等3-5分、大数据量6-8分）
  - 性能要求：0-6分（普通0-2分、高3-4分、严苛5-6分）
  - 重构规模：0-6分（无重构0分、局部重构1-3分、大面积重构4-6分）

**异常与边界**：
- 无法评估复杂度：默认medium
- 系统上下文缺失：降低业务规则评分权重

**验收标准（GWT）**：
- [ ] Given 功能点包含5个业务规则，When 评估复杂度，Then business_rules_score在14-24之间
- [ ] Given 功能点对接2个外部系统，When 评估复杂度，Then integration_score在14-24之间
- [ ] Given 功能点使用新框架且数据量大，When 评估复杂度，Then technical_difficulty_score>15

**关联**：SCN-004；API-006（内部Agent调用）

#### REQ-007：修改轨迹记录
**目标/价值**：
记录PM对AI评估结果的修改，用于后续模式分析

**入口/触发**：
PM在评估页面修改功能点时

5. 系统记录修改轨迹：
   - 操作类型（delete/adjust/add）
   - 操作对象（feature_id）
   - 修改原因（reason_code）
   - 补充说明（notes）
   - 原始AI推理（original_ai_reasoning）
   - 操作时间（timestamp）
   - 操作人（actor）

**输入/输出**：
- 输入：
  - `operation`: string（delete/adjust/add）
  - `feature_id`: string
  - `reason_code`: string（必填）
  - `notes`: string（可选）
- 输出：
  - `trace_id`: string
  - `recorded_at`: string（ISO timestamp）

**业务规则**：
- 修改原因为必填项
- 记录包含原始AI推理（由系统从AI初始评估结果获取），便于后续分析
- 原始AI推理获取方式：

**异常与边界**：
- 原因未选择：提示必填，不允许保存
- 记录失败：返回错误，但不阻止修改操作

**验收标准（GWT）**：
- [ ] Given PM删除功能点，When 弹出原因选择，Then 显示删除类原因选项且必填
- [ ] Given PM选择原因并保存，When 查询修改轨迹，Then 包含operation=delete和选择的原因
- [ ] Given PM未选择原因，When 尝试保存，Then 提示必填且不允许保存

**关联**：SCN-005；API-007

#### REQ-008：专家差异统计
**目标/价值**：
统计AI估算与专家估值的差异，用于AI效果分析

**入口/触发**：
所有专家完成评估后

**前置条件**：
- 至少3位专家完成评估

**主流程**：
1. 收集AI估算值（每个功能点的人天）
2. 收集各专家估算值
3. 计算每个功能点的专家平均值
4. 计算AI与专家均值的偏差：
   - 偏差% = (AI估算 - 专家均值) / 专家均值 × 100%
5. 判断偏差方向：正值为AI高估，负值为AI低估
6. 生成差异统计表
7. 保存到评估结果JSON文件

**输入/输出**：
- 输入：
  - `ai_estimation`: array[object]（AI估算结果）
  - `expert_estimations`: array[array[object]]（各专家估算）
- 输出：
  - `deviation_report`: object（偏差报告）
    - `by_feature`: array[object]（按功能点统计）
    - `summary`: object（汇总：平均偏差、最大偏差、偏差方向分布）

**业务规则**：
- 专家平均值 = (专家1 + 专家2 + ... + 专家N) / N
- 偏差%保留1位小数
- 偏差方向：AI低估（偏差< -10%）、基本一致（-10% ≤ 偏差 ≤ 10%）、AI高估（偏差> 10%）
- 异常值处理：专家估值与均值偏差>20%时标记异常，不计入均值

**异常与边界**：
- 专家不足3人：使用实际参与人数，但在报告中标注
- 专家估值为0或空：跳过该专家

**验收标准（GWT）**：
- [ ] Given 3位专家完成评估，When 生成差异报告，Then by_feature包含每个功能点的AI估算、专家均值、偏差%
- [ ] Given AI估值为2人天，专家均值为2.5人天，When 计算偏差，Then 偏差%为-20%且方向为AI低估
- [ ] Given 某专家估值为异常高值，When 计算均值，Then 标记异常且不计入均值

**关联**：SCN-006；API-008（内部计算）

5. 生成embedding向量
6. 写入向量库，标记knowledge_type和level：
   - 普通文档：knowledge_type=document，level=normal
   - 历史评估文档：knowledge_type=document，level=l0
7. 更新系统画像完整度评分（文档部分）
8. 返回导入统计

**输入/输出**：
- 输入：
  - `system_id`: string（可选；与 `system_name` 二选一；用于绑定系统并更新完整度）
  - `system_name`: string（可选；与 `system_id` 二选一；用于绑定系统并更新完整度）
  - `knowledge_type`: string（document）
  - `level`: string（normal/l0；默认normal）
  - `file`: file
- 输出：
  - `imported`: int
  - `failed`: int
  - `errors`: array[string]

**业务规则**：
- 支持文档类型：
  - 文档：docx、doc、pdf、pptx、txt
  - 表格：xlsx、xls、csv
  - 不支持：ppt（如需导入请先转换为 pptx 或 pdf）
  - 旧格式解析：`.doc` 通过转换/抽取工具提取文本；`.xls` 使用解析库抽取表格文本（详见 REQ-027、REQ-NF-007）
- L0文档在检索时降权（相似度得分×0.3），单次最多返回5条
- L0文档不计入系统材料完整度评分的"文档"计数
- 向量库元数据包含source_file、knowledge_type、level、created_at
- 文档完整度评分：0篇为0分，1-5篇10%，6-10篇30%，10+篇40%
- 未绑定系统（无法识别 system_id/system_name）时：允许导入入库，但不更新系统画像完整度
- 绑定系统并更新完整度时需权限校验：仅系统清单中该系统主责或B角可导入（否则返回 `AUTH_001`；见 REQ-NF-004 / API-011）

**异常与边界**：
- 文件类型不支持：返回错误提示
- 文件解析失败：记录错误，支持重试
- embedding服务不可用：本次导入整体失败并返回错误信息（可重试，`EMB_001`）
- embedding生成失败（单条问题）：记录失败条目并返回errors，不影响其他条目

**验收标准（GWT）**：
- [ ] Given PM选择目标系统并上传文档，When 导入完成，Then 知识库包含knowledge_type=document的条目且通过API-012查询完整度评分增加

**说明**：知识导入页面不展示导入历史/最近任务列表；导入结果以统计与提示反馈即可

**关联**：SCN-009；API-011

#### REQ-023：多角色切换（activeRole）
**目标/价值**：
支持多角色用户在不同角色视角间快速切换，使菜单、权限提示与默认数据范围一致，避免“同一用户多套交互”带来的认知负担。

**入口/触发**：
应用右上角“角色切换器”（Header 区域）

**前置条件**：
- 当前用户拥有的角色列表 `roles` ≥ 2（如 manager+expert）

**主流程**：
1. 用户登录后系统加载 `roles` 列表
2. 若 `roles` ≥ 2，Header 展示角色切换器；默认激活最高权限角色（admin > manager > expert > viewer）
3. 用户切换活跃角色后：
   - 菜单与页面可见操作按活跃角色刷新
   - 任务管理与相关列表的默认数据范围随活跃角色变化（见 REQ-010）
4. 页面刷新/重新进入应用时，若存在上次选择的活跃角色且仍在 `roles` 中，则优先使用；否则回退到默认角色

**输入/输出**：
- 输入：`roles`: array[string]
- 输出：`activeRole`: string（admin/manager/expert/viewer）

**业务规则**：
- 活跃角色仅影响**前端菜单与默认筛选**，不改变后端资源级授权（后端仍以用户真实 `roles` 判定）
- 角色切换不要求重新登录/刷新JWT
- 角色切换器仅展示用户实际拥有的角色

**异常与边界**：
- 用户仅单一角色：不展示角色切换器或置灰
- `activeRole` 异常（不在 `roles` 内）：自动回退到默认角色

**验收标准（GWT）**：
- [ ] Given 用户拥有 manager+expert 两种角色，When 点击角色切换器切换到 expert，Then 菜单与任务列表数据范围切换为专家视角
- [ ] Given 用户拥有 admin+manager+expert，When 登录系统，Then 默认激活 admin 角色
- [ ] Given 用户刷新页面，When 上次选择角色仍有效，Then 仍保持该活跃角色

**关联**：SCN-008

#### REQ-024：画像AI总结与采纳
**目标/价值**：
将“知识导入”的材料自动转化为信息看板可编辑的字段建议，减少PM重复填写成本，同时保持PM最终决策权与可追溯性。

**入口/触发**：
- 自动：代码扫描入库（REQ-002）、ESB导入（REQ-003）、知识库文档导入（REQ-011）成功并绑定到系统后
- 交互：PM在信息看板查看AI建议并“采纳/忽略”（不自动覆盖人工编辑字段）

**前置条件**：
- 目标系统已绑定（system_id/system_name）
- 至少存在一种材料来源（代码/ESB/文档）

**主流程**：
1. 任一“知识导入”成功后，系统创建并异步执行“画像AI总结”任务（不阻塞导入接口返回）
2. AI总结从最新材料中提取并生成7个画像字段的建议，写入系统画像草稿：
   - `ai_suggestions`：按字段给出建议文本
   - `ai_suggestions_updated_at`
3. 系统向主责PM发送“画像AI总结完成/失败”通知（可跳转到信息看板；失败通知包含重试入口）
4. PM进入信息看板后，逐字段对比“当前值 vs AI建议”并执行：
   - **采纳**：将字段值更新为AI建议，并将对应 `field_sources.<field>=ai`
   - **忽略**：不修改字段当前值，仅标记本轮建议已忽略（实现可简化为前端本地忽略或服务端记录）
5. PM手动编辑字段并保存草稿时，系统将对应 `field_sources.<field>=manual`

**输入/输出**：
- 输入：代码扫描摘要、ESB条目、知识库文档文本（以系统维度聚合）
- 输出：系统画像草稿的 `ai_suggestions/field_sources`；通知中心的“画像AI总结完成/失败”通知；（可选）手动重试触发记录

**业务规则**：
- 不自动覆盖：当字段当前来源为 `manual` 时，后台AI总结不得直接覆盖字段当前值；仅允许更新 `ai_suggestions`
- AI建议为辅助信息：只有在PM“采纳”后才会影响字段当前值；发布流程仍遵循 REQ-004
- 失败处理：AI总结失败不影响导入与画像草稿更新；必须发送失败通知（`notification.type=system_profile_summary_failed`，payload包含 `system_id/system_name/link/error_code/error_reason`），并提供手动重试入口（信息看板“重试生成AI建议”，调用 API-018）；默认不做自动重试
- 重试策略：同一 `system_id` 同一时刻最多运行1个“画像AI总结”任务；当存在运行中任务时，手动重试请求返回“生成中”提示且不重复排队（幂等）

**异常与边界**：
- 系统未绑定（缺少system_id/system_name）：不触发AI总结
- 材料过少/无法总结：允许生成空建议，并在通知/页面提示“材料不足”

**验收标准（GWT）**：
- [ ] Given PM对系统完成一次知识导入，When AI总结完成，Then 通知中心出现“画像AI总结完成”通知且可跳转到信息看板
- [ ] Given 信息看板存在某字段的AI建议，When PM点击“采纳”，Then 字段值更新且 `field_sources` 标记为 ai
- [ ] Given 某字段已被PM手动编辑（field_source=manual），When AI总结任务完成，Then 系统不应自动覆盖该字段当前值
- [ ] Given 系统画像AI总结失败，When 失败事件发生，Then 通知中心出现“画像AI总结失败”通知且包含 `error_code` 与“重试生成AI建议”入口
- [ ] Given PM收到“画像AI总结失败”通知，When 点击“重试生成AI建议”，Then 系统受理重试并在完成后发送“画像AI总结完成”通知（或在信息看板展示“生成中”状态）

**关联**：SCN-003, SCN-009, SCN-015；API-004、API-011、API-017、API-018

#### REQ-027：旧格式文件支持（DOC + XLS）
**目标/价值**：
兼容业务中仍大量存在的 `.doc` 与 `.xls` 旧格式文件，减少用户手工转换成本，提升“发起评估/知识导入”的可用性。

**入口/触发**：
- 发起评估（UploadPage）：上传需求文档创建任务
- 知识导入（系统画像）：ESB导入、知识库文档导入

**前置条件**：
- 用户已登录且具备对应入口的权限（任务创建=PM；知识导入=系统主责或B角PM）

**主流程**：
1. 客户端文件选择/拖拽时允许选择 `.doc` 与 `.xls`（在现有支持格式基础上扩展）
2. 服务端对文件类型进行校验（扩展名 + MIME/魔数校验为主），不通过则返回“文件格式不支持”错误
3. 服务端解析并抽取内容：
   - `.doc`：通过转换/抽取工具提取文本
   - `.xls`：解析表格并抽取可读文本（按行/列拼接或结构化提取）
4. 解析后的文本进入既有流程：
   - 发起评估：作为任务输入进入AI解析与评估（SCN-004）
   - 知识导入：写入知识库并更新完整度（SCN-002/009）

**业务规则**：
- 各入口的“大小限制/并发/超时”等安全约束与 REQ-NF-007 一致
- `.xls` 在不同入口语义不同：发起评估（API-016）按“需求表格”做文本抽取；ESB导入（API-003）需符合ESB字段口径；知识库文档导入（API-011）按系统知识表格抽取
- 解析失败应返回可定位的错误提示（不暴露服务端路径与敏感信息）

**异常与边界**：
- 文件解析失败：返回错误并允许重试
- 文件内容为空/不可读：返回错误或导入0条（按入口契约定义）

**验收标准（GWT）**：
- [ ] Given PM上传`.doc`或`.xls`格式的需求文档，When 创建任务并触发AI解析，Then 系统成功接收并完成文本抽取
- [ ] Given PM在知识导入上传`.doc`或`.xls`文件，When 导入完成，Then 文件被成功解析并入库且不要求用户预转换格式

**关联**：SCN-002, SCN-004, SCN-009；API-003、API-011、API-016

#### REQ-NF-007：旧格式解析安全
**需求分类**：安全
**适用范围**：旧格式文件解析（`.doc/.xls`），覆盖发起评估上传与知识导入/ESB导入等入口（见 REQ-027）

**指标与口径**：
- 最小权限：解析过程不应以root权限运行（如容器内可通过运行用户/权限隔离实现）
- 隔离目录：解析过程仅在隔离临时目录内读写，不允许写入任意路径
- 超时控制：单文件解析设置超时（默认≤60s，可配置）；超时视为解析失败
- 资源限制：单文件解析的CPU/内存限制可配置（环境允许时启用）
- 清理：解析成功/失败均需清理临时文件（避免磁盘堆积）

**验收方法/证据**：
- 代码审查：检查临时目录、文件名清洗、超时/清理逻辑、外部工具调用参数
- 安全测试：
  - 上传超大/畸形文件应被拒绝或在超时后失败
  - 路径穿越/特殊文件名不应导致越界写入

**异常与边界**：
- 外部工具不可用：返回明确错误（不进入业务流程），并记录告警日志
- 解析失败：返回可定位的错误提示（不暴露服务端路径/命令行细节）

**关联**：REQ-027

### 6.3 指标与计算口径

**PM修正率**：
- 公式：`PM修正率 = (删除操作数 + 调整操作数) / AI初始功能点数 × 100%`
- 统计周期：发布后30天
- 分母选择：AI初始功能点数（不包含PM新增）
- AI初始功能点数定义：AI完成“首次评估”时输出的功能点列表数量（冻结，不随PM后续修改变化）；系统必须保存 `task.ai_initial_feature_count`（=len(`task.ai_initial_features`)）
- 操作数口径：统计周期内 `modification_trace` 的记录条数（按 `operation=delete/adjust` 分类汇总）
- 取值范围：≥0%（当“调整操作数”较多时可>100%）
- 目标值：≤10%
- 基线收集方案：在 Requirements 阶段完成前，统计最近10次评估任务的PM修正率与PM新增数作为基线（用于“基线→目标”对比）

**PM新增数（AI遗漏）**：
- 公式：`PM新增数 = 新增操作数`
- 统计周期：发布后30天
- 操作数口径：统计周期内 `modification_trace` 的记录条数（`operation=add`）
- 说明：PM新增数不纳入PM修正率，用于单独衡量AI遗漏情况

**系统画像完整度评分**：
- 公式：`完整度 = 代码扫描30% + 文档40% + ESB30%`
- 代码扫描：有数据为30%，否则0%
- 文档：0篇为0分，1-5篇10%，6-10篇30%，10+篇40%（仅统计 `knowledge_type=document` 且 `level=normal`，不包含历史评估文档L0）
- ESB：有数据为30%，否则0%

**复杂度评分**：
- 业务规则复杂度分（0-35）：根据规则数量和复杂度评分（详见 REQ-006）
- 集成点数量分（0-35）：根据对接系统数和接口复杂度评分（详见 REQ-006）
- 技术难度分（0-30）：根据技术栈成熟度/数据量/性能要求/重构规模评分（详见 REQ-006）
- 最终分数：`业务规则分 + 集成点分 + 技术难度分`（0-100）
- 映射规则：0-40为low，41-70为medium，71-100为high

**AI与专家偏差**：
- 公式：`偏差% = (AI估算 - 专家均值) / 专家均值 × 100%`
- 专家均值：`(专家1 + 专家2 + ... + 专家N) / N`
- 偏差方向：偏差< -10%为AI低估，> 10%为AI高估

---

**效能看板指标（对齐最终评估）**：

> 统计基准时间：默认使用 `task.frozen_at` 作为“最终评估口径冻结时间”（冻结时写入涉及系统、系统拆分人天、系统负责人快照等）。所有以“最终评估”为口径的统计默认以此时间落窗，避免后续变更导致历史口径漂移。
> 冻结触发：当任务状态从 `in_progress` 变更为 `completed` 时，系统自动写入 `task.frozen_at` 与 `task.owner_snapshot`（仅写一次）。

**评估任务完成量**：
- 公式：`完成量 = count(task where frozen_at ∈ [start_at, end_at])`
- 维度：时间（周/月）、系统、负责人（快照主责）、项目

**角色处理量（评估任务数）**：
- PM处理量：`count(task where owner_snapshot.owner_id = pm_id and frozen_at ∈ window)`
- 专家处理量：`count(task where expert_id ∈ task.expert_ids and frozen_at ∈ window)`（按任务去重）
- AI处理量：`count(task where ai_involved=true and frozen_at ∈ window)`
- 展示：TOP10（支持按角色切换）

**系统改造次数 / 系统改造工作量（人天）**：
- 系统改造次数：`count(distinct task_id where final_estimation_days_by_system contains system_id with days>0 and frozen_at ∈ window)`
- 系统改造工作量：`sum(final_days where final_estimation_days_by_system contains system_id and frozen_at ∈ window)`
- 说明：系统改造口径以“评估结论里要改的系统”为准（=最终评估按系统拆分）

**材料完整度（系统排行）**：
- 取值：`system_profile.completeness_score`（0-100）
- 排行榜：按当前分值排序，展示 TOP10 + Bottom10
- 可选过滤：仅展示在统计窗口内曾出现在评估中的系统（避免“冷系统”影响榜单）

**AI评估准确率（对齐最终评估）**：
- 适用范围：仅统计 `ai_involved=true` 且存在 `final_estimation_days_total` 的任务
- 任务级误差：
  - `abs_error_days = abs(ai_estimation_days_total - final_estimation_days_total)`
  - 当 `final_estimation_days_total >= 1`：`pct_error = abs_error_days / final_estimation_days_total`
  - 当 `final_estimation_days_total < 1`：百分比误差不稳定，命中率改用绝对误差阈值
- 命中率（默认阈值，可配置）：
  - 当 `final >= 1`：命中条件 `pct_error <= 0.20`
  - 当 `final < 1`：命中条件 `abs_error_days <= 0.5`
  - `hit_rate = count(hit)/count(tasks_in_scope)`
- MAE（人天）：`mean(abs_error_days)`
- Bias：`mean(ai_estimation_days_total - final_estimation_days_total)`
- 时间趋势：按 `frozen_at` 聚合（周/月），展示 hit_rate/MAE/Bias

**AI准确率（系统维度）**：
- 适用范围：统计 `ai_involved=true` 且存在 `final_estimation_days_by_system` 的任务；按 `(task_id, system_id)` 粒度聚合
- 系统级误差：
  - `abs_error_days_sys = abs(ai_days_sys - final_days_sys)`
  - 当 `final_days_sys >= 1`：`pct_error_sys = abs_error_days_sys / final_days_sys`
  - 当 `final_days_sys < 1`：命中率使用绝对误差阈值（同上）
- 产出：按系统的 hit_rate/MAE/Bias（支持Top/Bottom展示）

#### API-001：代码扫描任务（提交/查询）
**调用方/权限**：项目经理

**接口**：
- `POST /api/v1/code-scan/jobs`
- `GET /api/v1/code-scan/jobs/{job_id}`

**请求方式（POST）**：
- JSON（推荐）：使用 `repo_path`（本地路径或 Git URL）
- `multipart/form-data`（备选）：使用 `repo_archive`（上传压缩包）

**请求参数（POST JSON）**：
- `system_name` string，必填
- `system_id` string，可选
- `repo_path` string，必填（本地路径或Git URL）
- `force` boolean，可选，默认false（true时强制创建新任务）
- `options.paths` array[string]，可选，默认 `["src/main/java"]`
- `options.exclude_dirs` array[string]，可选，默认 `[".git","target","build"]`

**请求参数（POST multipart）**：
- `system_name` string，必填
- `system_id` string，可选
- `repo_archive` file，必填（zip/tar.gz）
- `force` string，可选，默认false（true时强制创建新任务）
- `options_json` string，可选（JSON字符串，包含 paths/exclude_dirs）

**响应（JSON）**：
- `job_id` string
- `status` string：queued/running/completed/failed/timeout
- `created_at` string（ISO时间）
- `progress` int（0-100，GET可选）
- `result_path` string（completed时返回）
- `message` string（failed/timeout时返回）

**输入校验与边界**：
- 权限校验：仅允许对自己负责的系统发起扫描（系统清单 owner_id=当前用户 或 `backup_owner_ids` 包含当前用户），否则返回 `AUTH_001`
- `repo_path` 为本地路径时：必须在服务端配置的 allowlist 根目录内，否则返回 `SCAN_004`
- `repo_path` 为 Git URL 时：仅允许 `http/https/ssh` 协议；默认可配置为“禁用”，或仅允许 host allowlist；未启用/不在allowlist 视为“无法访问”并返回 `SCAN_001`（建议 message 明确原因）
- `repo_archive` 解压失败/格式不支持：返回 `SCAN_005`
- `repo_archive` 解压后超出大小/文件数限制：返回 `SCAN_006`
- 幂等性：相同 `repo_path` 或 `repo_archive` 内容hash匹配时返回已有 job_id；支持 `force=true` 参数强制创建新任务

**错误码**：`AUTH_001`、`SCAN_001`、`SCAN_002`、`SCAN_004`、`SCAN_005`、`SCAN_006`

**示例**：
```json
{
  "system_name": "core-banking",
  "repo_path": "https://git.example.com/core-banking.git",
  "options": {
    "paths": ["src/main/java"],
    "exclude_dirs": [".git", "target", "build"]
  }
}
```

```json
{
  "job_id": "scan_20260206_0001",
  "status": "queued",
  "created_at": "2026-02-06T17:00:00Z"
}
```

```json
{
  "job_id": "scan_20260206_0001",
  "status": "running",
  "progress": 35,
  "created_at": "2026-02-06T17:00:00Z"
}
```

- `GET /api/v1/notifications`（分页查询通知列表）
- `GET /api/v1/notifications/unread-count`（未读数）
- `PUT /api/v1/notifications/{notification_id}/read`（标记已读）
- `PUT /api/v1/notifications/read-all`（全部标记已读）
- `DELETE /api/v1/notifications/clear-read`（清理已读）
- `DELETE /api/v1/notifications/{notification_id}`（删除单条）

**响应（示例字段）**：
- 列表：`items`（含 notification_id/type/status/payload/created_at）、`total/page/page_size`
- 未读数：`unread_count` int

**错误码**：`AUTH_001`

**说明**：
- 本CR新增通知类型：
  - `system_profile_summary_ready`（画像AI总结完成）
  - `system_profile_summary_failed`（画像AI总结失败）
- payload 建议包含：`system_id`、`system_name`、`link`；失败通知额外包含 `error_code`、`error_reason`
- 通知留存：默认90天（可配置）；系统每日自动清理“已读且超过留存期”的通知；用户也可手动清理已读/删除单条

---

#### API-018：画像AI总结重试（生成AI建议）
**调用方/权限**：项目经理（manager；系统主责或B角）

**接口**：
- `POST /api/v1/system-profiles/{system_id}/ai-suggestions/retry`

**请求参数**：
- `system_id` string（path），必填

**响应（JSON）**：
- `job_id` string
- `status` string（queued/running）
- `message` string

**错误码**：`AUTH_001`、`PROFILE_002`、`SUMMARY_001`

**资源级权限**：
- 仅允许对系统清单中该系统主责（owner_id=当前用户）或B角（`backup_owner_ids` 包含当前用户）的系统触发；否则返回 `AUTH_001`

**说明**：
- 用于“画像AI总结失败”后的手动重试；也可用于人工刷新AI建议（不影响已编辑字段的 `manual` 来源规则）
- 幂等：当该系统已有运行中的“画像AI总结”任务时，接口不重复创建任务，直接返回当前 `job_id` 并提示“生成中”
