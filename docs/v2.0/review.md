## 关键发现（按优先级）

### RVW-REQ-001（P1）提案与需求在复杂度评估维度上不一致

**证据**：
- `proposal.md` L67、L232：写明"三维度：业务规则+集成+技术"
- `proposal.md` L164-170：复杂度评估模型已改为"二维度评估（业务规则50% + 集成点50%）"
- `requirements.md` L28：明确"复杂度二维度评估模型（业务规则+集成点）"

**风险**：
- 提案中存在"三维度"和"二维度"两种描述，容易造成混淆
- 实现时可能不清楚应该按哪个标准

**建议修改**：
将 `proposal.md` L67、L232 中的"三维度"统一改为"二维度"：
```markdown
### 具体目标
3. 优化**复杂度评估**（二维度：业务规则+集成点，技术难度作为调整因子）
```

**验证方式**：
```bash
grep -n "三维度\|二维度" docs/v2.0/proposal.md
```

### RVW-REQ-002（P1）缺少 API 需求章节

**证据**：
- `requirements.md` L74：ID前缀规则中定义了"API需求：API-xxx"
- `requirements.md` L467、L512、L561等多处引用了"关联：...；API-001"
- 但文档中没有定义 API-001 到 API-013 的具体接口定义

**风险**：
- 设计阶段缺少API契约依据
- 前后端对接时可能产生理解偏差
- 无法追溯接口到需求

**建议修改**：
在 `requirements.md` 第6章"数据与接口"中添加"6.4 API需求"章节，定义：
- API-001：代码扫描任务提交
- API-002：代码扫描结果入库
- API-003：ESB文档导入
- API-004：系统画像管理
- API-005：系统画像检索（内部）
- API-006：复杂度评估（内部）
- API-007：修改轨迹记录
- API-008：专家差异统计（内部）
- API-009：AI效果分析
- API-010：任务管理查询
- API-011：知识库导入
- API-012：系统材料完整度查询
- API-013：评估详情查询

每个API包含：方法、路径、请求参数、响应格式、错误码

**验证方式**：
```bash
grep -n "API-00[0-9]" docs/v2.0/requirements.md
```

### RVW-REQ-003（P2）提案章节编号错误

**证据**：
- `proposal.md` L186-200："#### 4.2 管理员"和"#### 4.2 项目经理"重复编号
- L186使用了"4.2 管理员"，L200也使用了"4.2 项目经理"

**风险**：
- 文档结构不规范
- 可能导致引用错误

**建议修改**：
将 `proposal.md` L200 的"#### 4.2 项目经理"改为"#### 4.3 项目经理"
后续章节编号相应调整：
- "#### 4.3 专家" → "#### 4.4 专家"
- "#### 4.4 通用" → "#### 4.5 通用"

**验证方式**：
```bash
grep -n "^#### 4\." docs/v2.0/proposal.md
```

### RVW-REQ-004（P2）部分非功能需求缺少可测试性细节

**证据**：
- `requirements.md` L1111-1116：非功能需求列表中，验收方式描述较笼统
- 如 REQ-NF-002 向量检索性能 "<500ms（P95）"未说明测试数据量

**风险**：
- 测试时可能无法确定是否符合要求
- 验收标准可能产生争议

**建议修改**：
在非功能需求验收标准中补充：
- REQ-NF-002：明确"向量库规模10万条，单次查询top_k=20时的P95值"
- REQ-NF-003：明确"使用包含10种敏感信息的测试用例"
- REQ-NF-005：明确"10个任务同时提交时的并发表现"

### RVW-REQ-005（P2）场景 SCN-008、SCN-009、SCN-010 缺少异常处理

**证据**：
- `requirements.md` L321-340：SCN-008 任务管理状态分组 - 无异常处理
- `requirements.md` L343-364：SCN-009 知识库分类导入 - 无异常处理
- `requirements.md` L367-386：SCN-010 评估页UI优化 - 无异常处理

**风险**：
- 测试阶段缺少异常场景的验收依据
- 实现时可能遗漏异常处理

**建议修改**：
为这三个场景补充异常与边界处理：
```markdown
**异常与边界处理**：
- 任务列表加载失败：显示错误提示，提供重试按钮
- 知识库文件类型不支持：提示支持格式
- 系统材料完整度获取失败：显示"完整度未知"
```

### RVW-REQ-006（P2）需求追踪矩阵不完整

**证据**：
- `requirements.md` L72-73 定义了 ID 前缀规则：场景 SCN-xxx、需求 REQ-xxx、API API-xxx
- 文档中没有完整的追溯矩阵（场景→需求→API→测试）

**风险**：
- 设计和实现时可能遗漏某些需求
- 测试覆盖度无法验证

**建议修改**：
在文档末尾添加"7.2 需求追溯矩阵"：
| 场景ID | 关联需求 | 关联API | 测试用例 |
|---|---|---|---|
| SCN-001 | REQ-001, REQ-002 | API-001, API-002 | TEST-001~TEST-005 |
| SCN-002 | REQ-003 | API-003 | TEST-006~TEST-008 |
| ...

**验证方式**：
检查是否所有 SCN-ID 都有对应的 REQ-ID 和 API-ID
