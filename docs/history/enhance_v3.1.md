# 需求评估系统优化方案建议书

> 版本：v3.1
> 日期：2026-02-01
> 目标：全面提升评估准确性，聚焦系统识别与功能拆分
> 核心调整：基于数据源实际情况重新定位

---

## 零、核心决策记录（v3.1）

### 0.1 数据来源实际情况

| 数据源 | 实际内容 | 可靠性 | 用途 |
|-------|---------|--------|------|
| 系统台账 | 系统名称、基本信息 | ✅ 可靠 | 系统识别 |
| 架构文档 | 技术架构、部署情况 | ⚠️ 功能清单通常缺失/不准 | 参考补充 |
| ESB接口清单 | 跨系统调用接口 | ⚠️ 只覆盖集成点 | 集成点识别 |
| **代码** | **实际功能的唯一"事实来源"** | ✅ 可靠 | **功能清单提取** |

**结论**：台账和架构文档无法提供可靠的功能清单，ESB只覆盖跨系统调用，代码扫描成为获取功能清单的主要手段。

### 0.2 业务场景定位

```
评估结果用于采购决策，专家可能直接采用AI结果
→ 功能遗漏 = 采购风险（漏估工作量）
→ 错误估算（需求未提及的功能被纳入）= 更严重的采购风险（高估工作量）

因此：功能拆分的准确性（不遗漏、不夸大） > 单纯准确率
核心原则：严格遵循需求文档，不擅自添加未提及的功能点
```

### 0.3 方案调整

```
第一阶段（本周）：
├── 系统台账 → 系统识别（已存在，"系统清单配置"）
├── 代码扫描试点（1-2个核心系统）→ 功能清单提取
├── ESB接口清单 → 集成点识别（补充）
└── 人工审核 → 确认准确性
```

## 一、问题诊断

### 1.1 核心发现

| 维度 | 现状 | 启示 |
|------|------|------|
| 评估现状 | 无"特别准确"的标杆案例 | 需要建立黄金数据集 |
| 专家一致性 | 专家之间评估一致 | 主要依赖专家经验，方法论尚未文档化 |
| 最大偏差源 | 需求理解偏差 | 不是评估技术问题，是理解问题 |
| AI定位 | 初评工具，拆分准确率要求高 | **系统识别 + 功能拆分是核心** |
| 澄清价值 | 澄清问题只有"部分影响" | 不能只靠澄清，需提升AI能力 |
| 数据来源 | 台账/架构文档功能清单缺失/不准 | 代码是功能事实来源 |

### 1.2 问题本质

```
AI评估不准 = 系统识别偏差 + 功能拆分偏差 × 复杂度假设错误
```

**结论**：系统识别和功能拆分是准确性提升的"杠杆点"。功能拆分的准确性尤其关键——既不能遗漏功能，也不能夸大范围（擅自添加需求未提及的功能），两者都会造成采购决策风险。

## 二、优化方向全景图

### 2.1 短期优化（1-2周）⭐⭐⭐

#### 方向一：多源数据驱动的系统画像

**目标**：提升系统识别准确率，补充集成点信息

**方案**：
```
系统台账（已存在） → 系统识别
ESB接口清单 → 集成点识别
代码扫描 → 功能清单提取（核心）
```

**数据来源优先级**：
- P0：系统台账（"系统清单配置"已存在）、代码扫描（功能清单提取核心手段）
- P1：典型评估案例（验证优化效果）、ESB接口清单（补充集成点识别）
- P2：黄金数据集（持续优化）、系统架构文档（参考补充）

**产出**：
- 系统画像JSON（包含功能清单、集成点、技术栈）
- 接口→系统的映射关系

---

#### 方向二：功能点分类与拆分规则化

**目标**：提升功能拆分一致性

**方案**：建立功能点分类体系和拆分模板

| 功能类型 | 典型描述 | 基准人天 | 拆分规则 |
|---------|---------|---------|---------|
| 单表查询 | 查询XX信息 | 0.5-1 | 独立功能点 |
| 列表展示 | XX列表+筛选分页 | 1-2 | 独立功能点 |
| 单表CRUD | 新增/编辑XX | 1-2 | 独立功能点 |
| 审批流程 | XX审批 | 2-4 | 按审批节点拆分 |
| 报表统计 | XX报表 | 2-5 | 独立功能点 |
| 接口开发 | 对接XX系统 | 1-3 | 独立功能点 |

**拆分规则**：
```
规则1：每个功能点应该是独立可测试的单元
规则2：复合功能 = 1个主功能 + N个子功能
规则3：跨系统交互单独列为功能点
规则4：数据迁移/配置/测试不作为功能点（但计入总工时）
```

---

#### 方向三：建立小规模黄金数据集

**目标**：建立评估准确性标杆

**实施节奏**：短期建立筛选机制和数据结构，中长期持续积累案例

**选择标准**：
1. 专家评估一致性高（多个专家评估结果接近）
2. 实际执行偏差小（预估vs实际接近）
3. 覆盖典型场景（CRUD/审批/报表/接口等）
4. 有完整的原始需求文档

**数据量目标**：50-100个高质量案例 > 1000个低质量案例（持续积累）

**数据结构**：
```json
{
  "case_id": "CASE-001",
  "requirement_doc": "原始需求文档",
  "expert_evaluation": {
    "systems": ["客户管理系统"],
    "features": [
      {"feature": "客户查询", "days": 1, "complexity": "低"},
      {"feature": "客户创建", "days": 2, "complexity": "中"}
    ]
  },
  "actual_execution": null,
  "deviation_analysis": null
}
```

---

#### 方向九：代码结构分析（核心）

**目标**：提取系统功能清单，补充功能遗漏

**适用场景**：
- 系统台账有系统名称但无功能清单
- 架构文档缺失或过时
- ESB未覆盖的内部功能

**方案**：
```
选择1-2个核心系统试点 → 扫描源文件 → 提取功能清单 → 人工审核 → 推广
```

**技术路径**：
- 扫描项目代码仓库的源文件（Java/其他语言）
- 使用正则/AST解析提取，根据系统实际架构适配：
  - **Web层/API入口**：Controller、Router、Handler、Endpoint等对外接口
  - **业务逻辑层**：Service、Manager、Business等业务处理类
  - **批处理/定时任务**：Scheduled、Job、Task、Batch等
  - **其他模式**：根据系统实际架构（如MVC、分层、微服务等）动态适配扫描策略
- 生成功能清单并与人工审核结合

**试点决策**：
- 若试点系统功能覆盖率>80%，考虑推广
- 若人工审核通过率高，继续推广

**关键问题**：
- 如何过滤琐碎代码？→ 只保留带注解的业务方法、入口类、业务处理类
- 如何转化为人天？→ 建立"代码特征→功能类型→人天"映射

### 2.2 中期优化（1个月）⭐⭐

#### 方向四：相似案例检索与推荐

**目标**：为每个功能点提供参考依据

**方案**：
```
新需求功能点 → 向量化 → 检索Top-K相似案例 → 展示差异点
```

**展示格式**：
```
当前功能点：客户创建

相似案例1：客户列表查询（2人天）- 相似度85%
差异点：
  - 你的是"创建"，案例是"查询" → +0.5人天
  - 你的是"客户"，案例是"会员" → 无差异

相似案例2：会员信息创建（2.5人天）- 相似度78%
差异点：
  - 你的是"客户"，案例是"会员" → 无差异
  - 案例有"数据校验"，你没提 → -0.5人天

综合建议：2±0.5人天，置信度85%
```

---

#### 方向五：多模型集成评估

**目标**：综合多种方法，提高置信度

**模型组合**：

| 模型 | 作用 | 权重 | 输出 |
|------|------|------|------|
| 相似度匹配 | 找历史案例 | 40% | 区间 |
| 规则引擎 | 行业基准 | 30% | 基准值 |
| LLM理解 | 语义拆解 | 30% | 判断 |

**输出格式**：
```
功能点：客户创建

相似案例：1.5-2.5人天（基于3个相似案例）
规则引擎：2人天（标准CRUD基准）
LLM预测：2.5人天（考虑数据校验）

综合建议：2±0.5人天
置信度：85%（高）
```

---

#### 方向六：需求文档质量评分

**目标**：在评估前识别问题文档

**评分维度**：
```
质量评分 = 完整性(30%) + 明确性(30%) + 可理解性(20%) + 结构化(20%)
```

**分级标准**：
| 评分 | 建议 |
|------|------|
| < 60分 | 建议补充需求后再评估 |
| 60-80分 | 可以评估，标注"置信度：低" |
| > 80分 | 正常评估流程 |

---

### 2.3 长期优化（2-3个月）⭐

#### 方向七：持续学习闭环

**目标**：从偏差中学习

**流程**：
```
评估 → 执行 → 对比偏差 → 分析原因 → 更新规则/知识库
```

**偏差分类**：
- **系统识别偏差** → 完善系统画像
- **功能拆分偏差** → 完善拆分规则
- **人天预估偏差** → 更新基准人天

---

#### 方向八：评估置信度标注

**目标**：从"确定数字"转向"区间+置信度"

**输出格式**：
```
功能点：客户创建
评估区间：1.5-2.5人天
置信度：85%（高）
置信依据：
  - 有3个相似案例，偏差在±0.5人天内
  - 系统画像完整，边界清晰
```

**实施建议**（系统当前未实现）：
- 在上传需求时，增加"AI评估质量评价"入口（专家可对AI初评结果标注准确度）
- 在专家打分时，记录专家对AI评估的修正幅度，作为置信度校准依据
- 收集标注数据用于后续模型优化和偏差分析

## 六、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 代码仓库访问困难 | 功能清单无法提取 | 优先处理核心系统，分批获取权限 |
| 专家时间有限 | 功能清单审核缓慢 | 分批进行，优先核心系统 |
| 代码特征映射不准确 | 人天预估偏差 | 先小范围试点，建立映射规则后推广 |
| 优化效果不明显 | 投入产出不成正比 | 先试点验证，再全面推广 |
