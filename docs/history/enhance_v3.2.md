# 需求评估系统优化方案建议书

> 版本：v3.2  
> 日期：2026-02-01  
> 目标：在“用于采购、必须严肃可审计”的前提下，全面提升评估准确性，优先解决 **系统识别准确性** 与 **功能拆分合理性**，并以“区间+置信度+证据”方式输出，降低误用风险。

---

## 零、核心决策记录（对齐 v3.1，补充 v3.2）

### 0.1 数据来源实际情况（v3.1 已确认）

| 数据源 | 实际内容 | 可靠性 | 用途 |
|---|---|---|---|
| 系统台账（系统清单配置） | 系统名称、基本信息（模块/功能清单缺失或不准） | ✅（名称层面） | 规范系统命名、候选系统全集 |
| 非结构化材料（DOCX/PPTX/PDF） | 业务边界、术语、流程、变更背景（口径不一/可能过期） | ⚠️ | 业务能力边界证据、术语/对象抽取、澄清问题 |
| ESB 接口清单 | 跨系统调用接口/对接点 | ⚠️（仅集成点） | 集成点识别、联调风险提示 |
| 代码（Spring Boot/Java） | 实际入口、真实依赖、实现位置 | ✅（事实层） | **功能清单/系统能力目录提取（核心）**、归属校验 |

**结论（v3.1 核心定位）**：当台账/材料无法稳定提供功能清单、需求又缺少技术标识时，必须通过**代码扫描**构建“系统能力目录（功能清单）”作为事实底座；ESB 用于集成点识别；非结构化材料用于业务边界与术语。

### 0.2 采购风险口径（v3.1 核心原则）

```
功能遗漏 = 漏估工作量（采购风险）
功能夸大 = 将需求未提及的功能纳入（更严重的采购风险：高估工作量）
```

核心原则：
- **严格遵循需求文档**，不擅自添加未提及的功能点。
- 代码扫描的“能力目录”用于 **召回与校验**：帮助定位需求可能对应的既有能力、提醒可能的集成/联调工作，但**不得自动扩大需求范围**。

### 0.3 方案定位（v3.2 增强机制）

- **LLM 主导**：语义理解、候选生成、拆分草拟、澄清问题与可读解释。
- **证据约束 + 阻断**：无证据/证据冲突的结论只能输出为 `maybe` 并给出 `questions`，不得进入可直接用于采购的确定结论（`selected`）。
- **可审计**：每个系统归属与功能点都携带最小证据包（来源材料/代码入口/集成点/日期/可信度）。

## 一、背景与问题复述（v3.2的新增约束）

### 1.1 现状约束（用户补充）

1. **系统台账（系统清单）可导入**，但通常只有系统名称，**模块清单缺失**，功能描述也可能不准确/不最新。  
2. **非结构化材料（DOCX/PPTX/PDF）可以提供**，准确度相对更高，但格式/颗粒度不统一、版本不一定最新。  
3. 需求文本通常 **不包含接口码/交易码/服务名**（技术标识缺失）。  
4. 最常见需求：**改造（含接口对接）** 与 **新增功能**。  
5. 多数系统为 **Spring Boot/Java**，代码规模大（20万~50万行，核心系统可接近100万行）。

### 1.2 采购场景的关键要求

- 评估结果可能被专家直接复用/参考，且用于采购：**必须“可追溯、可审计、可解释、可复核”**。
- 不能只追求“看起来很准”的单点数字：要显式管理不确定性，避免把猜测当结论。
- 必须同时控制两类采购风险：**功能遗漏（低估）** 与 **功能夸大（高估）**，且默认更严防止“擅自添加需求未提及功能点”。

### 1.3 核心发现（对齐 v3.1）

| 维度 | 现状 | 启示 |
|---|---|---|
| 评估现状 | 无“特别准确”的标杆案例 | 需要建立小规模黄金数据集 |
| 专家一致性 | 专家之间评估相对一致 | 方法论存在，但需要模板/规则化固化 |
| 最大偏差源 | 需求理解偏差 | 需要证据约束 + 澄清问题机制 |
| AI定位 | 初评工具，拆分准确率要求高 | **系统识别 + 功能拆分是核心杠杆点** |
| 数据来源 | 台账/材料难提供可靠功能清单；ESB仅覆盖集成点 | 代码扫描构建能力目录（事实层）是关键补齐手段 |

### 1.4 问题本质（对齐 v3.1）

```
AI评估不准 = 系统识别偏差 + 功能拆分偏差 × 复杂度假设错误
```

对采购场景而言，除了“准”，更关键是：**功能拆分必须“不遗漏、不夸大”**，且默认更严防止“擅自扩大需求范围”。

## 二、核心结论（v3.2的策略升级）

### 2.1 结论一：LLM可以主导，但必须“被证据约束”

**LLM主导**：用于语义理解、候选生成、拆分草拟、提出澄清问题、形成可读解释。  
**证据主导约束**：决定输出能否进入“确定结论”（selected）而不是“猜测”（maybe）。

一句话：**LLM主导“想”，证据主导“算不算数”。**

### 2.2 结论二：不能过度依赖ESB清单，但必须把它用对

ESB接口清单对“系统内大量功能”覆盖不足，这是事实。  
因此 ESB 在本方案中定位为：
- **集成证据/风险提示**：用于“改造含对接”场景的上下游关系、联调风险、典型变更面；
- **不是唯一定位手段**：系统识别/拆分主要依靠“业务能力边界证据”与“术语/对象语义证据”。

### 2.3 结论三：优先级仍然是“系统识别 + 功能拆分”

人天本质依赖“归属正确 + 粒度正确 + 风险显式化”。  
因此 v3.2 的主线为：

```
系统台账/非结构化材料/ESB/代码扫描（核心系统试点→推广）
        ↓（证据检索 + 候选系统榜单）
系统识别（LLM主导 + 证据约束 + 阻断）
        ↓（按系统）
功能拆分（类型化 + 模板化 + 归属复核）
        ↓
区间估算（置信度 + 风险因子 + 可审计理由）
        ↓
专家复核/修正（记录为黄金数据）
        ↓
持续校准（画像/词表/模板/阈值）
```

## 三、证据体系与知识库设计（核心：业务能力边界）

### 3.1 证据分层（不单押任何单一来源）

| 证据层 | 主要来源 | 覆盖面 | 典型用途 | 风险 |
|---|---|---:|---|---|
| A：业务边界证据（主力） | 架构说明、操作手册、投产/变更说明、评审纪要（DOCX/PPTX/PDF） | 高 | 系统识别、边界校准、术语/对象抽取 | 可能过期、口径不一 |
| B：集成证据（强约束） | ESB/网关服务治理清单 | 中 | 集成关系、联调风险、上下游提示 | 覆盖不到系统内功能 |
| C：实现证据（事实层） | Spring Boot 代码扫描（Controller/Feign/Dubbo/MQ/Job） | 高 | **功能清单/能力目录提取（核心）**、真实入口/依赖校验 | 成本高，需要MVP+人工审核 |
| D：台账证据（底座） | 系统清单/名称/别名/负责人 | 低 | 名称规范化、归档、权限/责任人 | 信息少、不够准 |

> 原则：任何结论都可以由LLM生成，但能否进入“selected/可用于采购结论”，取决于证据层的覆盖与一致性。

### 3.2 知识库两层架构（推荐）

#### A层：系统证据库（非结构化RAG）

目标：让系统识别/拆分“有据可依”，并能对外解释“为什么这么判”。  

**入库对象**：DOCX/PPTX/PDF 的原文切块（chunk），每个 chunk 必须携带最小元数据：

```json
{
  "system_name": "新一代核心",
  "doc_type": "架构说明/操作手册/投产说明/评审纪要/其他",
  "doc_date": "2025-11-08",
  "source_org": "架构组",
  "trust_level": "高/中/低",
  "version_hint": "v3.4",
  "chunk_id": "AUTO-UUID",
  "chunk_text": "……",
  "source_file": "xxx.docx#p12"
}
```

#### B层：系统画像（结构化system_profile，人工可确认）

目标：把A层的“证据碎片”汇总成“业务能力边界画像”，并支持持续修订。  

**字段建议（以业务边界为中心）**：
- `business_goal`（业务目标）
- `core_functions`（核心能力列表，尽量动宾结构）
- `in_scope`（系统做什么/边界）
- `out_of_scope`（系统不做什么/明确排除）
- `business_objects`（关键业务对象/名词）
- `terminology`（别名/简称/术语映射）
- `integration_points`（典型对接点，上下游系统/渠道）
- `key_constraints`（权限、合规、时效、性能、数据一致性等关键约束）

**关键规则**：画像每个字段必须能回链到A层证据来源（至少1条），否则字段标为“待确认”。

### 3.3 防污染机制（采购场景必备）

- **不覆盖、只叠加**：新材料导入不替换旧材料；通过 `doc_date + trust_level` 做排序权重。
- **冲突显式化**：同一系统画像字段若证据冲突，输出“冲突提示+来源列表”，不自动拍板。
- **证据可回放**：每个系统归属/功能点都能看到证据摘要与来源文件/页码（或段落标识）。

### 3.4 代码扫描：系统能力目录/功能清单（v3.1 核心）

目标：从代码中提取“系统能力目录（功能清单）”，用于 **系统识别召回、拆分参考、遗漏校验与归属复核**，解决“台账/材料无稳定功能清单”的根因。

**范围建议（Spring Boot）**：
- Web/API 入口：`@RestController` / `@RequestMapping` / `@GetMapping` 等（过滤 `actuator`、健康检查等非业务端点）
- 定时/批处理：`@Scheduled`、Job/Task/Batch 框架入口
- 消息入口：Kafka/Rabbit 等 Listener（如 `@KafkaListener`）
- 对外调用/依赖：`@FeignClient`、Dubbo、HTTP Client、MQ Producer（用于集成点识别与联调风险提示）

**产出建议（统一 JSON 结构，便于入库/检索/审计）**：
```json
{
  "system_name": "示例系统",
  "generated_at": "2026-02-01T00:00:00",
  "entries": [
    {
      "entry_type": "api",
      "entry_id": "GET /customers/{id}",
      "owner": "com.xxx.customer.CustomerController#get",
      "keywords": ["客户", "查询", "customer"],
      "related_calls": ["Feign:RiskService#check", "MQ:topic_xxx"],
      "summary": "客户信息查询（含风控校验）",
      "risk_tags": ["跨系统对接", "权限"]
    }
  ]
}
```

**采购风控约束**：
- 代码扫描产物是“能力目录”，不是“本次需求范围”。能力目录中**未在需求中出现**的条目，只能作为“可能相关/待确认”提示，不得直接计入估算。
- 代码扫描必须配合 **人工抽样审核**（噪音率/覆盖率）再推广，避免“技术噪音”被误当成功能范围。

## 四、评估链路：LLM主导 + 证据约束 + 阻断

### 4.1 系统识别（目标：减少误识别/漏识别）

#### 4.1.1 输入

- 需求文本（原文）
- 系统清单（名称/别名）
- A层证据库检索命中（Top-K chunk）
- B层已确认画像（如有）
- ESB清单（仅作为集成风险/依赖提示，不要求强匹配）
- 代码扫描产物：系统能力目录/入口清单（用于召回与校验，不自动扩需求范围）

#### 4.1.2 输出（强制分层）

系统识别输出必须分为三类：
- `selected_systems`：证据足、可进入拆分
- `maybe_systems`：证据不足/冲突，必须标注原因
- `questions`：最小澄清集（3~8个），用于把 maybe 转为 selected

示例（概念）：
```json
{
  "selected_systems": [
    {"name": "新一代核心", "confidence": "高", "reasons": ["命中证据:xxx", "边界:开户/账户…"]}
  ],
  "maybe_systems": [
    {"name": "支付中台", "confidence": "低", "reason": "仅有‘支付’字面命中，缺少业务对象/流程证据"}
  ],
  "questions": [
    "本次改造是否涉及支付路由/渠道收单？",
    "需求中的‘对账’是银企对账还是内部账务核对？"
  ]
}
```

#### 4.1.3 阻断规则（严肃模式）

当满足任一条件时，系统不得进入 `selected`：
- 证据命中不足（如：仅关键词命中、无边界/对象证据）
- 证据冲突且未解释（如：两份材料对边界定义相反）
- 需求关键对象/流程无法落到任何画像边界

阻断后允许输出：
- `maybe_systems + questions`
- 以及“需要人工确认后再拆分/再估算”的提示

> 建议默认“保守模式”：宁可多 maybe，也不把猜测当结论输出给采购链路。

### 4.2 功能拆分（目标：粒度一致、可测试、归属可复核）

#### 4.2.1 拆分原则（v3.2明确化）

1. **核心原则：严格遵循需求文档**，不擅自添加未提及的功能点；来自“能力目录/画像”的潜在相关项只能作为 **提示/待确认**，不得直接计入估算。  
2. **功能点=独立可交付/可测试单元**（避免把一坨大需求当一个点）。  
3. **跨系统交互单列功能点**（接口改造/联调/数据映射/权限对接）。  
4. **粒度阈值**：单点建议 0.5~5 人天；>5 必须二次拆分并说明。  
5. **每个功能点必须携带证据标签**：`[归属依据] [集成点] [待确认]`。  
6. **数据迁移/配置/测试/上线联调等**建议作为“非功能工作项/风险因子”显式化，不强行塞进业务功能点，但必须计入总工时（通常体现在区间上限与风险提示里）。

#### 4.2.2 类型化 + 模板化（应对“改造/新增”两大高频）

**先分类，再生成**，减少模型自由发挥导致的颗粒度漂移：

| 功能类型 | 典型描述 | 基准人天 | 拆分规则 |
|---|---|---:|---|
| 单表查询 | 查询XX信息 | 0.5-1 | 独立功能点 |
| 列表展示 | XX列表+筛选分页 | 1-2 | 独立功能点 |
| 单表CRUD | 新增/编辑XX | 1-2 | 独立功能点 |
| 审批流程 | XX审批 | 2-4 | 按审批节点拆分 |
| 报表统计 | XX报表 | 2-5 | 独立功能点 |
| 接口开发/对接 | 对接XX系统 | 1-3 | 独立功能点（联调单列/风险提示） |

**拆分规则（对齐 v3.1）**：
```
规则1：每个功能点应该是独立可测试的单元
规则2：复合功能 = 1个主功能 + N个子功能
规则3：跨系统交互单独列为功能点
规则4：数据迁移/配置/测试不作为业务功能点（但计入总工时）
```

- 改造类（含接口对接）：优先拆出
  - 变更对象（字段/规则/流程节点）
  - 影响面（入口/报表/批处理/权限）
  - 对接点（上下游系统、联调、报文/字段映射）
- 新增功能类：优先拆出
  - 新业务对象（主数据/交易/关系）
  - 新流程（角色、状态流转、审批/校验）
  - 新查询/报表（维度、口径、导出）

#### 4.2.3 归属复核（减少跨系统误拆）

对以下“高风险信号”触发跨系统复核（输出提示，不强行挪动）：
- 关键词：对账/报送/清算/征信/影像/审批引擎/主数据/统一认证/签章/短信邮件等
- 明显跨系统动作：调用、对接、同步、推送、回调、订阅、接口联调

### 4.3 人天输出：区间+置信度（而非伪精确单点）

#### 4.3.1 输出形态

每个功能点输出：
- `estimate_range`：如 `[1.5, 2.5]`
- `confidence`：高/中/低
- `key_factors`：导致上下限的关键因子（数据量、规则复杂度、集成点数量、非功能约束）
- `assumptions`：假设清单（未澄清的点）

#### 4.3.2 采购风控规则

- 当 `confidence=低` 或 `assumptions` 数量超过阈值：必须提示“不可直接用于采购定价，需要专家确认/补充需求”。  
- 当系统识别存在 `maybe_systems` 且数量>0：总工时只能给“区间总量 + 风险提示”，不输出单点“确定总人天”。

## 五、验证体系：如何衡量“更准了”

### 5.1 P0指标（优先）

| 指标 | 目标 | 测量方式 |
|---|---:|---|
| 系统识别准确率（Precision优先） | >90% | `selected_systems` 与专家/经理最终确认对比 |
| 漏系统率（Recall） | 持续下降 | 专家最终确认里AI漏掉的系统比例 |
| 功能拆分一致性 | >80% | 专家最终拆分与AI拆分的重合度（可用Jaccard/人工抽样） |
| 功能清单覆盖率（试点系统） | >80% | 代码扫描提取的能力目录与用户/专家审核版功能清单对比 |
| 功能遗漏率（AI拆分遗漏） | 持续下降 | 专家最终拆分里 AI 漏掉的功能点比例 |
| 功能夸大率（AI多拆/扩范围） | 持续下降（更关注） | AI 拆分中“需求未提及但被纳入”的功能点比例 |
| 低置信度占比 | 可控 | 按系统/功能点统计（用于风险管理，不是越低越好） |

### 5.2 P1指标（辅助）

- 专家编辑成本：平均修改条数/耗时下降
- 关键字段修改率：系统归属/功能点命名/依赖关系的修改率下降
- 证据命中率：每个 selected 系统/功能点是否都有可回放证据

## 六、实施路线图（先试点，再推广）

> 原则：先“单点击穿”（系统识别+拆分可审计），再 All-in 扩大覆盖；避免一次性引入全量代码分析导致过度工程化。

### 阶段0：试点准备（1周）

- 定义证据元数据标准（system_name、doc_type、doc_date、trust_level…）
- 选 2 个试点系统（一个问题系统 + 一个对照系统）
- 每个系统准备 3~5 份材料（新旧各一份更佳，DOCX/PPTX/PDF）
- 获取 1~2 个试点系统代码仓库只读访问权限（用于能力目录提取）
- 导出 ESB 接口清单（如可，作为集成点/联调风险提示）
- 准备 2~3 个典型需求包（用于前后对比验证）
- 明确“严肃模式”阈值（默认保守）

### 阶段1：代码扫描MVP（功能清单/能力目录）+ 证据库导入（1~2周）

- 开发代码扫描工具（MVP，面向 Spring Boot）：Controller/Job/MQ/Feign/Dubbo 等入口与依赖提取
- 执行 1~2 个系统扫描，生成“系统能力目录（功能清单）”
- 用户+专家抽样审核能力目录（噪音率/覆盖率），形成“审核版能力目录”
- A层证据库导入（非结构化材料，携带元数据）
- 导入 ESB 接口清单（仅集成点/上下游/联调风险提示）

### 阶段2：系统识别升级（1~2周）

- 系统台账作为候选系统全集；融合 A层证据库 + 审核版能力目录 + ESB 集成点
- 输出分层（selected/maybe/questions）+ 阻断规则落地（默认保守）
- 建立“系统校准卡片”：候选榜单/证据摘要/缺失画像提示（便于人工复核）

### 阶段3：功能拆分模板化 + 区间估算（1~2周）

- 建立功能类型模板库（CRUD/列表/审批/报表/接口对接等）
- 功能点输出强制证据标签 + 归属复核提示（减少误拆/漏拆）
- 人天输出“区间+置信度+假设”，并在低置信度/假设过多时给出阻断提示
- 记录专家修正（系统归属修正/拆分合并拆分/区间调整）作为后续训练/校准数据

### 阶段4：黄金数据集 + 推广（2~3个月）

- 沉淀 50~100 个高质量案例（优先从试点系统与高频需求类型开始）
- 推广代码扫描到其他核心系统（以 ROI 门槛推进，优先误判频发/材料不足系统）
- 上线中期能力：相似案例检索、多模型集成、需求文档质量评分、持续学习闭环、置信度校准

## 七、资源需求（最小可行）

### 7.1 数据

- 系统清单（已有）
- 试点系统材料（DOCX/PPTX/PDF）：每系统 3~5 份
- 试点系统代码仓库只读访问权限（1~2个系统）
- ESB接口清单：作为集成证据与风险提示（可选但推荐）
- 典型需求包（2~3个）：需求原文 + 专家最终系统/拆分结果（用于对比验证）

### 7.2 业务/专家参与（低成本）

- 每周 1 次 30~60 分钟：对试点输出做抽样复核（主要是系统归属与拆分粒度）
- 选 2~3 个典型需求包：用于前后对比
- 能力目录/功能清单人工审核：建议 1~2 小时/系统（先抽样后全量）

## 八、风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 材料过期/口径冲突 | 误判边界 | 元数据+新鲜度/可信度加权；冲突显式化；阻断不拍板 |
| 需求技术标识缺失 | 难直接匹配接口 | 用“业务对象/术语/边界证据”做主匹配，ESB仅做集成提示 |
| LLM幻觉 | 采购风险 | 证据约束+阻断；无证据只能maybe；输出强制带来源 |
| 代码仓库访问困难 | 无法构建能力目录 | 优先试点系统分批获取只读权限；阶段性用非结构化材料兜底 |
| 能力目录被误当成需求范围 | 高估采购（严重风险） | 严格“需求约束”：未在需求出现只能“待确认/提示”，不得直接计入估算 |
| 代码扫描范围失控/噪音高 | 成本高、推进慢、误导拆分 | MVP+入口过滤（业务端点优先）+人工抽样审核；以ROI门槛推广 |
| 专家时间有限（审核/标注） | 审核慢、黄金数据集难积累 | 抽样审核+分批推进；优先核心系统与高频需求类型 |
| 结果被直接复用 | 放大错误 | 默认保守；低置信度不得输出可用于采购的单点结论 |

## 九、附录

### 9.1 术语

- **证据库（A层）**：非结构化材料切块后的可检索证据集合，用于支撑系统识别/拆分的可追溯依据。  
- **系统画像（B层）**：围绕“业务能力边界”的结构化摘要，可人工确认，作为更稳定的约束输入。  
- **功能清单覆盖率**：代码扫描提取的能力目录，经人工审核后对系统实际可见功能的覆盖程度（试点阶段可用抽样评估）。  
- **阻断**：当证据不足/冲突时，禁止输出确定结论，只输出 maybe + 澄清问题。

### 9.2 输出建议（给前端/报告）

- 系统维度：候选榜单 + 证据卡片 + 置信度 + 澄清问题
- 功能点维度：类型标签 + 归属依据 + 集成点 + 假设/待确认 + 区间+置信度
