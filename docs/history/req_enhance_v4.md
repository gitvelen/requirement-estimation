# 需求评估系统升级工程需求文档（V4）

> 文档版本：v4.0  
> 编写日期：2026-02-01  
> 输入来源：`enhance_v3.2.md`（已对齐 `enhance_v3.1.md` 关键决策）  
> 本期目标：在“用于采购、必须严肃可审计”的前提下，优先提升 **系统识别准确性** 与 **功能拆分合理性**，并控制两类采购风险：**功能遗漏（低估）** 与 **功能夸大（高估）**（默认更严防止范围扩张）。

---

## 0. 术语与约定

- **系统台账/系统清单**：企业已有的系统列表（名称为主，模块/功能清单缺失或不准）。
- **系统标识（system_id）**：系统简称/编码（如 `SOFP`），用于跨数据源对齐；系统名称（system_name）用于展示。本需求不规定数据库主键设计。
- **证据库（A层）**：非结构化材料（DOCX/PPTX/PDF）切块后的可检索证据集合，用于支撑系统识别/拆分的可追溯依据。
- **系统画像（B层）**：围绕“业务能力边界”的结构化摘要，可人工确认，作为更稳定的约束输入；每个字段需可回链证据来源。
- **能力目录/功能清单（事实层）**：由代码扫描产出，用于召回与校验；不得自动扩大需求范围。
- **证据等级（evidence_level）**：系统/功能点在本次任务中可用证据的综合等级（E0~E3），用于决定是否允许进入 `selected`、是否触发严肃模式阻断。
- **selected/maybe/questions**：系统识别分层输出：确定系统/不确定系统/最小澄清问题集。
- **严肃模式（阻断）**：证据不足或冲突时禁止输出“可直接用于采购”的确定结论，仅允许输出 `maybe + questions` 并提示需人工确认。
- **未接入系统（un-onboarded）**：缺少代码与材料等关键证据的系统；严肃模式下只能输出 `maybe + questions`，不得形成可用于采购的确定结论。

### 0.1 默认参数（可配置，本期默认值）

| 参数 | 默认值 | 说明 |
|---|---:|---|
| 严肃模式 | 开启 | 默认启用阻断规则 |
| A层证据检索 Top-K | 5 | 用于系统识别/拆分注入的证据块数量 |
| A层证据检索阈值 | 0.60 | 低于阈值视为"命中不足"；相似度算法：余弦相似度 |
| ESB检索 Top-K | 8 | 用于集成点摘要/澄清问题候选 |
| ESB检索阈值 | 0.55 | 场景/服务名更短，阈值略放宽 |
| ESB检索范围 | 提供方+调用方 | 默认同时按“提供方/调用方”两种视角命中；可切换为仅提供方或仅调用方（可配置） |
| ESB有效状态 | 正常使用 | 默认仅“有效状态”参与检索/注入/统计；其它状态（如“废弃使用”）仅留痕不默认参与 |
| ESB包含废弃条目 | false | `include_deprecated=false` 时，废弃条目不进入检索与统计口径（可配置） |
| questions 数量 | 3~8 | 超出则按优先级截断（Top-N） |
| assumptions 阈值 | 4 | `assumptions_count >= 4` 触发采购风险提示/低置信度 |
| selected_systems 最大数 | 10 | 防止输出过长影响审阅效率 |
| maybe_systems 最大数 | 8 | 超出则保留更高证据/更高召回价值者 |

> 推荐排序分数（用于相似度接近时的“可信/新鲜”加权）：`rank_score = similarity × trust_weight × recency_weight`。  
> `trust_weight` 默认：高=1.00，中=0.85，低=0.70；`recency_weight`：≤180天=1.00，181~365天=0.90，366~730天=0.80，>730天=0.70，缺失日期=0.75。

## 1. 背景与目标

### 1.1 背景

现有评估链路中，AI 需要先做“系统识别 + 功能拆分”，专家可能直接参考或复用 AI 结果，且评估结果用于采购决策。因此：
- **功能遗漏**会导致漏估（采购风险）。
- **功能夸大/擅自扩范围**会导致高估（更严重的采购风险）。

同时，实际数据源现状为：
- 台账/材料无法稳定提供可靠功能清单；
- 需求文本通常缺少接口码/服务名等技术标识；
- ESB 接口清单只覆盖集成点；
- 代码（Spring Boot/Java）是“功能事实层”的可行来源。

### 1.2 本期目标（P0）

1. **系统识别更准且可审计**：输出 `selected/maybe/questions`，每个结论可回放证据。
2. **功能拆分更合理且不扩范围**：拆分遵循需求，不擅自新增；每个功能点带证据标签与待确认项。
3. **输出可用于采购的风控形态**：人天输出“区间+置信度+假设”，低置信度/假设过多时触发阻断提示。
4. **形成可持续校准闭环**：记录专家修正，沉淀黄金数据集的最小结构。

### 1.3 非目标（本期不做）

- 不做端到端“自动采购定价/自动招采报价”。
- 不做模型训练/微调、也不引入复杂中台（K8s/CI-CD/全链路可观测性等）。
- 不做全量代码语义理解；代码侧仅做“入口/依赖/能力目录”的轻量事实抽取。
- 不强制支持“扫描版 PDF 的 OCR”（本期仅保证可提取文本的 PDF）。

## 2. 角色与权限（沿用现有系统）

| 角色 | 本期新增/调整的能力 |
|---|---|
| 项目经理（manager） | 维护证据库/画像/能力目录（试点），发起评估任务并查看系统识别/拆分的证据与风险提示；导入时审核证据材料与能力目录 |
| 管理员（admin） | 仅配置与用户/系统清单管理；证据库权限可选开放（默认关闭，可通过配置开启） |
| 专家（expert） | 不直接维护知识库；在评估页可看到"证据摘要/置信度/假设"，并可**在线预览证据原文全文（DOCX/PPTX/PDF，仅在线预览，不提供下载）**；访问范围仅限其参与任务关联的证据材料 |

## 3. 总体流程（V4升级后的主链路）

```
系统台账 + 证据库(A) + 系统画像(B) + 能力目录(代码扫描) + ESB(集成点)
        ↓（证据检索 + 候选系统榜单）
系统识别（LLM 主导 + 证据约束 + 严肃模式阻断）
        ↓（按系统）
功能拆分（类型化/模板化 + 证据标签 + 归属复核）
        ↓
区间估算（置信度 + 假设 + 风险因子 + 可审计理由）
        ↓
专家评估与修正（记录为黄金数据）
        ↓
持续校准（证据/画像/模板/阈值）
```

## 4. 功能需求（可验收）

> 本章节按“功能模块 → 需求点 → 验收标准”描述。编号用于研发拆分与测试追踪。

### 4.1 证据库（A层）：非结构化材料入库与检索（FR-A）

#### FR-A1 证据材料导入

**描述**：项目经理可为某个系统导入 DOCX/PPTX/PDF 材料，系统将其解析为文本并切块入库，保存为“证据块”。  

**最小元数据（必填/可选）**：
- 必填：`system_name`、`doc_type`、`trust_level(高/中/低)`
- 可选：`doc_date`、`source_org`、`version_hint`

**验收标准**：
1. 仅允许 `manager` 导入；无权限返回 403。
2. 导入后可查询到证据块数量增加，且每条证据块包含 `system_name/source_file/chunk_id` 等可回放字段。
3. 对于无法抽取文本的文件（如扫描 PDF），导入应失败并给出可理解错误（而不是“导入成功但无内容”）。

#### FR-A2 证据检索

**描述**：输入查询文本（例如需求片段），可按 `system_name` 过滤检索证据块，返回 Top-K 证据及相似度。  

**验收标准**：
1. 检索结果每条包含：`similarity`、`chunk_text`（可截断）、`source_file`、`doc_date/trust_level`（如有）。
2. 支持 `system_name` 过滤与不指定系统的全局检索两种模式。
3. 默认 `top_k=5`、`similarity_threshold=0.60`（可配置）。
4. 相似度算法：余弦相似度（由向量数据库/检索引擎决定）。
5. 对同一查询，返回排序需受 `trust_level + doc_date` 加权影响（相似度接近时优先更可信/更新材料；默认加权见"0.1 默认参数"）。

#### FR-A3 证据可回放

**描述**：在系统识别/功能拆分结果中，每个结论需携带“最小证据包”。  

**验收标准**：
1. 在任务详情/编辑页可查看每个 `selected_system` 的证据摘要列表（至少 1 条）。
2. 每条证据可定位到来源文件（文件名+页码/段落标识或等价定位信息），并支持"在线预览全文"（不提供下载）：
   - 优先：原文件渲染预览（如 PDF 直接预览，DOCX/PPTX 可转为 PDF/HTML 预览）
   - 降级：若无法渲染原文件，则展示“抽取文本预览”，并提示“版式可能与原文件不一致”
3. 预览与检索边界：图片内容仅展示，不做 OCR，不参与检索。
4. 权限控制：专家仅可查看其参与任务关联证据的全文；不得浏览全库证据材料。

### 4.2 系统画像（B层）：业务能力边界画像（FR-B）

#### FR-B1 画像字段与证据回链

**描述**：为每个系统维护结构化画像（以业务能力边界为中心），字段包括：
- `business_goal`
- `core_functions`（列表）
- `in_scope`
- `out_of_scope`
- `business_objects`（列表）
- `terminology`（别名/术语映射）
- `integration_points`（上下游/渠道/对接点）
- `key_constraints`（关键约束）

每个字段需可回链到证据来源（A层证据块或能力目录条目）。

**验收标准**：
1. 画像支持创建/编辑/发布（发布态用于在线评估，草稿态不参与约束）。
2. 发布画像中每个非空字段至少关联 1 条证据引用；缺少引用时该字段必须标记为“待确认”。

#### FR-B2 画像在系统识别/拆分中的注入

**描述**：当系统画像存在时，系统识别与功能拆分需根据画像状态区别处理。

**画像状态处理规则**：
- **已发布画像**：作为强约束，系统识别与功能拆分必须优先注入画像内容用于约束。
- **草稿态画像**：作为弱参考信号，具体行为：
  - **不影响相似度分数计算**：rank_score计算时不因草稿画像存在而调整
  - **不作为证据引用**：不计入"最少1条证据"的要求
  - **仅用于提示性文字**：在候选系统理由/澄清问题建议中引用，并标注"草稿态"

**验收标准**：
1. 对同一需求，在"有已发布画像"与"无画像"两种情况下，输出需体现差异（如：有画像时 `maybe` 减少或理由更充分）。
2. 当引用草稿态画像时，输出需标注"草稿态"提示，且不作为强制约束依据。

### 4.3 代码扫描：能力目录/功能清单（事实层）（FR-C）

#### FR-C1 代码扫描（MVP）

**描述**：对 **Java/Spring Boot** 系统代码仓执行扫描，抽取能力目录（入口/依赖/关键词），用于系统识别召回与拆分校验。

**抽取范围（MVP，最小支持清单）**：
- API 入口：`@RestController`/`@Controller`/`@RequestMapping`/`@GetMapping`/`@PostMapping` 等
- 定时/批处理入口：Spring `@Scheduled`（Quartz/XXL-Job 等其他框架本期不保证）
- 消息入口：`@KafkaListener`、RabbitMQ `@RabbitListener`（其他 MQ 本期不保证）
- 对外依赖：`@FeignClient`（扫调用方代码）、Dubbo Consumer/Provider（扫对应侧代码）、HTTP Client、MQ Producer

产出"能力目录 JSON"，可导入作为知识条目供检索与引用。

**验收标准**：
1. 能对示例工程（Java/Spring Boot）扫描输出能力目录（至少包含 API 入口与依赖调用）。
2. 能覆盖上述“抽取范围（MVP）”中的至少 3 类入口/依赖（例如：Controller + Feign + Scheduled）。
3. 能过滤明显非业务端点（如 health/actuator），或提供可配置过滤规则并默认开启。
4. 能力目录每条条目至少包含：`entry_type/entry_id/owner/summary/keywords/related_calls`（字段允许为空但必须存在）。

#### FR-C2 能力目录用于召回与校验（不得扩范围）

**描述**：能力目录用于：
- 帮助系统识别召回候选系统；
- 帮助功能拆分校验遗漏与归属；
- 提示潜在集成/联调点。

但能力目录条目 **不得自动转成需求范围**。

**验收标准**：
1. 在功能拆分结果中，如果引用能力目录作为提示，必须标记为 `[待确认]`，且默认不计入“确定功能点”。
2. 若需求文本未出现对应业务对象/行为，系统不得仅凭能力目录自动新增功能点。

### 4.4 ESB 接口服务清单：集成点索引与场景词表（FR-I）（P0）

> 背景：需求文本通常不包含接口码/交易码/服务名等强技术标识，但 ESB 清单包含 **交易名称/服务场景/提供方/调用方/状态** 等信息，可用于"集成点识别、场景词表、联调风险提示、澄清问题候选"，并为缺少代码/材料的系统提供部分兜底。  
> 重要约束：ESB 仅覆盖“跨系统集成点”，**不能**作为系统内功能清单直接拆分业务功能点（防止范围扩张）。

#### FR-I1 ESB 清单导入

**描述**：支持导入 ESB 服务治理明细表（交易/场景/提供方/调用方），形成“可检索的集成点索引”。  

**可选：系统维度统计/字典表导入（用于对齐 system_id 与补充规模信息）**：
- 字段示例：`system_id/system_name/owner/center/total_interface_count/no_call_interface_count`（其余列按 `extra` 留存）
- 约束：仅作为系统元数据与风险提示信号，不参与“系统内功能点拆分”。

**导入假设（按现状口径写死）**：
- 一行 = 一个调用方（同一交易/场景可能对应多行不同调用方）。

**最小字段（明细表）**：
- 必填：`provider_system_id`、`consumer_system_id`、`service_name`、`status`
- 推荐：`service_code`（交易码）、`scenario_code`（服务场景码）、`provider_system_name`、`consumer_system_name`、`remark`

**默认字段映射（优先按列名自动识别，可在导入时配置映射）**：
- `service_code` ⇐ `交易码`
- `scenario_code` ⇐ `服务场景码`
- `provider_system_id` ⇐ `提供方系统简称`
- `provider_system_name` ⇐ `提供方中文名称`
- `service_name` ⇐ `交易名称`
- `consumer_system_id` ⇐ `调用方系统简称`
- `consumer_system_name` ⇐ `调用方中文名称`
- `status` ⇐ `状态`
- `remark` ⇐ `备注`

**验收标准**：
1. 仅允许 `manager` 导入；无权限返回 403。
2. 支持导入格式：CSV + XLSX（允许多 Sheet，按列名自动识别；无法识别时需提示“缺少必填列”并失败）。
3. 导入后可按 `provider_system_id/consumer_system_id` 查询到该系统关联的 ESB 条目数。
4. 状态过滤默认值：
   - `valid_statuses=["正常使用"]`
   - `"废弃使用"` 导入后仅留痕，不参与默认检索/注入/统计口径（`include_deprecated=false`）。
5. 支持基本去噪：缺少必填字段的行不入库，并在导入结果中返回“跳过数量+原因摘要”。

#### FR-I2 ESB 索引检索与统计

**描述**：支持按“需求文本/关键词”检索相关 ESB 条目，并用于生成“集成点摘要”。  

**验收标准**：
1. 检索结果至少包含：`provider_system_id/consumer_system_id/service_name/scenario_code/status/similarity`（或等价排序分数）；如有名称字段需同时返回 `provider_system_name/consumer_system_name`。
2. 支持按系统过滤检索（优先按 `system_id`，兼容按 `system_name`），并支持检索范围：
   - `scope=both`（默认）：系统作为提供方或调用方均可命中
   - `scope=provider`：仅按系统作为提供方命中
   - `scope=consumer`：仅按系统作为调用方命中
3. 提供按系统维度统计（默认仅统计“有效状态”）：
   - `active_entry_count`（有效条目数）
   - `deprecated_entry_count`（废弃条目数，可选展示）
   - `active_unique_service_count`（有效交易名称去重数，可用于联调风险提示）
   - 如导入了“系统维度统计表”，额外展示：`total_interface_count/no_call_interface_count`（用于风险提示，不影响拆分范围）
4. 默认 `top_k=8`、`similarity_threshold=0.55`、`include_deprecated=false`（可配置）。

#### FR-I3 在系统识别/功能拆分中的使用（不扩范围）

**描述**：
- 系统识别阶段：ESB 的 `service_name(交易名称)` 与 `provider/consumer` 信息用于提升候选系统召回、生成更贴近业务的 `questions`，以及输出“可能涉及的集成点”提示（不要求需求中出现交易码/场景码）。
- 功能拆分阶段：当需求明确涉及“对接/同步/回调/联调/订阅”等跨系统动作时，可引用 ESB 明细条目作为 `[集成点]` 证据或提示，输出建议的“提供方/调用方/交易名称/服务场景码”。

**验收标准**：
1. ESB 只能作为“集成点证据/提示”，不得被当作系统内功能清单直接生成业务功能点。
2. 若引用 ESB 条目而需求未明确提及该对接/场景，必须标记为 `[待确认]` 且默认不计入确定估算汇总。
3. 若命中条目的 `status` 非“正常使用”（如“废弃使用”），必须标记为 `[疑似废弃]`，且默认不计入确定结论/确定集成点。

---

### 4.5 系统识别：LLM 主导 + 证据约束 + 阻断（FR-S）

#### FR-S1 输出分层（selected/maybe/questions）

**描述**：系统识别必须输出三部分：
- `selected_systems`：可进入拆分
- `maybe_systems`：证据不足或冲突
- `questions`：最小澄清问题集（优先级排序，面向业务可回答）

**questions 数量控制**：
- 目标数量：3~8个高优先级问题
- 超出时：按"能否把 maybe 转为 selected"的优先级排序，取 Top-N
- 展示时：可按优先级分组（高/中/低），允许用户展开查看更多

**验收标准**：
1. 输出 JSON 结构固定；字段缺失视为不通过。
2. 每个 `selected_system` 必须关联至少 1 条证据引用（来自 A/B/C 任一层）。
3. `questions` 必须与 `maybe_systems` 的"不确定原因"对齐（能把 maybe 转为 selected）。
4. 当问题数量超过 8 个时，按优先级排序并高亮前 3~8 个，其余可折叠展示。
5. 默认 `selected_systems<=10`、`maybe_systems<=8`（可配置）；超出时按证据等级/召回价值截断并说明截断原因。

#### FR-S2 严肃模式阻断

**描述**：当证据不足/冲突或关键对象无法落到任何系统边界时，禁止输出确定结论。  

**验收标准**：
1. 触发阻断时：`selected_systems` 为空或仅包含证据充分的系统；其余系统进入 `maybe_systems` 并给出 `questions`。
2. 阻断提示在 UI 中明确展示：“需人工确认后再拆分/估算，当前结果不可直接用于采购结论”。

---

### 4.6 功能拆分：类型化/模板化 + 证据标签 + 归属复核（FR-F）

#### FR-F1 拆分原则（防止范围扩张）

**描述**：
- 严格遵循需求文档，不擅自添加需求未提及功能点；
- 功能点需独立可测试；
- 跨系统交互单列（接口改造/联调/数据映射/权限对接）；
- 单点建议 0.5~5 人天，>5 必须二次拆分并说明；
- 每个功能点必须携带标签：`[归属依据] [集成点] [待确认] [非功能工作项]`；
- 数据迁移/配置/测试/上线联调等以 `[非功能工作项]` 标签形式，挂载在相关业务功能点上，计入区间上限。

**非功能工作项处理**：
- 作为标签挂在某条业务功能点上（如"客户创建"功能点带 `[非功能工作项: 测试/上线联调]` 标签）。
- 在人天估算时，非功能工作项计入区间。
- 纯非功能工作场景（如"纯数据迁移"、"纯环境配置"）不在功能点工作量评估考虑范围，转为人工评估。
- 汇总时非功能工作项不单独列支，通过标签可查看明细。

**验收标准**：
1. 生成的每条功能点必须包含上述标签（至少留空占位也可）。
2. 若出现"疑似新增范围"的功能点，必须标记为 `[待确认]` 且默认不进入确定估算汇总。
3. 非功能工作项以标签形式挂载，并在人天区间中体现影响。

#### FR-F2 类型化模板（对齐 v3.1）

**描述**：提供并使用功能类型模板：
- 单表查询、列表展示、单表CRUD、审批流程、报表统计、接口开发/对接等。

**验收标准**：
1. 拆分结果中每条功能点必须标注“功能类型”（或可推断的模板ID）。
2. 对“改造/新增”两类高频需求，拆分结构需体现其差异（改造优先影响面/对接点，新建优先对象/流程/查询报表）。

#### FR-F3 归属复核（高风险信号）

**描述**：对包含高风险信号（对账/报送/清算/征信/影像/审批引擎/统一认证/签章/短信等）或明显跨系统动词（对接/同步/回调/订阅/联调）的功能点，触发跨系统归属复核提示（不自动挪动）。  

**验收标准**：
1. 触发复核的功能点必须追加可读提示（如“疑似更偏向系统X，建议复核”）并附证据/相似度（如有）。
2. 系统不得自动改变功能点归属（保持“提示不自动挪”的原则）。

---

### 4.7 人天输出：区间+置信度+假设（FR-E）

#### FR-E1 输出结构

**描述**：每个功能点必须输出：
- `estimate_range`（min/max）
- `confidence`（高/中/低）
- `assumptions`（未澄清假设）
- `key_factors`（上下限关键因子，如数据量/规则复杂度/集成点数量/NFR）

**区间估算算法**（混合模式）：
- **基准区间**：从功能类型模板取基准范围（如"单表CRUD"基准1-2人天）。
- **动态调整**：
  - 有明确规则时用规则（如：每多1个集成点+0.5~1人天，数据量级超预期+20~50%）
  - 无明确规则时由 LLM 依据 `assumptions`/`key_factors` 调整
- **两者结合**：最终区间 = 基准区间经规则或 LLM 调整后得出（可记录调整依据及方式）。

**验收标准**：
1. 任一功能点缺少上述字段，视为不通过。
2. 当 `confidence=低` 或 `assumptions_count>=4` 时，必须显示采购风险提示，并在汇总处标记"不可直接用于采购结论"（阈值可配置，默认值见“0.1 默认参数”）。
3. 强制低置信度规则：当 `evidence_level` 为 E0/E1 时，`confidence` 必须为“低”（不得输出中/高）。

---

### 4.8 指标与记录（FR-M）

#### FR-M1 关键指标落库（文件存储即可）

**描述**：系统需记录用于验收与后续校准的指标：
- 系统识别准确率（Precision）、漏系统率（Recall）
- 功能拆分一致性
- 功能清单覆盖率（试点）
- 功能遗漏率（AI 漏拆，相对专家最终修正）
- 功能夸大率（AI 多拆/扩范围，相对专家最终修正）
- 低置信度占比

**计算基准**：
- **本期以“专家最终修正结果”作为基准**（用于验收与校准）；后续如接入“实际执行”数据，可增加第二套口径对比。
- 功能遗漏率 = (专家最终有而 AI 未拆分的功能点数) / (专家最终总功能点数)。
- 功能夸大率 = (AI 多拆而专家最终不需要的功能点人天) / (专家最终总人天)。

**验收标准**：
1. 完成一次任务后可在"效果统计/报告"中查看上述指标的基础统计（允许试点阶段为抽样统计）。

---

### 4.9 兜底策略：证据等级与降级输出（FR-Z）

> 目标：解决“无代码库/无材料”的系统接入问题，并在采购严肃模式下避免输出不可控结论。

#### FR-Z1 证据等级（evidence_level）计算

**描述**：对每个候选系统与功能点计算证据等级（E0~E3），用于驱动是否允许 `selected` 与是否触发阻断。

**规则**（可配置，本期支持配置界面，以下为默认值）：
- **E3（高）**：具备 `FR-C` 能力目录（代码扫描）+（`FR-A` 证据命中 或 `FR-B` 已发布画像）。
- **E2（中）**：具备 `FR-A` 证据命中 或 `FR-B` 已发布画像，并且具备 `FR-I`（ESB）或 `FR-C`（能力目录）之一作为补证据。
- **E1（低）**：仅具备 `FR-I`（ESB），或具备 `FR-A` 但无其他补证据（能力目录/发布画像/ESB）。
- **E0（无）**：除系统名称外无任何可用证据。

**配置能力**：
- 支持通过配置界面调整证据等级判定规则（如增删证据源权重、调整阈值）。
- 配置变更需记录变更日志，便于追溯影响范围。
- 支持项目经理对单个系统的 `evidence_level` 做“结果修正”（当判定结果不合理时），但必须填写备注，且仅影响当前任务（不改变全局规则）。

**验收标准**：
1. 系统识别输出中，每个系统包含 `evidence_level` 与 `missing_evidence`（缺少哪些证据源）。
2. UI 能展示证据等级，并提示"如何补齐"。
3. 提供配置界面允许调整证据等级规则（默认值如上），并记录变更人/时间/变更前后内容。
4. 权限控制：
   - 规则配置：仅 `admin` 可修改证据等级规则；`manager` 可查看。
   - 结果修正：`manager` 可对任务内单个系统的 `evidence_level` 修正，但必须备注并记录修正日志（修正前/后、原因、操作者、时间）。

#### FR-Z2 降级输出与阻断

**描述**：当证据等级不足时，必须降级输出并控制风险。

**规则**（默认严肃模式）：
- E3/E2：允许进入 `selected_systems`，并进入拆分/估算。
- E1：默认不得进入 `selected_systems`，只能进入 `maybe_systems + questions`；如需强行确认，由 `manager` 单方确认即可，但必须低置信度并扩大区间。
- E0：阻断（不进入拆分/估算），仅输出"缺失证据+补齐清单+问题集"。

**验收标准**：
1. E1/E0 场景下，系统明确提示“当前结果不可直接用于采购结论”。
2. 阻断时仍输出可执行的最小澄清问题集（3~8个）与补齐建议（材料/代码/画像最小表单）。
3. 强行确认需留痕：记录确认人（manager）、确认时间、确认理由；并在报告中标记 `manual_override=true`。
4. 强行确认作用域：仅影响**当前任务**（不改变系统全局证据等级规则/不改变系统的默认 evidence_level）。

#### FR-Z3 最小补齐（系统画像最小表单）

**描述**：当系统缺少材料与代码时，提供“最小补齐表单”（可由 `manager` 手工填写）以解除阻断，字段至少包含：
- `in_scope/out_of_scope`
- `core_functions`（Top10）
- `business_objects`
- `integration_points`

**UI提示（本期要求）**：
- 当系统“无代码 + 无材料”且依赖最小画像进入 `selected` 时，必须在表单顶部与任务详情页显著提示（高风险标识），并在报告中保留该标识。
- 建议文案（可调整）：**“该系统未接入代码库且未上传材料，本次识别/拆分基于项目经理填写的最小画像，存在较高不确定性，请重点复核。”**

**验收标准**：
1. 未接入系统在补齐并发布最小画像后，证据等级至少达到 E2（或达到可配置门槛），允许进入 `selected`。
2. 若通过最小画像进入 `selected` 时仍为“无代码 + 无材料”，任务页与报告必须展示上述高风险提示，并在数据结构中保留标识（例如：`missing_evidence` 包含 `code/material`）。

## 5. 接口需求（建议稿，供研发评审）

> 具体路径可与现有 API 规范对齐；本节提供最小接口集合，便于可验收交付。

### 5.1 证据库（A层）

- `POST /api/v1/knowledge/evidence/import`：导入证据材料（DOCX/PPTX/PDF + 元数据）
- `POST /api/v1/knowledge/evidence/search`：检索证据块（支持 system_name 过滤/全局）
- `GET /api/v1/knowledge/evidence/stats?system_name=`：证据统计

### 5.2 画像（B层）

- `GET /api/v1/system-profiles/{system_name}`：获取画像（含草稿/发布态）
- `PUT /api/v1/system-profiles/{system_name}`：保存画像（字段+证据引用）
- `POST /api/v1/system-profiles/{system_name}/publish`：发布画像

### 5.3 代码扫描与能力目录

- `POST /api/v1/code-scan/run`：触发扫描（system_name + repo_path + options；`repo_path` 为后端可访问的本地仓库路径）
- `GET /api/v1/code-scan/status/{job_id}`：查询进度
- `GET /api/v1/code-scan/result/{job_id}`：获取能力目录
- `POST /api/v1/code-scan/commit/{job_id}`：将能力目录入库（知识条目）

### 5.4 ESB 服务清单

- `POST /api/v1/knowledge/esb/import`：导入 ESB 明细表（交易/场景/提供方/调用方/状态；支持列映射）
- `POST /api/v1/knowledge/esb/search`：检索 ESB 条目（支持按 `system_id/system_name` 过滤、`scope=both|provider|consumer`、`include_deprecated`）
- `GET /api/v1/knowledge/esb/stats?system_id=&system_name=`：ESB 统计（有效条目数/废弃条目数/有效交易去重数等）

### 6.1 P0 验收（必须全部通过）

1. 证据库支持 DOCX/PPTX/PDF 导入与检索，且证据可回放（来源文件可定位）。
2. 系统识别输出严格为 `selected/maybe/questions`，并实现严肃模式阻断（无证据不进 selected）。

4. 人天输出为"区间+置信度+假设"，低置信度时有明确采购风险提示。
5. 代码扫描能对示例工程（Java/Spring Boot）产出能力目录，并用于召回/校验（不自动扩范围）。
6. 对"无代码/无材料"的系统具备兜底：输出证据等级、按等级降级/阻断，并给出可执行的补齐清单与最小澄清问题集。
7. ESB 服务清单支持导入/检索/统计，并在系统识别/拆分中提供"交易/场景"级集成点提示（不扩范围）。
8. 证据等级规则支持配置界面调整、权限控制（仅admin可改）与变更留痕。

### 6.2 P1 验收（建议通过）

1. 画像字段具备证据回链与发布态控制，并在系统识别/拆分中生效。
2. 指标面板能查看"功能遗漏/夸大/覆盖率"等关键指标（试点允许抽样统计）。
