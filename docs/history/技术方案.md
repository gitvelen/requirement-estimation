## 0. 摘要（Executive Summary）

需求评估系统将“需求文档”转化为可用于采购/投标/评审的“可审计工作量评估”输出。系统核心链路为：

1. 需求文档上传与解析（抽取需求正文）
2. 系统识别（`selected/maybe/questions` 分层输出）
3. 按系统拆分功能点（模板化输出、带证据标签与归属复核提示）
4. 区间估算与风险提示（置信度、假设、风控标识）
5. 多专家多轮盲评与偏离度分析（偏离度触发下一轮）
6. 报告生成与版本管理（Excel为主，必要时PDF）

为提升准确性与可追溯性（尤其采购严肃场景），系统引入四类约束/事实信号并实现权限与审计闭环：

- A层证据库：DOCX/PPTX/PDF 材料导入、切块检索与在线预览（不下载）
- B层系统画像：能力边界结构化摘要（草稿/发布），字段可回链证据
- I层ESB索引：集成点明细导入/检索/统计，用于提示与澄清问题候选
- C层代码扫描：Spring Boot/Java 轻量扫描产出能力目录（事实层），用于召回与归属校验提示（禁止扩范围）

当前实现以“文件存储（data/*.json）+ 文件锁 + 原子写”为主，适合单机/小规模快速落地；可平滑演进至数据库/对象存储/作业系统以支持规模化。

## 1. 背景、目标、非目标与约束

### 1.1 背景与问题

传统需求拆分与工作量评估存在：

- 拆分与估算高度依赖专家经验，效率低且难复用
- 口径不统一，导致同类需求估算偏差大
- 过程不可追溯：为什么判断是这个系统？为什么拆成这些点？依据是什么？
- 采购/投标场景中，“漏估/扩范围”会直接带来成本与合规风险

### 1.2 目标（Goals）

- G1 准确性优先：优先提升系统识别准确性与拆分合理性，再给出人天
- G2 可审计：每个系统/功能点输出必须可回链到证据或事实依据
- G3 严肃模式阻断：证据不足或冲突时禁止输出确定结论，仅输出 `maybe + questions`
- G4 人机协作闭环：项目经理/专家可以修正，修正需留痕并形成后续校准数据
- G5 可部署：提供一键部署能力，默认资源成本可控

### 1.3 非目标（Non-Goals）

- N1 不做端到端自动报价/自动招采定价（仅做评估与风控提示）
- N2 不做全量代码语义理解（代码扫描仅做事实层能力目录提取）
- N3 禁止自动扩范围（无论证据/代码/ESB召回到什么，都只能“提示复核”，不得自动新增需求范围）
- N4 不强制OCR扫描版PDF（仅支持可抽取文本的PDF入库检索）

### 1.4 关键约束（Constraints）

- C1 当前存储为文件：单机可用；多实例强一致不保证（需演进）
- C2 证据预览：仅在线预览，不提供下载（专家范围需按任务收敛）
- C3 模型依赖：DashScope Key 未配置时系统可启动，但 AI/Embedding 能力可能不可用或走降级策略（以部署配置为准）

## 2. 质量属性与验收口径（Quality Requirements）

### 2.1 质量目标（优先级排序）

1. 正确性：不扩范围；证据不足触发阻断；权限边界不可突破
2. 可追溯性：输出带证据引用；覆盖修正可回放
3. 可用性：部署后可完成登录与主流程；健康检查可用
4. 可维护性：模块边界清晰；可在不影响主链路的情况下迭代证据/画像/ESB/扫描
5. 性能：中小规模单机可用；可通过Milvus/异步化扩展

### 2.2 典型质量场景（Quality Scenarios）

- Q-01（安全）：专家预览证据必须校验其参与任务，否则 403；且不提供下载
- Q-02（正确性）：证据不足时，系统识别必须降级为 `maybe + questions` 并显著提示高风险
- Q-03（可追溯）：项目经理对任务内证据等级覆盖修正必须强制填写备注并写入审计
- Q-04（鲁棒性）：证据导入若无法抽取文本（扫描PDF），必须失败并给出可理解原因
- Q-05（可用性）：Nginx对外80端口可访问登录页，`/api/v1/health` 返回 healthy

## 3. 术语与数据分层（Domain Glossary）

### 3.1 分层与输出约定

- 系统识别分层：
  - selected：当前证据足以“确定涉及该系统”（严肃模式下需满足证据等级规则）
  - maybe：证据不足或存在冲突，仅作为候选，需澄清
  - questions：最小澄清问题集（3~8条为宜）
- 证据分层：
  - A层证据库：非结构化材料切块检索（可回放、可预览）
  - B层系统画像：结构化能力边界（草稿/发布）
  - I层ESB索引：集成点明细（只代表集成点，不代表功能清单）
  - C层能力目录：代码扫描事实层（用于召回与校验提示，不扩范围）
- 证据等级（E0~E3）：综合反映“本任务对该系统的证据可用性”，用于阻断与风险提示

#### 4.3.1 主链路时序（简化）

```

前端 → 后端：POST /api/v1/requirement/upload
后端 → 文档解析：抽取需求正文
后端 → Agent编排：系统识别 → 功能拆分 → 区间估算（含风险）
后端 → 存储：写 task_storage.json（原子写）

manager → 后端：保存功能点/系统操作/重拆等
manager → 后端：提交给管理员
admin → 后端：分配专家
expert → 后端：拉取评估数据、保存草稿、提交
后端：计算偏离度，必要时触发下一轮
admin/manager：下载报告（版本管理）
```

#### 4.3.2 证据预览时序（专家）

```
expert → 后端：GET /api/v1/knowledge/evidence/preview/{doc_id}?task_id=...
后端：校验 expert 是否参与 task_id；校验 doc_id 是否在该任务关联证据中
后端：PDF 返回 inline FileResponse；否则返回抽取文本预览（JSON）
```

## 6. 数据与存储（当前文件存储方案，详细）

### 6.1 总体原则

当前采用“文件存储 + 文件锁 + 原子写（tmp + replace）”：

- 写入前获取 `.lock` 文件互斥锁（fcntl，不支持时降级为进程内锁）
- 修改写入使用 `xxx.tmp` 临时文件，成功后 `os.replace` 原子替换
- 读写均做容错：文件不存在返回默认空结构；解析失败记录日志并返回兜底

### 6.2 数据目录布局

```
data/
  task_storage.json                # 任务主数据
  task_modifications.json          # 修改记录
  users.json                       # 用户与专家库字段
  departments.json                 # 部门下拉（可选）
  notifications.json               # 通知
  activity_logs.json               # 活动日志
  knowledge_store.json             # 本地向量库（LocalVectorStore）
  evidence_docs.json               # 证据材料元数据（A层）
  system_profiles.json             # 系统画像（B层）
  esb_index.json                   # ESB索引（I层）
  code_scan_jobs.json              # 扫描作业索引（C层）
  evidence_level_rules.json        # 证据等级规则（系统级）
  audit_logs.json                  # 审计日志（规则变更/覆盖修正等）
  templates/                       # XLSX模板等（用户导入模板、系统清单模板等）
  v4_demo/                         # 演示数据与指标快照（如存在）
uploads/
  evidence/<system>/<doc_id>/...   # 证据原件（不下载，仅预览）
logs/
  app.log                          # 后端日志
```

#### 6.3.1 users.json（用户与专家库字段）

用途：

- 登录认证（username/password_hash）
- 权限（roles）
- 专家分配校验（is_active、on_duty、roles包含expert）
- 个人中心展示（display_name、avatar等）

结构（示例）：

    "id": "user_admin",

    "department": "xxx",

约束：

- username 全局唯一
- roles 取值：admin/manager/expert（可多角色）
- 专家分配必须满足：roles含expert、is_active=true、on_duty=true

## 7. 核心模块详细设计（比设计文档更细的“实现级”说明）

> 本章按“模块 → 职责 → 关键流程 → 边界/异常 → 关键参数”描述，直接对应代码落点。

### 7.1 认证与权限（JWT + 角色校验 + Admin API Key）

职责：

- 提供登录、获取当前用户、改密等能力
- 支撑角色权限校验（admin/manager/expert）
- 对管理写接口提供二级保护（X-API-Key 或管理员token）

实现要点：

- JWT：
  - 由 `backend/api/auth.py` 创建与校验（HS256）
  - token payload 含 user_id/username/roles
  - `Authorization: Bearer <token>` 作为业务接口默认鉴权方式
- 角色校验：
  - `require_roles([...])` 做路由级保护
  - 注意“多角色”用户（manager+expert）菜单与默认页逻辑
- 管理写接口保护：
  - `require_admin_api_key` 支持两种方式：
    1) X-API-Key = ADMIN_API_KEY
    2) Bearer token 且 user roles 含 admin
  - 生产建议配置 ADMIN_API_KEY；DEBUG 下可能放行（便于调试）

风险与建议：

- 生产必须修改 `JWT_SECRET`，并配置 `ADMIN_API_KEY`
- 建议为登录失败/鉴权失败补充统一错误响应与限流（防爆破）

### 7.2 任务与工作流（Task Workflow）

职责：

- 管理任务生命周期：创建、AI处理、编辑、提交、分配、评估、多轮、报告、归档/删除
- 管理任务数据结构与默认字段补齐（兼容旧任务结构）

实现要点（文件存储）：

- 任务主存储：`data/task_storage.json`
- 并发控制：跨进程文件锁 + 原子写
- 兼容补齐：对旧任务缺失字段进行 `setdefault` 补齐（保证前端/接口可用）

工作流状态（建议口径）：

- draft：草稿（AI完成但未提交管理员）
- awaiting_assignment：待分配（manager提交后）
- evaluating：评估中（已分配专家，可能多轮）
- completed：已完成（评估完成且报告生成）
- archived：已归档（从活跃列表收起）

关键边界：

- 多轮评估：基于偏离度阈值触发（默认 20%），最多轮数默认 3
- 删除/归档：需要明确权限边界（admin/manager），并考虑审计与报告留存

### 7.3 Agent 编排（系统识别/拆分/区间估算）

职责：

- 将“模型输出”放入可控框架：召回信号注入 + 规则约束 + 结果落盘

设计原则：

- LLM 主导生成，但必须被“证据/事实/规则”约束
- 严肃模式下：证据不足禁止给确定结论
- 输出可审计：每条结论携带 reasons/evidence/risk_flags

编排输入（按优先级）：

1) 需求正文（requirement_content）
2) 系统清单（主系统/子系统映射）
3) B层画像（发布态强约束，草稿态弱提示）
4) A层证据检索摘要（Top-K证据块）
5) I层ESB摘要（集成点与澄清问题候选）
6) C层能力目录摘要（入口/接口/调用点）

编排输出：

- `ai_system_analysis`：selected/maybe/questions（含 evidence_level/confidence/reasons/evidence）
- `systems_data`：按系统的功能点列表（含 estimate_range/confidence/assumptions/risk_flags/evidence_level/remark）

关键异常：

- 模型/embedding不可用：任务仍可进入“人工编辑链路”，但需明确提示“AI能力不可用/已降级”

### 7.4 文档解析与内容抽取（DOCX/PPTX/PDF）

职责：

- 从上传文件中抽取可用于识别/拆分的文本
- 对证据库导入提供“按页/段/slide”的结构化切片

实现要点：

- DOCX：段落与表格解析
- PPTX：按slide提取文本
- PDF：按页提取文本（不做OCR）

边界：

- 扫描版PDF（无可提取文本）应视为导入失败（证据库场景）
- 大文件限制：导入类接口默认 50MB 上限

### 7.5 向量检索与Embedding（LocalVectorStore / 可选Milvus）

职责：

- 提供 A层证据、系统知识、能力目录、ESB 条目的相似检索

当前默认实现：

- 向量存储：`LocalVectorStore`，落盘 `data/knowledge_store.json`
- 检索：全量扫描 + 余弦相似度
- 可配置：Top-K、threshold

Embedding：

- 默认对接 DashScope embedding
- 允许在缺 Key 时启动服务（具体降级策略由部署配置决定）

建议参数（与需求/场景保持一致）：

- 证据检索：Top-K=5，threshold=0.60
- ESB检索：Top-K=8，threshold=0.55，scope=both，include_deprecated=false，valid_status=正常使用

性能与演进：

- 本地向量库适合小规模；数据量增大建议切 Milvus
- 建议增加 metadata filter 能力（系统/类型/日期/信任等级）以减少无效召回

### 7.6 证据库（A层）：导入、切块、检索、预览与权限

职责：

- 证据材料导入：保存原件、抽取文本、切块、embedding、入向量库
- 证据检索：按系统/任务范围检索证据块，并返回定位信息
- 证据预览：仅在线预览，不提供下载；PDF优先原件内嵌，否则文本预览
- 权限边界：专家只能查看其参与任务关联证据

导入流程（实现级）：

1) 校验文件名、大小、doc_type（docx/pptx/pdf）
2) 生成 doc_id，并写入 `uploads/evidence/<system>/<doc_id>/...`
3) 解析并构建 segments（pdf按页、docx按段/表、pptx按slide）
4) segments 进一步 split 成 chunks（chunk_size=800，overlap=120，最多200块）
5) 对 chunk 计算 embedding，并写入 `knowledge_store.json`（knowledge_type=evidence_chunk）
6) 写入 `evidence_docs.json` 元数据（trust_level/doc_date等）

检索流程（实现级）：

- 输入 query + 可选 system_name/system_id + task_id
- expert 角色要求 task_id（否则 400）
- service 层校验 actor 是否可访问（task内证据范围约束）
- 在向量库中检索 Top-K，并返回：
  - similarity、chunk_text（可截断）、doc_id、loc、trust_level、doc_date、source_file 等

预览流程（实现级）：

- `GET /preview/{doc_id}?task_id=...`
- PDF：FileResponse（Content-Disposition inline）
- 非PDF：返回抽取文本（JSON），并提示“版式可能不一致”

异常与边界：

- 扫描PDF（无文本）→ 导入失败并提示
- 原件丢失/路径不存在 → 404
- 权限越界（专家不在任务范围）→ 403

### 7.7 系统画像（B层）：草稿/发布、证据回链、注入策略

职责：

- 维护“业务能力边界”结构化摘要
- 发布态作为强约束注入系统识别与拆分
- 每字段可回链证据引用以支持审计

字段建议（与实现保持一致）：

- business_goal / core_functions / in_scope / out_of_scope
- business_objects / terminology / integration_points / key_constraints

状态语义：

- draft：弱参考信号（不作为证据引用、不影响rank_score，仅用于提示文字）
- published：强约束（用于系统识别与拆分约束，且应具备 evidence_refs）

### 7.8 ESB 索引（I层）：导入、检索、统计与使用边界

职责：

- 导入 ESB 明细表（CSV/XLSX），产出可检索的集成点索引
- 用于“集成点摘要与澄清问题候选”（非功能清单）

导入要点：

- 自动识别列名（可支持 mapping_json 适配不同表头）
- 默认仅“正常使用”参与检索/统计；废弃条目可留痕但不默认参与

检索要点：

- scope：both/provider/consumer（默认 both）
- include_deprecated：默认 false
- 输出包含：provider/consumer、service_code/scenario_code/service_name、状态与来源行号

边界：

- ESB 仅代表集成点，不能自动扩需求范围

### 7.9 代码扫描（C层）：作业、结果、提交与边界

职责：

- 对 Spring Boot/Java 仓库做轻量扫描，产出“能力目录”（入口/接口/任务/消息等）
- 用于召回与归属校验提示（事实层），禁止扩范围

作业语义：

- run：提交 repo_path，生成 job_id
- status：查询进度与状态
- result：查看扫描结果
- commit：提交结果进入可用索引（用于后续召回）

边界：

- 不承诺扫描所有框架与语言（当前MVP聚焦Spring Boot）
- 扫描结果仅提示，不得自动新增功能点

### 7.10 证据等级规则与任务内覆盖（Rule Engine + Override）

职责：

- admin 配置证据等级规则（系统级）
- manager 允许对任务内单系统证据等级覆盖修正（需备注、留痕）
- 在“严肃模式”下用于阻断（决定 selected 是否允许）

规则与审计：

- 更新规则写入审计日志（包含操作者与新规则内容）
- 任务内覆盖写入任务数据 + 审计日志（包含备注）

### 7.11 区间估算与风险提示（Interval Estimation）

职责：

- 输出 estimate_range + confidence + assumptions + risk_flags
- 在低置信度/假设过多时触发风险提示与（严肃模式下）阻断提示

建议口径：

- estimate_range：`min~max`（人天）
- confidence：0~1
- assumptions：3~8条为宜，超过阈值触发风险标识
- risk_flags：例如 MISSING_EVIDENCE / HIGH_RISK_NO_CODE / SCOPE_UNCERTAIN 等

### 7.12 多专家多轮盲评与偏离度（Delphi-like）

职责：

- 支持 3 位专家独立盲评
- 支持多轮：偏离度超过阈值触发下一轮（最多3轮）
- 输出偏离度列表与轮次进度

实现要点：

- 每轮评估以 feature_id 作为稳定键聚合
- 计算专家均值、与AI估算（或区间中值）偏离度

### 7.13 报告生成与版本管理

职责：

- 生成 Excel 报告（主输出）
- 生成 PDF（可选，用于摘要/效果统计）
- 报告版本管理：每轮或每次确认生成新版本，支持下载历史

报告建议包含：

- 汇总：系统数、功能点数、总人天（区间/均值）、风险项统计
- 明细：按系统的功能点列表、证据等级、证据摘要/定位、专家评分、偏离度
- 审计：关键覆盖修正备注、规则版本信息（至少可回链）

### 7.14 通知、个人中心与活动日志

- 通知：任务分配、提交、报告生成等触发；支持未读数与批量已读
- 个人中心：资料、头像、活动日志（登录/改密等）
- 活动日志与审计日志是两个概念：
  - 活动日志偏“个人行为记录”
  - 审计日志偏“关键治理与可追责操作”

## 9. 安全与合规（Security）

### 9.1 数据分级（建议口径）

- 需求正文、拆分结果、估算结果：可能包含业务敏感信息
- 证据原件：通常为内部材料，属于高敏资产（必须纳入备份与权限控制）
- ESB明细：含系统间集成信息，敏感
- 代码扫描结果：可能暴露接口与路径信息，敏感

### 9.2 核心安全控制

- 访问控制：
  - JWT + 角色校验
  - 管理写接口二级保护（X-API-Key 或管理员token）
- 最小权限：
  - 专家只能在任务范围内预览/检索证据
  - 不提供证据下载
- 输入安全：
  - 上传文件大小与类型限制
  - 路径清洗避免目录穿越
- 密钥管理：
  - 生产必须修改 JWT_SECRET
  - 建议将 DASHSCOPE_API_KEY / ADMIN_API_KEY 通过安全的机密管理系统注入

### 9.3 建议补强项（后续演进）

- 登录与管理接口限流/防爆破
- 审计日志不可篡改（落库+WORM策略）
- 证据原件水印/访问审计

## 10. 可靠性与性能（Reliability / Performance）

### 10.1 可靠性策略

- 文件写入：锁 + 原子写
- 长任务：可通过后台线程/作业ID查询（部分模块已有雏形）
- 降级：
  - 模型不可用时允许进入人工编辑链路
  - 证据预览渲染失败时降级为抽取文本预览

### 10.2 性能关键点

- LocalVectorStore：全量扫描，数据量上来会慢（建议切Milvus）
- 证据切块参数：chunk_size/overlap/max_chunks 直接影响导入耗时与检索质量
- 报告生成：大任务可能耗时，建议后续异步化

容量与压测建议（后续补齐）：

- 单机：证据块数 < 10万 时可接受；超过建议 Milvus + metadata filter

  - `DASHSCOPE_API_KEY`（不配置可启动，但AI/Embedding可能不可用或降级）
  - `KNOWLEDGE_VECTOR_STORE=local|milvus`

### 13.3 数据备份建议

- 至少备份：`data/`、`uploads/`、`logs/`
- 证据原件属于审计关键材料，必须纳入备份与访问控制

### 13.4 常见故障排查（摘要）

- 无法进入登录页：
  - 检查 80 端口/安全组/防火墙
  - `docker-compose ps` 看 frontend 是否运行
- `/api` 502：
  - 检查 `frontend/nginx.conf` 是否反代到 `backend:8000`
  - 查看日志：`docker-compose logs -f frontend` 与 `docker-compose logs -f backend`
- AI/Embedding 不可用：
  - 检查 `DASHSCOPE_API_KEY`
  - 若启用降级策略，确认降级配置是否打开

## 14. 风险、限制与技术债（Risks / Tech Debt）

- 文件存储一致性：适合单机；规模化需落库/对象存储/分布式锁
- LocalVectorStore 性能：数据量增大需Milvus
- 文档预览能力：DOCX/PPTX 目前以抽取文本预览为主（版式不保证一致）
- 模型不确定性：需靠证据约束、阻断、人机协作降低风险

## 15. 演进路线（Roadmap）

按“先稳住准确性与审计，再做规模化”的顺序推进：

1) 存储演进：任务/评估/审计落库（PostgreSQL），证据原件上对象存储（MinIO）  
2) 检索演进：Milvus + metadata filter，支持更大规模与更强过滤  
3) 作业系统：导入/扫描/报告异步化（队列 + worker），提升可靠性与可观测性  
4) 安全与合规：细粒度授权、审计不可篡改、脱敏与水印、限流与告警

## 19. 目标态数据库设计（用于规模化演进，详细）

> 当前实现以文件为主。本章提供“目标态数据库”设计，便于后续规模化落库，且字段命名尽量与现有JSON结构对应，降低迁移成本。

### 19.1 设计原则

- 内部主键使用 UUID（或雪花ID）保证跨表引用稳定
- 任务/评估/审计等关键对象“不可篡改字段”单独建表或保留版本号
- 证据原件进对象存储（MinIO/S3），DB只存元数据与访问策略
- 向量索引使用 Milvus/pgvector（二选一），DB存 metadata 与权限边界

### 19.2 ER 概览（建议）

```
users ──< user_roles
  │
  ├──< audit_logs
  │
tasks ──< task_systems ──< features ──< feature_evidence_refs
  │             │
  │             └──< task_system_overrides
  │
  ├──< expert_assignments ──< evaluations
  │
  ├──< task_evidence_refs ── evidence_docs ──< evidence_chunks (向量/元数据)
  │
  ├──< report_versions
  │
  └──< notifications

system_profiles ──< profile_evidence_refs
esb_entries（可选向量索引）
code_scan_jobs ──< code_scan_items（能力目录，可选向量索引）
evidence_level_rules（版本化）
```

### 19.3 表结构（DDL示例，建议）

#### 19.3.1 users / user_roles

```sql
CREATE TABLE users (
  id              UUID PRIMARY KEY,
  username        VARCHAR(50) UNIQUE NOT NULL,
  display_name    VARCHAR(100) NOT NULL,
  password_hash   VARCHAR(255) NOT NULL,
  email           VARCHAR(100),
  phone           VARCHAR(20),
  department      VARCHAR(100),
  avatar_url      VARCHAR(255),
  on_duty         BOOLEAN NOT NULL DEFAULT TRUE,
  is_active       BOOLEAN NOT NULL DEFAULT TRUE,
  last_login_at   TIMESTAMP,
  created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE user_roles (
  user_id   UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  role      VARCHAR(20) NOT NULL,
  PRIMARY KEY (user_id, role)
);

CREATE INDEX idx_user_roles_role ON user_roles(role);
```

### 19.4 文件存储 → 落库迁移建议

- 第一步：保持接口契约不变，只替换底层存储实现（service层）
- 第二步：为每个对象引入稳定 id（现有 feature 已有 id；system_list 已引入稳定 id 语义）
- 第三步：将 evidence 原件迁移到对象存储，并保留原路径映射/访问审计
- 第四步：将向量索引迁移到 Milvus，并在DB保存 metadata 与权限边界

## 20. 发布与升级策略（Release / Upgrade）

### 20.1 版本与兼容策略

- API 版本：固定前缀 `/api/v1`，小版本兼容升级（新增字段不破坏旧前端）
- 数据兼容：
  - task_storage.json 在读取时执行 schema 补齐（setdefault）
  - 新字段尽量“可选”并提供默认值，避免老数据导致前端渲染失败

### 20.2 生产发布建议（Checklist）

- 安全：
  - 修改 `JWT_SECRET`
  - 配置 `ADMIN_API_KEY`
  - 检查 CORS 白名单（若不是同域反代）
- 可用性：
  - `GET /api/v1/health` 返回 healthy
  - 登录/任务列表/上传页面可访问
- 数据：
  - 备份 `data/`、`uploads/`、`logs/`
  - 检查证据原件权限与访问审计策略
- 运行：
  - 确认 80 端口对外可访问
  - 确认 `/api` 反代指向 backend:8000

### 20.3 升级回滚建议

- 升级前备份：`data/` 与 `uploads/`
- 变更尽量“向后兼容”（新增字段/新增文件），避免破坏旧数据
- 回滚时优先回滚容器镜像；数据保持不回滚（除非已做明确迁移脚本）
