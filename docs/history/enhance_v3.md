# 需求评估系统优化方案建议书

> 版本：v3.1
> 日期：2026-02-01
> 目标：全面提升评估准确性，聚焦系统识别与功能拆分

---

## 一、问题诊断

### 1.1 核心发现（基于苏格拉底式对话）

| 维度 | 现状 | 启示 |
|------|------|------|
| 评估现状 | 无"特别准确"的标杆案例 | 需要建立黄金数据集 |
| 专家一致性 | 专家之间评估一致 | 方法论存在，但未文档化 |
| 最大偏差源 | 需求理解偏差 | 不是评估技术问题，是理解问题 |
| AI定位 | 初评工具，拆分准确率要求高 | **系统识别 + 功能拆分是核心** |
| 澄清价值 | 澄清问题只有"部分影响" | 不能只靠澄清，需提升AI能力 |

### 1.2 问题本质

```
AI评估不准 = 系统识别偏差 + 功能拆分偏差 × 复杂度假设错误
```

**结论**：系统识别和功能拆分是准确性提升的"杠杆点"。

## 二、优化方向全景图

### 2.1 短期优化（1-2周）⭐⭐⭐

#### 方向一：接口文档驱动的系统画像

**目标**：提升系统识别准确率

**方案**：
```
服务总线接口清单 → 提取系统边界 → 构建系统画像 → 提升系统识别
```

**数据来源**：
- 服务总线接口治理文档（最可靠）
- 现有系统架构文档（需整理）
- 代码结构（可选，仅针对核心系统试点，见方向九）

**产出**：
- 标准化的系统画像JSON
- 接口→系统的映射关系

---

#### 方向二：功能点分类与拆分规则化

**目标**：提升功能拆分一致性

**方案**：建立功能点分类体系和拆分模板

| 功能类型 | 典型描述 | 基准人天 | 拆分规则 |
|---------|---------|---------|---------|
| 单表查询 | 查询XX信息 | 0.5-1 | 独立功能点 |
| 列表展示 | XX列表+筛选分页 | 1-2 | 独立功能点 |
| 单表CRUD | 新增/编辑XX | 1-2 | 独立功能点 |
| 审批流程 | XX审批 | 2-4 | 按审批节点拆分 |
| 报表统计 | XX报表 | 2-5 | 独立功能点 |
| 接口开发 | 对接XX系统 | 1-3 | 独立功能点 |

**拆分规则**：
```
规则1：每个功能点应该是独立可测试的单元
规则2：复合功能 = 1个主功能 + N个子功能
规则3：跨系统交互单独列为功能点
规则4：数据迁移/配置/测试不作为功能点（但计入总工时）
```

---

#### 方向三：建立小规模黄金数据集

**目标**：建立评估准确性标杆

**选择标准**：
1. 专家评估一致性高（多个专家评估结果接近）
2. 实际执行偏差小（预估vs实际接近）
3. 覆盖典型场景（CRUD/审批/报表/接口等）
4. 有完整的原始需求文档

**数据量**：50-100个高质量案例 > 1000个低质量案例

**数据结构**：
```json
{
  "case_id": "CASE-001",
  "requirement_doc": "原始需求文档",
  "expert_evaluation": {
    "systems": ["客户管理系统"],
    "features": [
      {"feature": "客户查询", "days": 1, "complexity": "低"},
      {"feature": "客户创建", "days": 2, "complexity": "中"}
    ]
  },
  "actual_execution": null,  // 后续补充实际执行情况
  "deviation_analysis": null  // 偏差分析
}
```

### 2.2 中期优化（1个月）⭐⭐

#### 方向四：相似案例检索与推荐

**目标**：为每个功能点提供参考依据

**方案**：
```
新需求功能点 → 向量化 → 检索Top-K相似案例 → 展示差异点
```

**展示格式**：
```
当前功能点：客户创建

相似案例1：客户列表查询（2人天）- 相似度85%
差异点：
  - 你的是"创建"，案例是"查询" → +0.5人天
  - 你的是"客户"，案例是"会员" → 无差异

相似案例2：会员信息创建（2.5人天）- 相似度78%
差异点：
  - 你的是"客户"，案例是"会员" → 无差异
  - 案例有"数据校验"，你没提 → -0.5人天

综合建议：2±0.5人天，置信度85%
```

---

#### 方向五：多模型集成评估

**目标**：综合多种方法，提高置信度

**模型组合**：

| 模型 | 作用 | 权重 | 输出 |
|------|------|------|------|
| 相似度匹配 | 找历史案例 | 40% | 区间 |
| 规则引擎 | 行业基准 | 30% | 基准值 |
| LLM理解 | 语义拆解 | 30% | 判断 |

**输出格式**：
```
功能点：客户创建

相似案例：1.5-2.5人天（基于3个相似案例）
规则引擎：2人天（标准CRUD基准）
LLM预测：2.5人天（考虑数据校验）

综合建议：2±0.5人天
置信度：85%（高）
```

---

#### 方向六：需求文档质量评分

**目标**：在评估前识别问题文档

**评分维度**：
```
质量评分 = 完整性(30%) + 明确性(30%) + 可理解性(20%) + 结构化(20%)
```

**分级标准**：
| 评分 | 建议 |
|------|------|
| < 60分 | 建议补充需求后再评估 |
| 60-80分 | 可以评估，标注"置信度：低" |
| > 80分 | 正常评估流程 |

---

### 2.3 长期优化（2-3个月）⭐

#### 方向七：持续学习闭环

**目标**：从偏差中学习

**流程**：
```
评估 → 执行 → 对比偏差 → 分析原因 → 更新规则/知识库
```

**偏差分类**：
- **系统识别偏差** → 完善系统画像
- **功能拆分偏差** → 完善拆分规则
- **人天预估偏差** → 更新基准人天

**自动化记录**：
- 每次专家修正自动记录
- 定期生成偏差分析报告
- 自动识别高频偏差模式

---

#### 方向八：评估置信度标注

**目标**：从"确定数字"转向"区间+置信度"

**输出格式**：
```
功能点：客户创建
评估区间：1.5-2.5人天
置信度：85%（高）
置信依据：
  - 有3个相似案例，偏差在±0.5人天内
  - 系统画像完整，边界清晰

功能点：数据迁移
评估区间：2-5人天
置信度：60%（中）
置信依据：
  - 数据量未知
  - 迁移规则不明确
  - 建议澄清后再评估
```

---

#### 方向九：代码结构分析（试点）

**目标**：验证代码分析对评估准确性的提升效果

**适用场景**（仅以下情况考虑）：
1. ESB未覆盖的核心系统（约20%）
2. 系统架构文档缺失或过时
3. 系统复杂度高，评估偏差频繁

**方案**：
```
选择1-2个核心系统试点 → 扫描Controller/API文件 → 提取接口清单 → 补充系统画像
```

**技术路径**：
- 扫描项目代码仓库的Controller/API文件（不包含全量代码）
- 使用正则/AST解析提取接口定义（服务名、路径、参数）
- 生成接口清单并与ESB数据合并，构建完整系统画像

**试点决策**：
- 若试点系统评估准确率提升>15%，考虑推广到其他核心系统
- 若提升不明显，停止代码分析方向

## 六、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 接口清单获取困难 | 系统画像无法构建 | 先用现有架构文档，逐步完善 |
| 专家时间有限 | 黄金数据集建设缓慢 | 分批进行，优先核心系统 |
| 历史数据质量差 | 无法用于训练 | 只用专家修正后的高质量案例 |
| 优化效果不明显 | 投入产出不成正比 | 先小范围验证，再全面推广 |
