## 当前需求（截面式）

### 1. 目标与范围

#### 1.1 目标

- **G1 准确性优先**：优先提升系统识别准确性与拆分合理性，再给出人天
- **G2 可审计**：每个系统/功能点输出必须可回链到证据或事实依据
- **G3 严肃模式阻断**：证据不足或冲突时禁止输出确定结论，仅输出 `maybe + questions`
- **G4 人机协作闭环**：项目经理/专家可以修正，修正需留痕并形成后续校准数据
- **G5 可部署**：提供一键部署能力，默认资源成本可控

#### 1.2 范围（In Scope）

- 需求文档自动解析（DOCX格式）
- AI驱动的系统识别（`selected/maybe/questions`分层输出）
- 按系统拆分功能点（模板化、带证据标签）
- 区间估算与风险提示（置信度、假设、风控标识）
- 多专家多轮盲评与偏离度分析
- 报告生成与版本管理
- 证据库（A层）：DOCX/PPTX/PDF材料导入与检索
- 系统画像（B层）：业务能力边界结构化摘要
- ESB索引（I层）：集成点明细导入/检索/统计
- 代码扫描（C层）：Spring Boot/Java能力目录提取
- 证据等级规则（E0~E3）与严肃模式阻断

#### 1.3 非目标（Non-goals）

- **N1** 不做端到端自动报价/自动招采定价（仅做评估与风控提示）
- **N2** 不做全量代码语义理解（代码扫描仅做事实层能力目录提取）
- **N3** 禁止自动扩范围（无论证据/代码/ESB召回到什么，都只能"提示复核"，不得自动新增需求范围）
- **N4** 不强制OCR扫描版PDF（仅支持可抽取文本的PDF入库检索）
- **N5** 本期不做K8s部署与CI/CD流水线
- **N6** 本期不做数据库持久化（PostgreSQL）和Redis缓存
- **N7** 本期不做MinIO对象存储和Milvus向量数据库（使用本地文件存储）

### 3. 功能需求（概要）

#### 3.1 核心能力清单

**P0 核心能力（必须实现）**：

- **需求文档解析**：自动解析DOCX格式需求文档，抽取"需求内容说明"章节
- **系统识别**：AI识别需求涉及的所有系统，输出`selected/maybe/questions`分层结果
- **功能点拆分**：按系统维度拆分功能点，支持类型化模板
- **区间估算**：输出人天区间+置信度+假设+风险因子
- **多专家评估**：3位专家独立盲评，支持最多3轮
- **偏离度分析**：计算专家均值与AI预估的偏离度，触发下一轮评估
- **报告生成**：生成Excel评估报告，支持版本管理
- **证据库（A层）**：DOCX/PPTX/PDF材料导入与检索
- **证据等级（E0~E3）**：计算证据等级并实现严肃模式阻断
- **ESB索引（I层）**：集成点明细导入/检索/统计

**P1 重要能力（建议实现）**：

- **系统画像（B层）**：业务能力边界结构化摘要，支持证据回链
- **代码扫描（C层）**：Spring Boot/Java能力目录提取
- **AI效果报告**：统计系统识别准确率、功能遗漏/夸大率等指标
- **COSMIC功能点分析**：基于数据移动的功能点分析

**P2 辅助能力（可选实现）**：

- **通知中心**：站内通知，支持未读数量角标
- **个人中心**：个人信息管理、密码修改、活动记录
- **配置管理**：系统清单、COSMIC规则、用户管理

#### 3.2 关键流程链接

```
需求文档上传 → 文档解析 → 系统识别（selected/maybe/questions）
    ↓
证据检索（A层）+ 画像注入（B层）+ ESB摘要（I层）+ 能力目录（C层）
    ↓
功能点拆分（按系统、模板化、带证据标签）
    ↓
区间估算（min/max + confidence + assumptions + key_factors）
    ↓
项目经理复核/修正 → 提交管理员 → 分配专家
    ↓
专家评估（3位独立盲评）→ 偏离度分析 → 触发下一轮（如需）
    ↓
报告生成（Excel，版本管理）→ 下载/归档
```

### 4. 非功能需求（概要）

#### 4.1 安全

- JWT身份认证，token过期时间可配置
- 基于角色的访问控制（RBAC）
- 证据预览权限控制：专家仅可查看其参与任务关联证据
- 管理写接口二级保护（X-API-Key或管理员token）
- 上传文件大小与类型限制
- 路径清洗避免目录穿越

#### 4.2 性能与容量

- 单机部署，支持中小规模使用
- 证据块数 < 10万 时可接受
- 本地向量库全量扫描，数据量增大建议切Milvus
- 长任务后台异步处理

#### 4.3 可用性与可靠性

- 文件写入：文件锁 + 原子写
- 模型不可用时允许进入人工编辑链路
- 证据预览渲染失败时降级为抽取文本预览
- 健康检查：`GET /api/v1/health`

#### 4.4 可观测性

- 后端日志：stdout + `logs/app.log`
- AI效果快照：命中率、相似度等
- 审计日志：规则变更、任务内覆盖修正、关键操作留痕
- 活动日志：个人行为记录

## 附录

### A. 术语表

| 术语 | 说明 |
|---|---|
| 系统台账/系统清单 | 企业已有的系统列表（名称为主，模块/功能清单缺失或不准） |
| 系统标识（system_id） | 系统简称/编码（如 `SOFP`），用于跨数据源对齐 |
| 证据库（A层） | 非结构化材料（DOCX/PPTX/PDF）切块后的可检索证据集合 |
| 系统画像（B层） | 围绕"业务能力边界"的结构化摘要，可人工确认 |
| 能力目录/功能清单（事实层） | 由代码扫描产出，用于召回与校验 |
| 证据等级（evidence_level） | 系统/功能点在本次任务中可用证据的综合等级（E0~E3） |
| selected/maybe/questions | 系统识别分层输出：确定系统/不确定系统/最小澄清问题集 |
| 严肃模式（阻断） | 证据不足或冲突时禁止输出"可直接用于采购"的确定结论 |
| 未接入系统（un-onboarded） | 缺少代码与材料等关键证据的系统 |

### B. 证据等级定义

| 等级 | 说明 |
|---|---|
| **E3（高）** | 具备能力目录（代码扫描）+（证据命中 或 已发布画像） |
| **E2（中）** | 具备证据命中 或 已发布画像，并且具备ESB或能力目录之一作为补证据 |
| **E1（低）** | 仅具备ESB，或具备证据但无其他补证据 |
| **E0（无）** | 除系统名称外无任何可用证据 |

### C. 默认参数

| 参数 | 默认值 | 说明 |
|---|---|---|
| 严肃模式 | 开启 | 默认启用阻断规则 |
| A层证据检索 Top-K | 5 | 用于系统识别/拆分注入的证据块数量 |
| A层证据检索阈值 | 0.60 | 低于阈值视为"命中不足" |
| ESB检索 Top-K | 8 | 用于集成点摘要/澄清问题候选 |
| ESB检索阈值 | 0.55 | 场景/服务名更短，阈值略放宽 |
| questions 数量 | 3~8 | 超出则按优先级截断（Top-N） |
| assumptions 阈值 | 4 | `assumptions_count >= 4` 触发采购风险提示/低置信度 |
| selected_systems 最大数 | 10 | 防止输出过长影响审阅效率 |
| maybe_systems 最大数 | 8 | 超出则保留更高证据/更高召回价值者 |
| 偏离度阈值 | 20% | 超过触发下一轮评估 |
| 最大评估轮次 | 3 | 多轮盲评上限 |

### D. 质量属性优先级

1. **正确性**：不扩范围；证据不足触发阻断；权限边界不可突破
2. **可追溯性**：输出带证据引用；覆盖修正可回放
3. **可用性**：部署后可完成登录与主流程；健康检查可用
4. **可维护性**：模块边界清晰；可在不影响主链路的情况下迭代证据/画像/ESB/扫描
5. **性能**：中小规模单机可用；可通过Milvus/异步化扩展
