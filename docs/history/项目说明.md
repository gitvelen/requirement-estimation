# 业务需求工作量评估系统 - 项目说明文档

## 一、项目背景

### 1.1 项目概述
业务需求工作量评估系统是一个基于Agent智能体和AI大模型的智能化需求评估工具，旨在解决传统工作量评估过程中存在的痛点：
- 人工拆分效率低、耗时长
- 评估标准不统一、偏差大
- 缺乏科学的方法论支撑
- 评估结果难以追溯和复用

### 1.2 核心价值
- ✅ **自动化**：自动解析需求文档、识别系统、拆分功能点
- ✅ **智能化**：基于AI大模型（阿里云通义千问）的智能分析
- ✅ **标准化**：统一的功能点拆分标准和COSMIC全功能点分析方法
- ✅ **可追溯**：完整的评估过程记录和结果输出
- ✅ **可集成**：提供RESTful API，易于集成到DevOps系统
- ✅ **人机协作**：支持人工修正AI评估结果，实现数据闭环

### 1.3 应用场景
- 需求评审阶段的工作量预估
- 项目投标时的工作量评估
- 项目管理中的资源规划
- DevOps系统的需求评估模块
- 软件开发过程的成本控制

系统支持人工对AI评估结果进行修正，实现以下能力：

- 添加新功能点
- 修改现有功能点（业务描述、复杂度、人天等）
- 删除不合理功能点
- 记录所有修改历史
- 确认后重新生成最终报告
- 修正数据用于优化后续AI评估（数据闭环）

## 四、功能点拆分规则

### 4.1 拆分原则

1. **系统维度优先**：按涉及系统进行一级拆分
2. **功能模块为辅**：系统内按功能模块进行二级拆分
3. **粒度适中**：单个功能点0.5-5人天
4. **独立可估算**：每个功能点可独立估算工作量
5. **依赖清晰**：明确标注功能点间的依赖关系

### 4.2 拆分方法

根据需求特征选择拆分方法：

| 需求类型 | 特征描述 | 推荐方法 |
|---------|---------|---------|
| 数据处理型 | 单角色、无流程、功能独立 | 按业务功能模块拆分 |
| 流程驱动型 | 多角色、强流程 | 按用户角色+场景拆分 |
| 界面优化型 | 单角色、无流程、独立优化点 | 按业务功能模块拆分 |
| 全新产品型 | 需求模糊、需快速验证 | 按MVP阶段拆分 |

### 4.3 复杂度评估标准

**高复杂度**：
- 涉及多系统集成
- 复杂算法或业务逻辑
- 核心业务功能
- 高风险或新技术

**中复杂度**：
- 标准业务逻辑
- 一定复杂度
- 常规功能

**低复杂度**：
- 简单增删改查
- 配置管理
- 查询展示

## 五、Delphi专家评估法

### 5.1 方法说明

Delphi法是一种结构化的专家预测方法，通过多轮匿名调查和反馈，使专家意见逐步收敛。

### 5.2 实施步骤

1. **第一轮**：5位专家独立估算每个功能点的工作量
2. **反馈**：展示统计结果（最大值、最小值、平均值、标准差）
3. **第二轮**：专家参考反馈，调整自己的估算
4. **第三轮**：重复反馈-调整过程（通常2-3轮）
5. **收敛**：取加权平均值作为最终估算

### 5.3 专家权重配置

系统支持配置不同专家的权重：
- 核心专家（专家1-3）：权重1.0
- 普通专家（专家4-5）：权重0.8
- 权重根据专家经验和历史准确度确定

## 六、COSMIC功能点分析

### 6.1 COSMIC方法简介

COSMIC（Common Software Measurement International Consortium）是一种国际标准的软件功能规模度量方法。

### 6.2 数据移动类型

COSMIC将软件功能分解为4种数据移动：

| 数据移动 | 说明 | 权重 |
|---------|------|------|
| E (Entry) | 输入数据移动 | 2 |
| X (Exit) | 输出数据移动 | 1 |
| R (Read) | 读取数据移动 | 2 |
| W (Write) | 写入数据移动 | 2 |

### 6.3 功能点计算

```
COSMIC功能点 = Σ(各数据移动权重)
例如：一个功能包含 2E + 1R + 1W = 2×2 + 1×2 + 1×2 = 8 CFPu
```

### 6.4 配置方式

系统支持在前端配置页面配置COSMIC规则，灵活调整各数据移动的权重。

## 九、知识库功能

### 9.1 功能说明

系统集成了基于向量数据库（Milvus）的知识库功能，可以：
- 上传系统文档到知识库
- 自动向量化文档内容
- 根据需求检索相似系统案例
- 将检索结果注入到Agent提示词中

### 9.2 技术实现

- **向量模型**: 阿里云 text-embedding-v2 (1024维)
- **向量数据库**: Milvus 2.3+
- **相似度算法**: 余弦相似度
- **检索策略**: TopK = 5, 相似度阈值 = 0.6

### 9.3 使用场景

当知识库启用时，系统识别Agent会：
1. 接收需求内容
2. 调用知识库检索相似系统
3. 将检索结果作为参考信息
4. 结合需求内容进行系统识别

## 十一、常见问题

### Q1: 支持哪些文档格式？
A: 目前仅支持.docx格式。后续可扩展支持.pdf、.txt等格式。

### Q2: 如何提高评估准确性？
A:
1. 提供高质量的需求文档
2. 启用知识库功能，上传历史系统文档
3. 使用人机协作功能修正AI评估结果
4. 定期优化提示词模板
5. 根据实际数据调整专家权重

### Q3: 能否自定义拆分规则？
A: 可以。通过修改`backend/prompts/prompt_templates.py`中的提示词模板来自定义拆分规则。

### Q4: 如何启用知识库功能？
A:
1. 部署 Milvus 向量数据库（见 docs/MILVUS_DEPLOYMENT.md）
2. 配置环境变量 `KNOWLEDGE_ENABLED=true`
3. 配置 Milvus 连接信息
4. 上传系统文档到知识库

### Q5: 人机协作修正会影响AI学习吗？
A: 会。修正数据会持久化存储，未来版本将用于优化AI模型（数据闭环）。
