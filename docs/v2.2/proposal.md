# 需求评估系统 v2.2 综合优化提案

## 一句话总结
为项目经理、专家、管理员精简页面冗余、重构导航与看板、强化编辑管控和自动化备注，提升日常评估工作的操作效率与数据可读性。

## 背景与现状
1. **页面信息冗余**：多个页面左上角 PageHeader 标题与侧边栏菜单重复（如"任务管理"页显示"进行中的任务"），增加视觉噪音。
2. **效能看板结构臃肿**：当前 5 个顶级 Tab（总览/排行榜/AI表现/系统影响/流程健康），"AI表现"使用率低，排行榜与报表混在一起不易聚焦。
3. **任务管理切换不直观**：进行中/已完成通过页面内 Tab 切换，用户需先进入页面再切换，不如侧边栏子菜单直观。
4. **任务详情页信息分散**：任务详情、专家评估进度分散在不同区域，报告版本列表实用性低。
5. **项目经理编辑缺乏管控**：预估人天可随意修改，实质性变更无需确认即可提交，存在误操作风险。
6. **系统画像知识导入结构不清**：代码扫描、ESB导入、知识导入三块并列，概念层次混乱。
7. **专家评估页 COSMIC 规则占用空间大**：右侧大卡片常驻，压缩了评估操作区域。
8. **备注字段依赖人工填写**：容易遗漏，格式不统一。

### 约束/前提
- 技术栈不变：React + Ant Design + FastAPI + 文件存储
- 角色体系不变：admin / manager / expert / viewer
- 不涉及数据库迁移或数据格式变更
- 需兼容现有数据（任务、评估记录等）

## 目标与成功指标
| 目标 | 基线（v2.1） | 目标（v2.2） | 统计口径 |
|------|-------------|-------------|---------|
| 页面冗余信息 | 8处冗余标题 | 0处 | 全站 PageHeader 审计 |
| 效能看板模块数 | 5个Tab混合 | 2个子菜单（排行榜+多维报表） | 菜单结构 |
| 任务切换步骤 | 进入页面→切Tab（2步） | 直接点子菜单（1步） | 用户操作路径 |
| PM误操作风险 | 无管控 | 实质性修改需确认+AI重评估 | 编辑提交流程 |
| 备注填写率 | 依赖人工 | AI自动生成，100%覆盖 | 备注字段非空率 |

## 目标用户与典型场景
- **项目经理（manager）**：日常创建/编辑任务、查看评估结果、导入系统画像
- **专家（expert）**：评估功能点、参考COSMIC规则
- **管理员（admin）**：查看效能数据、管理系统配置
- **只读用户（viewer）**：查看看板和任务列表

### TOP 场景
1. 项目经理编辑功能点时，新增功能点预估人天留空（由专家评估填写），修改核心字段后系统提示"保存并确认"再提交
2. 管理员查看效能看板，通过排行榜子菜单直接进入排名页面，每个排名Tab底部有计算逻辑说明
3. 专家评估时点击顶部"?"图标查看COSMIC规则，不占用主操作区域

## 方案概述

### 变更一：全站页面精简（删除冗余标题）

**要做什么**：删除 8 个页面的冗余 PageHeader 标题，简化 2 个页面标题。

**需删除标题的页面**：
| 页面 | 当前冗余标题 | 处理方式 |
|------|------------|---------|
| TaskListPage（进行中） | "进行中的任务" | 删除 PageHeader |
| TaskListPage（已完成） | "已完成的任务" | 删除 PageHeader |
| SystemListConfigPage | "系统清单配置" | 删除 PageHeader |
| CosmicConfigPage | "估算规则配置（COSMIC）" | 删除 PageHeader |
| UserManagementPage | "用户管理" | 删除 PageHeader |
| KnowledgePage | "知识库管理" | 删除 PageHeader |
| SystemProfileImportPage | "系统画像 - 知识导入" | 删除 PageHeader |
| SystemProfileBoardPage | "系统画像 - 信息看板" | 删除 PageHeader |

**需简化标题的页面**：
| 页面 | 当前标题 | 简化为 |
|------|---------|--------|
| EfficiencyDashboardPage | "效能看板 - XXX" | 仅保留子页面名称 |
| NotificationPage | "消息通知" | 删除（菜单已有） |

**为什么**：侧边栏菜单已标识当前位置，PageHeader 重复显示浪费空间。

### 变更二：效能看板重构

**要做什么**：将当前 5 个顶级 Tab 重构为 2 个侧边栏子菜单，删除"AI表现"模块。

**菜单结构**：
```
效能看板
  ├── 排行榜      /dashboard/rankings
  └── 多维报表    /dashboard/reports
```

**排行榜页面**（3 个 Tab 切换）：
| Tab | 内容 | 计算逻辑说明（右下角） |
|-----|------|---------------------|
| 评估效率排行 | 专家平均评估耗时排名 | 计算逻辑：评估效率 = 已完成评估数 / 累计评估耗时（天），仅统计近90天数据 |
| 任务提交排行 | 项目经理提交任务数排名 | 计算逻辑：统计近90天内各项目经理提交的任务总数，按数量降序排列 |
| 系统活跃排行 | 各系统评估任务数排名 | 计算逻辑：统计近90天内各系统关联的评估任务数，按数量降序排列 |

**多维报表页面**（3 张卡片）：
| 卡片 | 内容 |
|------|------|
| 总览统计 | 原"总览"Tab 内容：任务总数、完成率、平均耗时等汇总指标；并保留 v2.1 管理驱动指标中的：修正率、命中率、画像贡献 |
| 系统影响分析 | 原"系统影响"Tab 内容：按系统维度的评估数据分析 |
| 流程健康度 | 原"流程健康"Tab 内容：流程各环节耗时、瓶颈分析；并保留 v2.1 管理驱动指标中的：评估周期、偏差监控、学习趋势 |

**删除**："AI表现"模块入口整体下线（含路由、菜单项、组件）。其中 FUNC-022（AI效果报告，原 `/dashboard?page=ai`）随模块一并下线；相关 API/数据接口在实现阶段清理。

**为什么**：排行榜是高频查看场景，独立为子菜单减少层级；多维报表整合低频分析类内容；AI表现使用率低，删除减少维护成本。

**替代方案（对比）**：可选方案是保留 5 Tab 结构仅做排序/精简（或仅把排行榜抽为子菜单）。考虑到可发现性与信息架构一致性，本次选择用侧边栏子菜单承载高频入口，降低页面内切换成本。

**兼容重定向**（旧 URL 处理）：
- `/dashboard?page=rankings` → `/dashboard/rankings`
- `/dashboard?page=overview|system|flow` → `/dashboard/reports`
- `/dashboard?page=ai` → `/dashboard/reports`（提示：AI效果报告已下线）

### 变更三：任务管理子菜单化

**要做什么**：将"进行中/已完成"从页面内 Tab 切换改为侧边栏子菜单。

**菜单结构**：
```
任务管理
  ├── 进行中      /tasks/ongoing
  └── 已完成      /tasks/completed
```

**路由变更**：
| 原路由 | 新路由 | 说明 |
|--------|--------|------|
| /tasks（Tab=ongoing） | /tasks/ongoing | 进行中任务列表 |
| /tasks（Tab=completed） | /tasks/completed | 已完成任务列表 |
| /tasks?tab=ongoing | /tasks/ongoing | 兼容旧 query parameter |
| /tasks?tab=completed | /tasks/completed | 兼容旧 query parameter |
| /tasks | /tasks/ongoing（重定向） | 默认进入进行中 |
| /tasks/my-tasks | 保留 | manager 专属视图 |
| /tasks/my-evaluations | 保留 | expert 专属视图 |

**为什么**：子菜单一步到位，减少页面内切换操作；与效能看板子菜单风格统一。

### 变更四：任务详情页优化

**要做什么**：合并信息、删除低价值区域。

**调整内容**：
| 区域 | 当前状态 | 调整 |
|------|---------|------|
| 摘要卡片 | 仅显示基本信息 | 合入"任务详情"和"专家评估进度"关键字段 |
| 任务详情区域 | 独立区域 | 关键字段合入摘要，区域删除 |
| 专家评估进度 | 独立区域 | 关键字段合入摘要，区域删除 |
| 报告版本列表 | 独立区域 | 删除列表区域，但保留历史报告下载入口（见下） |
| 分析区域 | 功能点列表+图表 | 保留不变 |

**合入摘要的字段**：任务状态、创建时间、提交人、系统名称、功能点数量、专家评估状态（已评/待评/总数）、当前评估轮次。

**历史报告访问方式（替代路径）**：在摘要卡片提供"下载报告"按钮：默认下载最新报告；下拉菜单列出历史版本（按评估轮次/生成时间）供下载，满足“可下载历史版本”的诉求。

**为什么**：减少页面滚动，核心信息一屏可见；报告版本列表使用率极低。

### 变更五：项目经理编辑功能点管控

**要做什么**：限制预估人天编辑权限，实质性修改增加确认流程和 AI 重评估。

**预估人天字段**：
- 前端设为只读（disabled），项目经理不可编辑
- 新增功能点时，预估人天留空（不再默认填 1）
- 仅专家评估和 AI 评估可写入预估人天

**实质性修改定义**：
| 操作 | 是否实质性修改 | 触发动作 |
|------|--------------|---------|
| 新增功能点 | 是 | 保存并确认 → AI 重评估 |
| 删除功能点 | 是 | 保存并确认 → AI 重评估 |
| 修改功能点核心字段（名称/描述/所属系统） | 是 | 保存并确认 → AI 重评估 |
| 新增或重命名系统 | 是 | 保存并确认 → AI 重评估 |
| 删除系统（Tab） | 是 | 保存并确认 → AI 重评估 |
| 修改非核心字段（排序/备注） | 否 | 正常保存 |

**确认流程**：实质性修改 → 点击"保存并确认" → 弹窗提示"本次修改将触发 AI 重新评估，是否确认？" → 确认后保存并自动触发 AI 重评估。

**为什么**：防止项目经理随意修改预估人天影响评估结果；实质性变更需要 AI 重新评估以保证数据一致性。

**替代方案（对比）**：可选方案是保留可编辑但增加变更高亮/回滚，或仅对“删除/改核心字段”触发确认。考虑到误操作成本与一致性，本次统一将“增删/改核心/增改删系统”纳入实质性修改并强制确认。

### 变更六：系统画像知识导入重设计

**要做什么**：保留"知识导入"菜单，内部用 Tab 切换"代码扫描"和"文档导入"两个功能区。

**菜单结构**（不变）：
```
系统画像
  ├── 知识导入      /system-profiles/import
  └── 信息展示      /system-profiles/board（原"信息看板"改名）
```

**知识导入页面内部结构**：
```
┌─────────────────────────────────────────┐
│  [代码扫描]  [文档导入]     ← Tab 切换   │
├─────────────────────────────────────────┤
│                                         │
│  Tab 对应内容区                          │
│                                         │
└─────────────────────────────────────────┘
```

**代码扫描 Tab**：保留现有代码扫描功能，选择系统 → 触发扫描 → 展示结果。

**文档导入 Tab**：
- 合并原"ESB导入"和"知识导入"为统一的文档导入界面
- 支持"+"按钮添加文档，弹出下拉选择文档类型：
  | 文档类型 | 说明 |
  |---------|------|
  | 需求文档 | 系统需求规格说明 |
  | 设计文档 | 系统设计文档 |
  | 技术方案 | 技术方案设计文档 |
  | 历史评估报告 | 历史评估结果 |
  | ESB接口文档 | ESB 接口定义（原 ESB 导入功能） |
- 选择类型后上传文件，系统解析并存储
- 后端接口策略（方向性）：不新建统一导入 API，前端按文档类型分发到现有接口
  - ESB接口文档：沿用 `API-003 POST /api/v1/esb/imports`（必要时保留 `mapping_json`）
  - 其余 4 类文档：沿用 `API-011 POST /api/v1/knowledge/imports`（在请求中携带 `doc_type` 标记）
- ESB 类型特有功能（FUNC-018）处置：**保留**，避免能力回退
  - 保留能力：列名自动识别/映射预览、状态过滤（正常/废弃）、scope=provider/consumer/both 切换、有效/废弃/去重接口数统计
  - 交互约束：选择"ESB接口文档"时展示 ESB 专属导入表单与导入后检索/统计面板；其它类型保持通用上传交互

**信息展示**（原"信息看板"）：仅改名，功能不变。

**为什么**：代码扫描和文档导入是两种不同的知识获取方式，Tab 切换清晰分类；ESB 导入本质是文档导入的一种，统一管理更合理。

### 变更七：专家评估页 COSMIC 规则优化

**要做什么**：将右侧大卡片"COSMIC简明规则"改为顶部"?"图标，点击展开详情。

**当前**：右侧常驻 Card，宽度 520px，含 Collapse 折叠面板。
**改为**：页面顶部右侧放置"?"图标（Tooltip 或 Popover），点击弹出 COSMIC 规则详情。

```
┌─────────────────────────────────────────────┐
│  专家评估 - XXX任务          [?] ← 点击弹出  │
├─────────────────────────────────────────────┤
│                                             │
│  评估操作区（全宽）                           │
│                                             │
└─────────────────────────────────────────────┘
```

**为什么**：COSMIC 规则是参考信息，不需要常驻显示；释放横向空间给评估操作区。

### 变更八：备注字段 AI 自动生成

**要做什么**：备注字段改为 AI 自动生成、前端只读展示。

**生成时机**：
| 触发场景 | 生成内容示例 |
|---------|------------|
| 项目经理编辑功能点 | "PM 张三：新增功能点2个，修改功能点1个（名称变更）" |
| 专家完成评估 | "专家李四：完成评估，调整3个功能点人天" |
| AI 重评估完成 | "AI重评估：因功能点变更触发，更新5个功能点预估" |

**前端展示**：备注字段设为 readonly，显示 AI 生成的修改摘要，按时间倒序排列。

**历史备注数据处理**：保留既有人工备注不做清空/迁移；v2.2 起后端在同一备注字段中追加带时间戳的 AI 摘要（多行文本），前端仅做只读展示与倒序渲染，避免引入数据库迁移与结构化字段变更。

**为什么**：消除人工填写的遗漏和格式不一致问题；自动记录每次变更的关键信息，便于追溯。
