### RVW-001（P1）删除"AI表现"模块但未说明 FUNC-022 的处置方案
- 证据：proposal.md 变更二明确"删除：'AI表现'模块整体删除（含路由、组件、数据接口）"，但系统功能说明书中 FUNC-022（AI效果报告）的入口为"效能看板 → AI表现（`/dashboard?page=ai`）"（[系统功能说明书.md:1163](docs/系统功能说明书.md#L1163)）。proposal 未说明该功能是完全废弃、迁移到多维报表、还是降级处理。
- 风险：FUNC-022 涉及 SCN-022（AI效果报告），是 admin/manager/expert 三角色均使用的功能。若完全废弃需明确声明并纳入 Non-goals；若迁移需在 In Scope 中体现。当前状态下 Requirements 阶段无法判断该功能的去向。
- 建议修改：在 proposal.md 变更二中补充 FUNC-022 的处置决策（废弃/迁移/降级），并同步更新 In Scope 或 Non-goals。
- 验证方式：检查 proposal.md 变更二是否包含 FUNC-022/AI效果报告的明确处置说明。

### RVW-002（P1）删除"报告版本列表"但未说明历史报告的替代访问路径
- 证据：proposal.md 变更四"报告版本列表 → 整体删除"（[proposal.md:139](docs/v2.2/proposal.md#L139)），但 SCN-013（报告生成与版本）要求"支持下载历史版本"（[系统功能说明书.md:385-386](docs/系统功能说明书.md#L385-L386)），FUNC-013 验收标准要求"report_versions 新增一条且可下载 PDF"。
- 风险：删除报告版本列表后，用户失去在任务详情页访问历史报告版本的入口。如果没有替代路径，等于功能回退。
- 建议修改：明确历史报告版本的替代访问方式（如：合入摘要卡片的下载按钮、独立下载入口、或确认该功能确实废弃）。
- 验证方式：检查 proposal.md 变更四是否包含报告版本访问的替代方案说明。

### RVW-003（P1）缺少替代方案讨论
- 证据：review_template.md Proposal 清单要求"替代方案：至少比较 1 个替代方案（如适用）"。proposal.md 无"替代方案"章节，8 项变更均未讨论替代方案。
- 风险：部分变更存在合理替代（如：效能看板可保留 Tab 结构而非拆子菜单；COSMIC 规则可用侧边抽屉而非弹窗），缺少对比可能导致后续返工。
- 建议修改：至少对变更二（效能看板重构）和变更五（编辑管控）补充替代方案对比（1-2 句即可，说明为何选择当前方案）。
- 验证方式：检查 proposal.md 是否包含替代方案章节或在关键变更中内联对比。

### RVW-004（P1）变更五"实质性修改定义"遗漏"删除系统"场景
- 证据：proposal.md 变更五的实质性修改定义表（[proposal.md:156-163](docs/v2.2/proposal.md#L156-L163)）列出了"新增功能点/删除功能点/修改核心字段/新增或重命名系统"，但缺少"删除系统（Tab）"。
- 风险：删除系统 Tab 会移除该系统下所有功能点，影响面大于单个功能点删除，理应触发 AI 重评估。遗漏可能导致 Requirements 阶段定义不完整。
- 建议修改：在实质性修改定义表中补充"删除系统（Tab）→ 是 → 保存并确认 → AI 重评估"。
- 验证方式：检查实质性修改定义表是否包含"删除系统"行。

### RVW-005（P2）v2.1 管理驱动指标在重构后的归属未明确
- 证据：系统功能说明书 v2.1 增量能力提到"效能看板新增管理驱动指标组件（修正率/命中率/评估周期/画像贡献/偏差监控/学习趋势）"（[系统功能说明书.md:95](docs/系统功能说明书.md#L95)）。proposal.md 变更二的多维报表 3 张卡片（总览统计/系统影响分析/流程健康度）未明确提及这些指标的归属。
- 风险：管理驱动指标可能在重构中被遗漏，导致 v2.1 能力回退。
- 建议修改：在变更二多维报表描述中说明管理驱动指标归入哪张卡片（建议归入"总览统计"或"流程健康度"）。
- 验证方式：检查多维报表卡片描述是否覆盖管理驱动指标。

- 风险：低。历史数据处理可在 Requirements/Design 阶段细化，但提案阶段应有方向性说明。

### RVW-009（P1）变更六 ESB 导入合并到"文档导入"后，FUNC-018 特有功能处理方式未说明
- 证据：proposal.md 变更六将 ESB 导入合并为"文档导入"的一种文档类型（"ESB接口文档"，[proposal.md:208](docs/v2.2/proposal.md#L208)）。但系统功能说明书 FUNC-018（ESB索引 I层）具备独立的特有功能：列名自动识别、状态过滤（正常使用/废弃使用）、scope=provider/consumer/both 切换、有效/废弃条目统计与去重接口数（[系统功能说明书.md:1087-1104](docs/系统功能说明书.md#L1087-L1104)）。接口文档中 ESB 导入为独立 API（`API-003 POST /esb/imports`，[接口文档.md:88](docs/接口文档.md#L88)）。proposal 仅说"ESB 导入本质是文档导入的一种，统一管理更合理"，未说明合并后这些特有功能是保留、简化还是废弃。
- 风险：ESB 导入不是简单的文件上传——它有专用的列识别解析、状态过滤检索、scope 切换和统计能力。如果合并后这些功能丢失，等于 FUNC-018 能力回退。如果保留，则"统一文档导入"的前端需要对 ESB 类型做特殊处理，proposal 应说明这一点。
- 建议修改：在变更六中补充 ESB 导入合并后的功能处理策略：(1) ESB 特有的检索/统计/scope 功能是否保留；(2) 前端"文档导入"对 ESB 类型是否有特殊交互（如列映射预览）；(3) 后端 API-003 是否保留还是合入新统一接口。
- 验证方式：检查变更六是否包含 ESB 特有功能（列识别/状态过滤/scope/统计）的处置说明。

### RVW-010（P2）5 种文档类型与现有后端 API 的映射关系未明确
- 证据：proposal.md 变更六新增 5 种文档类型（[proposal.md:202-209](docs/v2.2/proposal.md#L202-L209)），但现有后端有两个独立导入 API：`API-003 POST /esb/imports`（ESB）和 `API-011 POST /knowledge/imports`（知识导入）。proposal 未说明前端"文档导入"是按文档类型路由到不同现有 API，还是需要新建统一导入 API。
- 风险：低。可在 Design 阶段细化，但 proposal 应给出方向性说明以便评估后端工作量。
- 建议修改：补充一句后端策略方向（如"前端按文档类型分发到现有 API，不新建统一接口"或"新建统一导入接口，内部按类型分发"）。
- 验证方式：检查变更六或 In Scope 是否包含后端 API 策略说明。

### RVW-011（P2）路由重定向策略覆盖不完整
- 证据：proposal.md 变更三路由变更表（[proposal.md:121-127](docs/v2.2/proposal.md#L121-L127)）仅列出 `/tasks → /tasks/ongoing` 重定向。但存在其他需要处理的旧 URL：(1) `/tasks?tab=completed`（旧 query parameter 格式）；(2) `/dashboard?page=ai`（FUNC-022 旧入口，见[系统功能说明书.md:1163](docs/系统功能说明书.md#L1163)）；(3) 通知中心（FUNC-023）历史通知可能包含指向旧 URL 的链接。
- 风险：低。旧 URL 访问量不大，但通知中的旧链接可能导致用户困惑。
- 建议修改：在风险表或变更三中补充需要重定向/处理的旧 URL 完整清单（至少覆盖 `/dashboard?page=ai` 和带 query parameter 的旧格式）。
- 验证方式：检查路由变更表或风险应对是否覆盖 `/dashboard?page=ai` 和旧 query parameter 格式。

### RVW-012（P2）FUNC-022 下线后主文档同步更新需求未提及

- 验证方式：检查 In Scope 或风险表是否包含主文档同步更新说明。
