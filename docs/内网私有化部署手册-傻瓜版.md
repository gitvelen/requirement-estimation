# 内网私有化部署手册（傻瓜版）

适用场景：前后端分离、无法访问外网、使用现有 `*internal*` 脚本一键部署。  
目标环境：
- 后端服务器：`10.62.22.121:443`
- 前端服务器：`10.62.16.251:8000`
- 后端基础镜像：`uv-mineru-opencv:0.1.0`
- PyPI 私服：`http://pypi.ppr.dev:8081/repository/shrbank-pypi-group/simple`
- 大模型 API Base：`http://10.73.254.200:30000/v1`（无真实 API Key 场景）

## 1. 你需要准备什么

1. 一份完整项目代码（当前仓库）。
2. 前端构建产物（满足其一即可）：
   - 本地先执行 `cd frontend && npm run build`，得到 `frontend/build/index.html` 后再打包上传（推荐）
   - 或单独上传 `frontend-build.tar.gz`
3. 两台服务器都安装好 `docker` 和 `docker-compose`。
4. 两台服务器都已离线导入镜像：
```bash
# 后端机 10.62.22.121
docker load -i uv-mineru-opencv_0.1.0.tar

# 前端机 10.62.16.251
docker load -i nginx_latest.tar
```

## 2. 本次使用到的文件

- 后端一键脚本：`deploy-backend-internal.sh`
- 前端一键脚本：`deploy-frontend-internal.sh`
- 后端 compose：`docker-compose.backend.internal.yml`
- 前端 compose：`docker-compose.frontend.internal.yml`
- 后端内网配置：`.env.backend.internal`
- 前端内网 nginx：`frontend/nginx.internal.conf`

> 注意：仓库 `.gitignore` 默认忽略 `frontend/build/`，直接从 GitHub 下载源码包通常不包含构建产物。

## 3. 上传文件（在你本地执行）

先打包项目（示例）：
```bash
cd /path/to/requirement-estimation-system
tar czf requirement-estimation-internal.tar.gz .
```

上传到两台服务器：
```bash
scp requirement-estimation-internal.tar.gz root@10.62.22.121:/home/admin/
scp requirement-estimation-internal.tar.gz root@10.62.16.251:/home/admin/
```

如果你不打包 `frontend/build`，再额外上传：
```bash
# 在本地项目目录执行（确保压缩包根目录就是 index.html/static）
cd frontend
tar czf /tmp/frontend-build.tar.gz -C build .

# 上传到前端机脚本默认位置
scp /tmp/frontend-build.tar.gz root@10.62.16.251:/home/admin/frontend-build.tar.gz

# 或保留你自己的文件名
scp frontend-build.tar.gz root@10.62.16.251:/home/admin/
```

## 4. 部署后端（10.62.22.121）

登录后端服务器：
```bash
ssh root@10.62.22.121
```

按顺序执行：
```bash
cd /home/admin
mkdir -p requirement-estimation
tar xzf requirement-estimation-internal.tar.gz -C requirement-estimation
cd requirement-estimation

# 可选：确认关键配置
grep -E "^(DEBUG|DASHSCOPE_API_BASE|LLM_MODEL|DASHSCOPE_API_KEY|PORT|WORKERS)=" .env.backend.internal

chmod +x deploy-backend-internal.sh
./deploy-backend-internal.sh
```

部署成功校验：
```bash
docker ps | grep requirement-backend
curl -fsS http://127.0.0.1:443/api/v1/health
```

## 5. 部署前端（10.62.16.251）

登录前端服务器：
```bash
ssh root@10.62.16.251
```

按顺序执行：
```bash
cd /home/admin
mkdir -p requirement-estimation
tar xzf requirement-estimation-internal.tar.gz -C requirement-estimation
cd requirement-estimation

# 方式A（推荐）：你本地构建后一起打包上传的 frontend/build
ls -lh /home/admin/requirement-estimation/frontend/build/index.html

# 方式B（可选）：如果没有 frontend/build，则准备 frontend-build.tar.gz
ls -lh /home/admin/frontend-build.tar.gz

chmod +x deploy-frontend-internal.sh
./deploy-frontend-internal.sh
```

部署成功校验：
```bash
docker ps | grep requirement-frontend
curl -I http://127.0.0.1:8000
```

## 6. 联调验证（必须做）

在任意可访问内网的机器上执行：
```bash
curl -fsS http://10.62.22.121:443/api/v1/health
curl -I http://10.62.16.251:8000
```

浏览器访问：
- 前端：`http://10.62.16.251:8000`

## 7. 常用运维命令

后端服务器：
```bash
cd /home/admin/requirement-estimation
docker-compose -f docker-compose.backend.internal.yml ps
docker-compose -f docker-compose.backend.internal.yml logs -f
docker-compose -f docker-compose.backend.internal.yml restart
docker-compose -f docker-compose.backend.internal.yml down
```

前端服务器：
```bash
cd /home/admin/requirement-estimation
docker-compose -f docker-compose.frontend.internal.yml ps
docker-compose -f docker-compose.frontend.internal.yml logs -f
docker-compose -f docker-compose.frontend.internal.yml restart
docker-compose -f docker-compose.frontend.internal.yml down
```

## 8. 失败排查（最短路径）

1. 容器没起来：先看容器日志。
```bash
docker logs requirement-backend
docker logs requirement-frontend
```

2. 前端页面打开但接口报错：检查前端 nginx 代理目标是否是 `10.62.22.121:443`。
```bash
grep -n "proxy_pass" /home/admin/requirement-estimation/frontend/nginx.internal.conf
```

3. 后端启动即失败：优先检查 `.env.backend` 是否由 `.env.backend.internal` 同步生成。
```bash
ls -lh /home/admin/requirement-estimation/.env.backend*
grep -E "^(DASHSCOPE_API_BASE|LLM_MODEL|DASHSCOPE_API_KEY)=" /home/admin/requirement-estimation/.env.backend
```

4. 离线镜像缺失：补 `docker load` 后重跑脚本。
