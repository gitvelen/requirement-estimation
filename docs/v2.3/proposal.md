## 一句话总结
- 在不新增独立可视化页面的前提下，增强后端深度代码扫描能力并做最小前端接入，优先降低系统/功能点误识别率，同时保留一键回滚到 `v2.2` 的能力。

## 用户确认决策（2026-02-26）
- D1（核心价值）：优先 `准确性`，即识别与归属更准。
- D2（范围边界）：选择“后端能力 + 最小前端改动（仅在现有页面展示关键结果）”。
- D3（关键验收线）：相比 `v2.2`，误识别率必须明确下降。
- D4（硬约束）：发布前必须保留一键回滚到 `v2.2` 的能力。

## 背景与现状
- 当前代码扫描链路在“能力条目（`capability_item`）→ Agent 评估使用”上存在断路，导致扫描结果无法稳定转化为评估输入。
- 规则提取仍有正则方案残留，面对复杂代码结构时容易漏提取或误提取。
- 现有结果缺少调用链、服务依赖、数据流、复杂度和变更影响面等结构化信号，导致“为什么判定为该系统/该功能点”的可解释性不足，间接影响准确率。
- GitLab 源代码扫描能力需要覆盖全量与增量场景（Archive / Compare / Raw），以支撑真实项目迭代流程。

### 约束/前提
- 沿用现有技术栈与部署形态，不引入重型新基础设施。
- 以现有页面做最小前端接入，不新增完整图谱可视化页面。
- 上线策略必须支持功能级回退和版本级回退，并可一键恢复 `v2.2`。

## 目标与成功指标
| 目标 | 基线（v2.2） | 目标（v2.3） | 统计口径 |
|---|---|---|---|
| 误识别率（核心） | 以 v2.2 基线回归集测得值 `B` 为准 | `<= B * 0.7`（至少下降 30%） | 同一回归集、同一判定口径，发布前对比 |
| `capability_item` 链路可达率 | 0%（链路断开） | >= 95% | 全量回归（通过用例 / 总用例） |
| AST 解析覆盖率 | 0%（未接入 tree-sitter） | >= 95% | AST 成功解析文件 / 可解析文件 |
| 调用图覆盖率 | 0% | >= 85% | 入图方法数 / 可分析方法数 |
| 数据流覆盖率 | 0% | >= 80% | 已识别读写关系实体数 / 可分析实体数 |
| 复杂度覆盖率 | 0%（无统一复杂度产物） | >= 95% | 具备 CC+WMC 指标的方法数 / 可分析方法数 |
| GitLab 扫描通过性 | 0/3，任务成功率基线未达标 | 路径 3/3（Archive/Compare/Raw）且任务成功率 >= 99% | 每次全量回归 + 发布前一次 |

## 目标用户与典型场景
- 目标用户：
  - manager：查看 AI 系统识别与功能点归属结果，减少人工纠偏成本。
  - admin：关注扫描质量与稳定性，控制上线风险。
- TOP 场景（2-3 条）：
  - manager 提交需求后，系统基于更完整代码信号生成更准确的系统识别与功能点归属。
  - manager 在现有页面看到关键扫描证据摘要（非新页面），快速判断是否需要人工修正。
  - admin 在发布前验证 GitLab 三种扫描模式均可用，确保可在真实仓库迭代场景运行。

## 方案概述
- 要做什么：
  - 打通 `capability_item` 到 Agent 评估的输入链路。
  - 以 AST（tree-sitter）替代关键正则提取路径。
  - 新增调用链分析、服务依赖图、数据流分析、复杂度度量。
  - 新增变更影响面分析，输出系统/功能点/接口级受影响摘要供评估使用。
  - 集成 GitLab 仓库读取能力，支持 Archive / Compare / Raw。
  - 在现有页面做最小前端展示（关键指标/证据摘要），不新增独立可视化页面。
- 为什么要做：
  - 当前最大痛点是“识别不准”，其根因在于代码信号深度不足和链路断路。
  - 以最小前端改动接入，可在控制范围与交付风险的同时实现准确率增益。
- 预期效果：
  - 误识别率显著下降。
  - 结果可追溯性提升，便于 PM 快速决策是否修正。
  - GitLab 扫描能力满足增量迭代场景。

## 范围界定
### 包含（In Scope）
- 后端：
  - `capability_item -> Agent` 链路打通。
  - AST 解析接入与正则路径替换。
  - 调用链、服务依赖、数据流、复杂度分析能力落地。
  - 变更影响面分析能力落地（系统/功能点/接口三级摘要）。
  - GitLab Archive / Compare / Raw 三路径接入。
- 前端（最小改动）：
  - 在现有页面补充关键扫描结果摘要/状态，不新增菜单与独立图谱页。
- 文档与验证：
  - 更新 `requirements/design/plan/test_report/deployment` 对应章节。
  - 补齐回滚 runbook 与验证命令。

### 不包含（Non-goals）
- 新增完整调用图/依赖图/数据流可视化页面。
- 重构现有任务评估主链路 UI 信息架构。
- 引入新的重型基础设施或改变部署形态。

## 关键验收锚点（Proposal 阶段初步明确）

### 必须做到（Must Have）
- [ ] P-DO-01: 在统一回归口径下，`v2.3` 误识别率相对 `v2.2` 至少下降 30%。
- [ ] P-DO-02: `capability_item` 能稳定进入 Agent 评估链路并参与结果判定。
- [ ] P-DO-03: GitLab 三种读取模式（Archive/Compare/Raw）全部通过回归验证。
- [ ] P-DO-04: 前端仅做最小接入，不新增独立可视化页面。
- [ ] P-DO-05: 发布前验证功能级与版本级回退路径，且可一键回退到 `v2.2`。
- [ ] P-DO-06: 变更影响面分析可用，并可输出受影响系统/功能点/接口摘要参与判定。

### 绝对不能出现（Prohibitions）
- [ ] P-DONT-01: 误识别率无改善或改善不可证明。
- [ ] P-DONT-02: 新增能力无法回滚到 `v2.2` 稳定基线。
- [ ] P-DONT-03: 超范围新增完整图谱可视化页面。

### 成功指标（可量化）
- [ ] P-METRIC-01: 误识别率 `<= B * 0.7`（同口径对比 v2.2）。
- [ ] P-METRIC-02: `capability_item` 链路可达率 >= 95%。
- [ ] P-METRIC-03: AST 覆盖率 >= 95%，调用图覆盖率 >= 85%，数据流覆盖率 >= 80%，复杂度覆盖率 >= 95%。
- [ ] P-METRIC-04: GitLab 路径通过率 = 3/3，且任务成功率 >= 99%。

## 备选方案（可选）
| 方案 | 优点 | 缺点 | 结论 |
|---|---|---|---|
| A：仅修链路 + AST | 成本最低，落地快 | 准确率提升上限有限，可解释性不足 | 不选 |
| B：后端深度增强 + 最小前端接入 | 满足准确率目标，范围可控，风险可管 | 需要补齐回归与回滚验证 | **推荐** |
| C：后端增强 + 完整前端可视化页 | 可视化最完整 | 超出当前范围，工期和风险显著增加 | 不选 |

## 风险与依赖
- 风险：AST/图分析引入性能开销。  
  - 应对：先完成准确率目标，再通过阈值与采样控制性能。
- 风险：GitLab 凭据/网络限制导致不稳定。  
  - 应对：保留本地/压缩包路径兜底，明确错误码与降级行为。
- 风险：新分析信号导致误报。  
  - 应对：在 Requirements 阶段固化“信号权重与冲突处理规则”，并通过回归集校正。
- 外部依赖：GitLab 访问权限、测试仓库样本、回归数据集维护。
