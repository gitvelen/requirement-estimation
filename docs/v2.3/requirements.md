## 1. 概述

### 1.1 目的与范围
本文档将 `docs/v2.3/proposal.md` 中已确认的深度代码扫描增强提案落为可验收、可追溯的需求基线（REQ/REQ-C/GWT），作为后续 Design/Planning/Implementation/Testing 的唯一需求输入。

**In Scope（与 proposal 一致）**：
- 打通 `capability_item -> Agent` 评估链路。
- 引入 AST 解析替换关键正则提取路径。
- 新增调用链分析、服务依赖分析、数据流分析、复杂度度量。
- 新增变更影响面分析（系统/功能点/接口三级摘要）。
- 打通 GitLab 代码来源三模式：Archive / Compare / Raw。
- 在现有页面做最小前端接入展示关键扫描证据，不新增独立图谱可视化页面。
- 保留功能级与版本级回滚能力，发布前完成回滚验证。

**Out of Scope / Non-goals（与 proposal 一致）**：
- 不新增完整调用图/依赖图/数据流可视化页面。
- 不重构现有任务评估主链路 UI 信息架构。
- 不引入新的重型基础设施，不改变现有部署形态。

### 1.2 背景、约束与关键假设
**现状与痛点**：
- 现有代码扫描以正则抽取为主，面对复杂代码结构时存在漏识别与误识别。
- `capability_item` 已能入库，但对 Agent 评估链路的有效利用不足，导致“扫描结果 -> 评估结果”的可达性不稳定。
- 现有扫描来源主要覆盖本地路径/压缩包，Git URL 路径在当前实现中未完成真实拉取闭环，难以支撑真实 GitLab 增量迭代场景。

**约束（必须满足）**：
- 沿用现有后端 API 与服务架构（FastAPI + 现有代码扫描服务 + 向量存储）。
- 权限边界不放宽：仅 `manager/admin` 可触发扫描，且非 admin 需满足系统 owner/B 角约束；作业查询/入库仅允许创建者（或 admin）。
- 高风险输入必须保留边界控制：本地路径 allowlist、Git host allowlist、压缩包大小/文件数限制、错误码可观测。
- 发布前必须验证两级回滚路径：L1（功能开关回滚）与 L2（版本回滚到 `v2.2`）。

**关键假设（含验证方向）**：
- A1：`/api/v1/code-scan` 现有作业接口继续作为主承载，v2.3 在此基础上扩展能力而非平行新建一套扫描任务体系。
- A2：深度扫描新增产物（调用图/依赖/数据流/复杂度/影响面）可通过统一扫描结果结构输出，并可被后续 Agent 链路消费。
- A3：误识别率基线 `B` 可由固定回归集与固定脚本计算得到，且发布前可重复复现（具体脚本路径与样本边界将在 §6.3 固化）。

### 1.3 术语与口径
- **`capability_item`**：代码扫描入库后的能力条目知识类型，用于系统识别与功能点归属推理输入。
- **深度扫描**：在基础入口提取之外，补充 AST、调用链、依赖、数据流、复杂度、影响面等结构化信号的扫描过程。
- **影响面分析**：基于代码变更生成受影响的系统/功能点/API 摘要，用于评估结果解释与人工校正。
- **GitLab 三模式**：
  - Archive：针对仓库归档包扫描；
  - Compare：针对两个提交区间的变更扫描；
  - Raw：针对仓库原始内容（分支/目录）扫描。
- **误识别率（核心指标）**：在固定回归集上，系统识别或功能点归属与标注真值不一致的样本占比。
- **链路可达率（M1）**：`capability_item` 被成功送入评估链路并参与判定的用例占比。
- **AST/调用图/数据流/复杂度覆盖率（M2-M5）**：对应可分析对象中被成功解析或产出指标的占比。
- **GitLab 扫描通过率（M6）**：Archive/Compare/Raw 三种模式通过数量占比，且同时满足任务成功率门槛。

## 2. 业务场景说明

### 2.1 角色与对象
- 角色
  - `manager`：发起代码扫描任务、查看扫描结果摘要、推动需求评估。
  - `admin`：审核扫描质量与稳定性、处理异常任务、执行发布前回滚演练。
  - `expert`：消费评估结果，关注系统识别与功能点归属准确性改进（间接受益角色）。
- 对象
  - `ScanJob`：扫描任务（含 `job_id`、状态、进度、结果路径、错误信息、来源类型）。
  - `CapabilityItem`：扫描入库能力条目（知识类型 `capability_item`）。
  - `DeepScanArtifact`：深度扫描结构化产物（AST、调用链、服务依赖、数据流、复杂度、影响面）。
  - `GitLabSource`：GitLab 扫描来源抽象（Archive / Compare / Raw）。
  - `EvaluationInputBundle`：Agent 评估输入包（能力条目 + 深度扫描摘要 + 影响面摘要）。
  - `RollbackRunbook`：回滚执行与验证步骤（L1 功能回滚 / L2 版本回滚）。

### 2.3 场景明细

#### SCN-001：提交本地路径扫描任务
**场景分类**：扫描  
**主要角色**：manager  
**相关对象**：ScanJob、GitLabSource  
**关联需求ID**：REQ-005、REQ-C004、REQ-C005  
**前置条件**：用户已登录且具备 manager/admin 角色；系统已配置 owner/B 角关系  
**触发条件**：用户提交本地绝对路径 `repo_path` 发起扫描  
**流程步骤**：
1. manager 选择目标系统并填写本地仓库绝对路径
2. 系统校验路径在 allowlist 范围内且目录存在
3. 系统创建扫描任务并返回 `job_id`、初始状态和进度
**输出产物**：ScanJob 任务记录  
**异常与边界处理**：路径不存在或越界时拒绝并返回统一错误码（如 `SCAN_004`）

#### SCN-002：提交归档包扫描任务
**场景分类**：扫描  
**主要角色**：manager  
**相关对象**：ScanJob、GitLabSource  
**关联需求ID**：REQ-005、REQ-C005  
**前置条件**：用户已登录且具备 manager/admin 角色；可访问目标系统  
**触发条件**：用户上传 `repo_archive` 发起扫描  
**流程步骤**：
1. manager 上传仓库压缩包并提交扫描
2. 系统执行安全解压（含路径穿越、链接文件、大小与文件数上限校验）
3. 系统创建任务并异步执行扫描
**输出产物**：ScanJob 任务记录、解压目录与扫描结果文件  
**异常与边界处理**：压缩包格式非法/超限时返回 `SCAN_005` 或 `SCAN_006`

#### SCN-003：提交 GitLab 三模式扫描任务
**场景分类**：扫描  
**主要角色**：manager/admin  
**相关对象**：ScanJob、GitLabSource、DeepScanArtifact  
**关联需求ID**：REQ-005、REQ-103  
**前置条件**：GitLab 凭据可用；目标仓库可访问  
**触发条件**：用户选择 Archive/Compare/Raw 任一模式发起扫描  
**流程步骤**：
1. 用户选择扫描模式并输入必要参数（仓库、分支或提交区间）
2. 系统获取对应源码快照并发起扫描
3. 系统返回任务状态并在完成后提供结果
**输出产物**：三模式扫描任务与结果记录  
**异常与边界处理**：凭据无效或来源不可访问时返回可追踪错误并不阻塞其他模式验证

#### SCN-004：查询扫描状态与失败原因
**场景分类**：监控  
**主要角色**：manager/admin  
**相关对象**：ScanJob  
**关联需求ID**：REQ-005、REQ-102  
**前置条件**：已有有效 `job_id`  
**触发条件**：用户查询作业状态  
**流程步骤**：
1. 用户访问任务状态接口或页面状态面板
2. 系统返回状态（queued/running/completed/failed/timeout）和进度
3. 若失败，系统返回可读错误信息与错误码
**输出产物**：任务状态快照  
**异常与边界处理**：任务不存在返回 `SCAN_002`；非创建者访问返回 `AUTH_001`

#### SCN-005：扫描结果入库与画像更新
**场景分类**：入库  
**主要角色**：manager/admin  
**相关对象**：CapabilityItem、ScanJob、EvaluationInputBundle  
**关联需求ID**：REQ-001、REQ-005、REQ-102  
**前置条件**：扫描任务状态为 completed  
**触发条件**：用户执行 ingest  
**流程步骤**：
1. 用户触发扫描结果入库
2. 系统将能力条目写入知识库，标记任务 `ingested=true`
3. 系统更新系统画像完整度并触发画像总结（失败不影响主流程）
**输出产物**：入库统计（success/failed/errors）、画像完整度变更  
**异常与边界处理**：任务未完成返回 `SCAN_003`；embedding 不可用返回 `EMB_001`

#### SCN-006：扫描产物参与 Agent 评估
**场景分类**：评估  
**主要角色**：manager/expert  
**相关对象**：CapabilityItem、DeepScanArtifact、EvaluationInputBundle  
**关联需求ID**：REQ-001、REQ-002、REQ-003、REQ-004  
**前置条件**：能力条目与深度扫描摘要已产生并可查询  
**触发条件**：用户提交需求评估或触发重评估  
**流程步骤**：
1. 系统检索相关 `capability_item` 与深度扫描摘要
2. 系统将结构化信号组装进入 Agent 输入包
3. Agent 输出系统识别与功能点归属结果，并附关键证据摘要
**输出产物**：评估结果 + 证据摘要  
**异常与边界处理**：当部分深度信号缺失时保底降级，不中断主评估流程

#### SCN-007：现有页面展示关键扫描证据
**场景分类**：展示  
**主要角色**：manager/admin  
**相关对象**：DeepScanArtifact、EvaluationInputBundle  
**关联需求ID**：REQ-006、REQ-C003  
**前置条件**：扫描结果与评估结果已生成  
**触发条件**：用户在现有任务/画像相关页面查看结果  
**流程步骤**：
1. 页面加载关键指标（覆盖率、链路可达率、任务成功率）与证据摘要
2. 页面提供来源说明与更新时间
3. 用户据此判断是否需要人工修正
**输出产物**：可读证据摘要展示  
**异常与边界处理**：数据缺失时显示明确空态或降级提示，不新增独立图谱页面入口

#### SCN-008：发布前执行双路径回滚演练
**场景分类**：回滚  
**主要角色**：admin  
**相关对象**：RollbackRunbook、ScanJob  
**关联需求ID**：REQ-008、REQ-C002  
**前置条件**：发布候选版本可部署；回滚命令可执行  
**触发条件**：发布前质量门禁  
**流程步骤**：
1. 执行 L1：关闭功能开关并重启服务，验证 API 健康与关键回归
2. 执行 L2：切换版本到 `v2.2` 并部署，验证同一回归集
3. 记录演练结果与恢复步骤
**输出产物**：回滚演练记录与验证证据  
**异常与边界处理**：任一路径失败时禁止发布并进入修复流程

#### SCN-009：输入边界与权限拒绝处理
**场景分类**：安全  
**主要角色**：manager/admin  
**相关对象**：ScanJob、GitLabSource  
**关联需求ID**：REQ-C004、REQ-C005  
**前置条件**：存在越权或非法输入  
**触发条件**：用户以非法参数或无权限身份请求扫描能力  
**流程步骤**：
1. 系统执行权限校验与输入边界校验
2. 对不满足条件的请求直接拒绝
3. 返回统一错误结构（含 `error_code`、`request_id`、`details.reason`）
**输出产物**：拒绝响应与审计日志  
**异常与边界处理**：拒绝不应造成系统状态污染或任务脏数据

### 3.2 功能性需求明细

#### REQ-001：`capability_item` 评估链路打通
**目标/价值**：确保扫描能力条目对系统识别与功能点归属“可达且可用”，直接支撑准确率提升。  
**入口/触发**：扫描结果入库后触发需求评估或重评估  
**前置条件**：扫描任务已完成并入库；目标系统存在有效任务  
**主流程**：
1. 扫描结果入库写入 `capability_item`
2. 评估请求触发时检索与任务相关的能力条目
3. 将能力条目注入 Agent 输入并参与判定
**输入/输出**：
- 输入：`system_id/system_name`、任务上下文、入库知识条目
- 输出：包含能力条目证据的评估结果
**业务规则**：
- 链路失败不应导致评估主流程不可用，需返回降级标识
- 同一任务重复查询结果应稳定，不允许随机丢失能力条目
**异常与边界**：
- 无能力条目时允许降级评估，并显式标记“代码证据不足”
- 条目检索失败时必须记录可追踪日志
**验收标准（GWT，必须可判定）**：
- [ ] GWT-REQ-001-01: Given 某系统已完成扫描并成功入库至少 1 条 `capability_item`，When 触发该系统任务评估，Then Agent 输入中包含 `capability_item` 证据且评估结果返回对应证据摘要
- [ ] GWT-REQ-001-02: Given 某系统无可用 `capability_item`，When 触发评估，Then 评估流程可完成且结果中标记“代码证据不足/降级”
- [ ] GWT-REQ-001-03: Given 固定回归集中的链路用例，When 执行全量回归，Then `capability_item` 评估链路可达率 `>=95%`
**关联**：SCN-005、SCN-006；指标：M1

#### REQ-002：AST 解析替换关键正则路径
**目标/价值**：降低复杂代码结构下的漏识别/误识别。  
**入口/触发**：代码扫描任务执行  
**前置条件**：可解析源码可访问  
**主流程**：
1. 扫描引擎按语言类型加载 AST 解析器
2. 以 AST 结果提取入口点、方法、调用关系基础节点
3. AST 失败时按策略降级并记录失败原因
**输入/输出**：
- 输入：源码文件与扫描参数
- 输出：AST 节点与提取结构
**业务规则**：
- AST 为主路径；降级路径仅用于容错，需可统计
- AST 输出需保留文件/行号定位信息
**异常与边界**：
- 语法错误文件允许跳过，但必须计入失败统计
- 不支持语言类型需显式标记为“不可解析”
**验收标准（GWT，必须可判定）**：
- [ ] GWT-REQ-002-01: Given 可解析代码文件集合，When 执行扫描，Then 每个成功解析文件均产出 AST 结构并包含文件定位信息
- [ ] GWT-REQ-002-02: Given 存在语法错误文件，When 执行扫描，Then 扫描任务不失败且错误文件在统计中可追踪
- [ ] GWT-REQ-002-03: Given 固定回归集，When 执行全量回归，Then AST 解析覆盖率 `>=95%`
**关联**：SCN-001、SCN-002、SCN-003；指标：M2

#### REQ-003：深度扫描产物生成
**目标/价值**：补足“仅入口提取”不足，增强可解释性与判定信号丰富度。  
**入口/触发**：代码扫描任务执行  
**前置条件**：AST 或可降级解析结果可用  
**主流程**：
1. 生成方法级调用关系
2. 识别服务依赖与外部调用关系
3. 识别实体读写关系并计算复杂度指标（CC/WMC）
**输入/输出**：
- 输入：解析后的代码结构
- 输出：调用图、依赖关系、数据流关系、复杂度指标
**业务规则**：
- 产物结构应统一并可被后续影响面分析与评估流程复用
- 覆盖率统计必须按“可分析对象”计算
**异常与边界**：
- 对动态反射调用可标注“不可静态确定”，不阻断任务
- 对超大文件可启用分段分析，避免任务超时
**验收标准（GWT，必须可判定）**：
- [ ] GWT-REQ-003-01: Given 具备方法调用关系的样例工程，When 扫描完成，Then 结果中存在方法级调用关系数据
- [ ] GWT-REQ-003-02: Given 存在跨服务调用的样例工程，When 扫描完成，Then 结果中存在服务依赖关系数据
- [ ] GWT-REQ-003-03: Given 存在实体读写逻辑的样例工程，When 扫描完成，Then 结果中存在数据流读写关系数据
- [ ] GWT-REQ-003-04: Given 可分析方法集合，When 扫描完成，Then 复杂度指标（CC+WMC）覆盖率 `>=95%`
**关联**：SCN-003、SCN-006；指标：M3/M4/M5

#### REQ-004：变更影响面分析
**目标/价值**：让“代码变更 -> 业务影响”可追溯，支撑评估解释和人工修正。  
**入口/触发**：增量扫描任务（尤其 Compare 模式）  
**前置条件**：可获得变更文件集合与深度扫描产物  
**主流程**：
1. 识别变更涉及的代码节点
2. 关联到系统、功能点、API 三层对象
3. 输出影响面摘要并注入评估输入
**输入/输出**：
- 输入：变更文件集、调用/依赖/数据流结果
- 输出：系统级、功能点级、API 级影响面摘要
**业务规则**：
- 影响面摘要必须包含来源证据（文件/方法/调用路径）
- 影响面缺失时需显式标记，而不是静默忽略
**异常与边界**：
- 无变更文件时返回空影响面摘要并标注原因

**验收标准（GWT，必须可判定）**：

- [ ] GWT-REQ-004-03: Given 存在 API 入口变更，When 扫描完成，Then 结果中包含 API 级影响面摘要并可回溯到源码位置
**关联**：SCN-003、SCN-006；指标：M4

#### REQ-005：GitLab 三模式扫描能力
**目标/价值**：覆盖真实仓库全量/增量扫描场景，避免只支持本地输入。  
**入口/触发**：创建扫描任务  
**前置条件**：GitLab 凭据与仓库访问配置有效  
**主流程**：
1. 接收扫描模式（Archive/Compare/Raw）及必要参数
2. 构建对应源码输入并创建扫描任务
3. 输出统一任务状态，支持查询、重试与强制重跑
**输入/输出**：
- 输入：系统标识、模式参数、可选 `force`

**业务规则**：
- 三模式共享同一任务状态机与结果契约

**异常与边界**：

- [ ] GWT-REQ-005-02: Given Compare 模式参数合法，When 提交扫描任务，Then 任务可创建并在回归中完成
- [ ] GWT-REQ-005-03: Given Raw 模式参数合法，When 提交扫描任务，Then 任务可创建并在回归中完成
- [ ] GWT-REQ-005-04: Given 三模式全量回归，When 汇总结果，Then 模式通过率为 `3/3` 且任务成功率 `>=99%`

#### REQ-007：扫描结果统一输出契约

**输入/输出**：

- [ ] GWT-REQ-007-03: Given UI 与 Agent 使用同一结果对象，When 渲染/评估，Then 不发生字段缺失导致的解析失败
**关联**：SCN-004、SCN-006、SCN-007

### 4.2 非功能需求明细

#### REQ-101：误识别率改进与可证明性
**需求分类**：质量  
**适用范围**：系统识别与功能点归属评估全流程  
**指标与口径**：
- 误识别率公式：`误识别样本数 / 总样本数`
- 基线 `B`：固定回归集在 `v2.2` 下的测得值
- 目标：`v2.3 误识别率 <= B * 0.7`
- 统计窗口：每次全量回归 + 发布前一次
**验收标准（GWT，必须可判定）**：
- [ ] GWT-REQ-101-01: Given 固定回归集与 `v2.2` 基线 `B` 已确定，When 在同一回归集执行 `v2.3` 评估，Then 误识别率 `<= B * 0.7`
- [ ] GWT-REQ-101-02: Given 输出测试报告，When 审查误识别率指标，Then 报告中包含分子/分母、样本集版本、计算脚本路径与执行时间
**验收方法/证据**：`test_report.md` 指标块 + 回归命令输出  
**异常与边界**：样本集发生变更时必须重新冻结基线并记录版本  
**关联**：M1、P-METRIC-01

#### REQ-102：覆盖率与链路可达率门槛
**需求分类**：可观测性  
**适用范围**：扫描链路、解析链路、深度分析链路  
**指标与口径**：
- M1 链路可达率 = `通过链路用例数 / 总链路用例数`，目标 `>=95%`
- M2 AST 覆盖率 = `AST 成功解析文件数 / 可解析文件数`，目标 `>=95%`
- M3 调用图覆盖率 = `入图方法数 / 可分析方法数`，目标 `>=85%`
- M4 数据流覆盖率 = `已识别读写关系实体数 / 可分析实体数`，目标 `>=80%`
- M5 复杂度覆盖率 = `具备 CC+WMC 指标的方法数 / 可分析方法数`，目标 `>=95%`
**验收标准（GWT，必须可判定）**：
- [ ] GWT-REQ-102-01: Given 一次全量回归执行完成，When 生成扫描统计报告，Then 报告中包含 M1-M5 全部指标的分子/分母/百分比
- [ ] GWT-REQ-102-02: Given 发布前门禁检查，When 读取本次全量回归统计，Then M1-M5 指标均达到各自阈值（95/95/85/80/95）
**验收方法/证据**：`test_report.md` + 扫描统计输出 + 门禁检查记录  
**异常与边界**：不可分析对象必须单独统计，禁止并入分母掩盖失败  
**关联**：M1-M5、P-METRIC-02/P-METRIC-03

#### REQ-103：GitLab 三模式稳定性门槛
**需求分类**：稳定性  
**适用范围**：GitLab Archive / Compare / Raw 三模式  
**指标与口径**：
- 模式通过率 = `已通过模式数 / 3`，目标 `= 3/3`
- 任务成功率 = `成功任务数 / 任务总数`，目标 `>=99%`
- 统计窗口：每次全量回归 + 发布前一次
**验收标准（GWT，必须可判定）**：
- [ ] GWT-REQ-103-01: Given 三模式回归用例全部执行，When 汇总结果，Then 模式通过率 `=3/3` 且任务成功率 `>=99%`
- [ ] GWT-REQ-103-02: Given 任一模式执行失败，When 生成回归报告，Then 报告明确失败模式、失败原因与重试结果，不遗漏其他模式统计
**验收方法/证据**：`test_report.md` + 扫描任务日志  
**异常与边界**：单模式失败可重试，但发布门禁判定必须以最终统计结果为准  
**关联**：M6、P-METRIC-04

## 5. 权限与合规
### 5.1 权限矩阵
| 角色 | 操作/权限 | 资源范围 | 备注 |
|---|---|---|---|
| admin | 创建扫描任务（本地/Archive/Compare/Raw） | 全系统 | 不受“任务创建者”限制，可查看全量任务 |
| manager | 创建扫描任务（本地/Archive/Compare/Raw） | 主责系统或 B 角系统 | 需通过 owner/B 角校验 |
| admin | 查询任务状态与结果 | 全量任务 | 可查看任意 `job_id` |
| manager | 查询任务状态与结果 | 仅本人创建任务 | 非创建者返回 `AUTH_001` |
| admin | 执行 ingest | 全量任务 | 可对任意 completed 任务入库 |
| manager | 执行 ingest | 仅本人创建任务 | 非创建者或状态不合法均拒绝 |
| expert | 查看评估结果中的代码证据摘要 | 任务可见范围 | 不允许触发扫描与入库 |
| viewer | 查看页面摘要展示（只读） | 页面可见范围 | 不允许触发扫描与入库 |

### 5.2 客户信息与合规
- **敏感字段与最小化原则**：
  - 扫描输入中涉及 `repo_path`、GitLab 访问凭据、仓库 URL 等敏感信息，日志输出必须脱敏（不得打印完整凭据）。
  - 结果展示仅输出必要摘要，不回显敏感连接信息。
- **密钥管理**：
  - GitLab 凭据采用环境变量或安全注入方式管理；禁止写入仓库与明文配置文件。
  - 凭据失效或权限不足时返回可判定错误码，不回传敏感细节。
- **输入边界**：
  - 本地路径必须绝对路径且命中 allowlist。
  - Git URL 必须命中 host allowlist。
  - 上传压缩包必须通过格式、大小、文件数量、链接文件检查。
- **审计与可追溯**：
  - 每次任务创建/查询/入库均记录 `request_id`、操作者、时间、结果码。
  - 发布前回滚演练记录必须可追溯到执行命令与结果。

### 6.2 错误码与提示语
| 错误码 | 场景 | 提示语 | HTTP状态码 | 处理建议 |
|---|---|---|---|---|
| `SCAN_001` | 仓库路径不可访问/Git URL 不可用 | 代码仓库路径不存在或无法访问 | 400/500 | 检查路径、Git 配置与权限 |
| `SCAN_002` | 任务不存在 | 扫描任务不存在 | 404 | 核对 `job_id` |
| `SCAN_003` | 任务状态不允许操作 | 扫描任务状态不允许该操作 | 400 | 等待任务完成后重试 |

| `SCAN_005` | 压缩包非法 | 仓库压缩包格式不支持或解压失败 | 400 | 使用受支持格式重新上传 |
| `SCAN_006` | 压缩包超限 | 仓库压缩包解压后超出大小或文件数量限制 | 400 | 缩小包体或分批扫描 |
| `SCAN_007` | 模式参数不完整或冲突 | 扫描模式参数非法或缺失 | 400 | 按模式补齐必填参数 |

| `AUTH_001` | 权限不足/越权访问 | 权限不足 | 403 | 使用具备权限的账号或资源 |

### 6.3 指标与计算口径
- **误识别率基线 B 固化（关闭 proposal 开放问题 #1）**：
  - 固定样本集路径：`tests/fixtures/regression/v23_misidentify_set.jsonl`
  - 计算脚本路径：`scripts/metrics/compute_misidentification_rate.py`
  - 基线冻结规则：样本集版本固定后生成 `B`，发布窗口内不得变更样本与脚本版本。
- **M1-M6 计算口径**（与 `status.md` 一致）：
  - M1：`capability_item` 评估链路可达率 = `通过用例数 / 总用例数`
  - M2：AST 覆盖率 = `AST 成功解析文件数 / 可解析文件数`
  - M3：调用图覆盖率 = `入图方法数 / 可分析方法数`
  - M4：数据流覆盖率 = `已识别读写关系实体数 / 可分析实体数`
  - M5：复杂度覆盖率 = `具备 CC+WMC 指标的方法数 / 可分析方法数`
  - M6：GitLab 扫描通过率 = `已通过模式数 / 3`，且任务成功率 `>=99%`
- **统计窗口**：每次全量回归 + 发布前一次。
- **精度与舍入**：百分比保留两位小数，四舍五入；分母为 0 时输出 `N/A` 并记录原因。

### 6.4 API 契约（最小集合）
| API-ID | 路径 | 方法 | 用途 | 权限 | 关键参数/返回 |
|---|---|---|---|---|---|
| API-001 | `/api/v1/code-scan/jobs` | POST | 创建扫描任务 | manager/admin + owner/B 角校验 | 入参：`system_name/system_id/repo_source/repo_path|repo_archive|git_params/options/force`；返回：`job_id/status/progress` |
| API-002 | `/api/v1/code-scan/jobs/{job_id}` | GET | 查询任务状态 | admin 或任务创建者 | 返回：`job_id/status/progress/result_path/message` |
| API-003 | `/api/v1/code-scan/jobs/{job_id}/ingest` | POST | 入库并更新画像 | admin 或任务创建者 | 返回：`success/failed/errors` |
| API-004 | `/api/v1/requirement/evaluate` | POST | 触发评估并消费证据 | manager/admin | 返回中包含代码证据摘要与影响面摘要 |
| API-005 | 现有任务/画像页面查询接口 | GET | 获取 EvidenceSummary | 页面可见角色 | 返回：链路可达率、覆盖率、任务成功率、更新时间、摘要文本 |

#### API-001 模式参数约束（强制）
| `repo_source` | 必填参数 | 禁止参数 | 备注 |
|---|---|---|---|
| `local` | `repo_path`（绝对路径） | `repo_archive`、`gitlab_*` | 路径需命中 allowlist |
| `archive` | `repo_archive` | `repo_path`、`gitlab_*` | 压缩包需通过安全解压校验 |
| `gitlab_archive` | `git_project_id`、`archive_ref` | `repo_path`、`repo_archive` | 用于仓库归档模式 |
| `gitlab_compare` | `git_project_id`、`compare_base_ref`、`compare_head_ref` | `repo_path`、`repo_archive` | 用于增量变更模式 |
| `gitlab_raw` | `git_project_id`、`raw_ref`、`raw_path` | `repo_path`、`repo_archive` | 用于分支目录原始内容模式 |

> 任一模式缺参、混参或冲突参数，统一返回 `SCAN_007`。

> 关闭 proposal 开放问题 #2：最小前端接入页面固定为“任务详情相关页面 + 系统画像信息展示页面”，展示字段固定为：`M1~M6 关键指标`、`影响面摘要`、`更新时间`、`数据来源`。  
> 关闭 proposal 开放问题 #3：GitLab 凭据管理采用环境变量/安全注入，不落盘；输入边界与错误码按 §5.2 与 §6.2 执行。
