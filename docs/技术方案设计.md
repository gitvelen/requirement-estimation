## 0. 摘要（Executive Summary）

需求评估系统将“需求文档”转化为可用于采购/投标/评审的“可审计工作量评估”输出，核心链路为：

1. 需求文档上传与解析（抽取正文）
2. 系统识别（`selected/maybe/questions` 分层）
3. 按系统拆分功能点（模板化输出 + 证据引用 + 归属复核提示）
4. 区间估算与风险提示（置信度、假设、风控标识）
5. 多专家多轮盲评与偏离度分析（偏离度阈值触发下一轮）
6. 报告生成与版本管理（PDF 为主；Excel 为历史兼容/按需）

为提升准确性与可追溯性（尤其采购严肃场景），系统引入四类约束/事实信号并实现“权限 + 审计”闭环：

- A层证据库：DOCX/PPTX/PDF 材料导入、切块检索与在线预览（不下载）
- B层系统画像：能力边界结构化摘要（草稿/发布），字段可回链证据
- I层 ESB 索引：集成点明细导入/检索/统计，用于提示与澄清问题候选（不等价于功能清单）
- C层代码扫描：Spring Boot/Java 轻量扫描产出能力目录（事实层），用于召回与归属校验提示（禁止扩范围）

当前实现以“文件存储（`data/*.json`）+ 文件锁 + 原子写”为主，适合单机/小规模快速落地；可平滑演进至数据库/对象存储/Milvus/异步作业，以支持规模化与更强审计能力。

## 1. 设计目标与非目标

### 1.1 设计目标

- 目标1（FUNC-001/FUNC-002/FUNC-024）：统一认证与授权（JWT + 角色校验），保证权限边界不可突破（401/403 清晰可观测）。
- 目标2（FUNC-005/FUNC-020/FUNC-021）：实现“证据约束 + 严肃模式阻断”，证据不足/冲突时禁止输出确定结论，仅输出 `maybe + questions`。
- 目标3（FUNC-008/FUNC-020/FUNC-021）：关键修正可审计：功能点编辑、证据等级覆盖修正、规则变更均需留痕，可回放与追溯。
- 目标4（FUNC-011/FUNC-012）：多专家多轮盲评收敛争议点：偏离度阈值触发下一轮，第二/三轮仅评估高偏离功能点。
- 目标5（FUNC-015~FUNC-019/FUNC-029）：四类事实信号（A/B/I/C）可独立演进并可注入主链路；任何召回仅用于提示复核，禁止扩范围。
- 目标6（FUNC-016）：证据材料对 expert 仅在线预览，不提供下载；范围收敛到其参与任务。
- 目标7（FUNC-025）：可部署可运维：Docker Compose 对外 80，`GET /api/v1/health` 可用，日志可用于快速定位 `/api` 502、模型不可用等问题。
- 目标8（FUNC-014）：数据可控：文件存储起步（data/*.json + 原子写），保留与清理策略可配置，支持平滑演进至 DB/Milvus/对象存储。

### 1.2 非目标（Non-goals）

- 不做端到端自动报价/自动招采定价（仅做评估与风控提示）。
- 不做全量代码语义理解（代码扫描仅做事实层能力目录提取）。
- 禁止自动扩范围（无论证据/代码/ESB召回到什么，都只能“提示复核”，不得自动新增需求范围）。
- 不强制 OCR 扫描版 PDF（仅支持可抽取文本的 PDF 入库检索）。

### 1.3 关键约束（Constraints）

- C1 当前存储为文件：单机可用；多实例强一致不保证（后续演进落库/对象存储/分布式锁）。
- C2 证据预览：仅在线预览（PDF inline / 文本抽取），不提供下载；expert 预览范围必须按任务收敛。
- C3 模型依赖：`DASHSCOPE_API_KEY` 未配置时允许系统启动，但 AI/Embedding 能力可能不可用或走降级策略（以配置为准）。

## 2. 系统上下文与边界

### 2.1 系统边界与输入输出

- 系统边界：输入为“需求文档 + 事实信号（A/B/I/C）+ 配置（系统清单/规则/用户）”，输出为“可审计的评估结果与报告版本”。
- 责任边界：系统负责任务工作流、权限控制、审计留痕、报告产出；不负责企业招采定价决策。

### 2.2 外部依赖与失败模式

- 外部依赖：DashScope（LLM + Embedding）；（可选）Milvus；浏览器；Nginx。
- 失败模式（必须可观测且可人工推进）：
  - 模型/Embedding 不可用：明确提示 + 允许人工编辑链路；检索能力可降级或跳过。
  - 文件存储异常：记录错误日志并返回可理解错误；避免 silent failure。
  - 证据预览渲染失败：降级为抽取文本预览（不泄露原件下载能力）。

### 2.3 质量属性与验收口径（Quality Requirements）

质量目标（优先级排序）：

1. 正确性：不扩范围；证据不足触发阻断；权限边界不可突破
2. 可追溯性：输出带证据引用；覆盖修正可回放
3. 可用性：部署后可完成登录与主流程；健康检查可用
4. 可维护性：模块边界清晰；可在不影响主链路的情况下迭代 A/B/I/C
5. 性能：中小规模单机可用；可通过 Milvus/异步化扩展

典型质量场景（用于验收抽样）：

- Q-01（安全）：专家预览证据必须校验其参与任务，否则 403；且不提供下载
- Q-02（正确性）：证据不足时，系统识别必须降级为 `maybe + questions` 并显著提示高风险
- Q-03（可追溯）：项目经理对任务内证据等级覆盖修正必须强制填写备注并写入审计
- Q-04（鲁棒性）：证据导入若无法抽取文本（扫描PDF等），必须失败并给出可理解原因
- Q-05（可用性）：Nginx 对外 80 端口可访问登录页，`/api/v1/health` 返回 healthy

### 2.4 术语与数据分层（Domain Glossary）

- 系统识别分层：
  - selected：当前证据足以“确定涉及该系统”（严肃模式下需满足证据等级规则）
  - maybe：证据不足或存在冲突，仅作为候选，需澄清
  - questions：最小澄清问题集（建议 3~8 条）
- 证据/事实分层：
  - A层证据库：非结构化材料切块检索（可回放、可预览）
  - B层系统画像：结构化能力边界（草稿/发布）
  - I层 ESB 索引：集成点明细（只代表集成点，不代表功能清单）
  - C层能力目录：代码扫描事实层（用于召回与校验提示，不扩范围）
- 证据等级（E0~E3）：综合反映“本任务对该系统的证据可用性”，用于严肃模式阻断与风险提示。

## 3. 总体架构

### 3.1 架构概述（多视图）

#### 3.1.1 系统上下文（Context View）

```
用户（admin/manager/expert）
        │ 浏览器
        ▼
Nginx（frontend 容器，80对外）
  ├─ 静态资源（React build）
  └─ /api/* 反代到 backend:8000
        ▼
FastAPI（backend 容器，8000）
  ├─ 认证与权限（JWT / require_roles）
  ├─ 任务与评估（工作流/多轮/报告/归档）
  ├─ Agent编排（系统识别/拆分/估算）
  ├─ 索引能力：证据库A / 画像B / ESB I / 代码扫描C / 系统知识
  └─ 规则与审计：证据等级规则/任务内覆盖/变更日志
        ▼
Storage（宿主机卷）
  ├─ data/*（结构化数据、索引、报告产物）
  ├─ uploads/*（证据原件）
  └─ logs/*（运行日志）

外部依赖：DashScope（LLM/Embedding），可选 Milvus（向量检索）
```

#### 3.1.2 模块视图（Module View）

后端（关键目录与职责）：

- `backend/app.py`：FastAPI 入口，注册各模块 router
- `backend/api/`：路由层（权限 + DTO + 编排入口）
- `backend/service/`：领域服务层（存储、检索、ESB、扫描、审计、效果指标、区间估算等）
- `backend/agent/`：Agent 与编排（系统识别、功能拆分、区间估算）
- `backend/prompts/`：提示词模板（约束输出、阻断、格式化）

前端（关键目录与职责）：

- `frontend/src`：React SPA（路由守卫 + 各业务页面）
- `frontend/nginx.conf`：静态资源 + `/api` 反代

#### 3.1.3 运行时视图（Runtime View）

主链路时序（简化）：

```
manager → 前端：上传需求
前端 → 后端：POST /api/v1/requirement/upload
后端 → 文档解析：抽取需求正文
后端 → Agent编排：系统识别 → 功能拆分 → 区间估算（含风险）
后端 → 存储：写 task_storage.json（原子写）
manager → 前端：进入编辑页复核/修正 → 保存
manager → 后端：提交给管理员
admin → 后端：分配专家
expert → 后端：拉取评估数据、保存草稿、提交
后端：计算偏离度，必要时触发下一轮
admin/manager：下载报告（版本管理）
```

证据预览时序（专家）：

```
expert → 后端：GET /api/v1/knowledge/evidence/preview/{doc_id}?task_id=...
后端：校验 expert 是否参与 task_id；校验 doc_id 是否在该任务关联证据中
后端：PDF 返回 inline FileResponse；否则返回抽取文本预览（JSON）
```

#### 3.1.4 部署视图（Deployment View）

默认推荐 Docker Compose（见 `docker-compose.yml`）：

- 对外：仅暴露 80（frontend）
- 后端：容器内监听 8000（不映射到宿主机，避免端口冲突）
- 卷挂载：`./data`、`./uploads`、`./logs`
- 反代：`/api/` → `http://backend:8000`
- 健康检查：`GET /api/v1/health`

### 4.2 数据模型

#### 4.2.1 当前存储策略（文件存储 + 原子写）

当前采用“文件存储 + 文件锁 + 原子写（tmp + replace）”：

- 写入前获取 `.lock` 文件互斥锁（fcntl；不支持时降级为进程内锁）
- 修改写入使用 `xxx.tmp` 临时文件，成功后 `os.replace` 原子替换
- 读取容错：文件不存在返回默认空结构；解析失败记录日志并返回兜底

#### 4.2.2 目录布局（默认）

```
data/
  task_storage.json           # 任务主数据（流程/拆分/评估/报告版本）
  task_modifications.json     # 修改记录（按实现）
  users.json                  # 用户与专家库字段
  departments.json            # 部门下拉
```

### 4.3 核心流程

#### 4.3.1 状态机（简化）

- 任务工作流（`workflow_status`）：

```
draft → awaiting_assignment → evaluating → completed → archived
```

- AI处理（`ai_status`）：`pending|processing|success|failed`（失败必须可解释并可人工推进）。

#### 4.3.2 严肃模式阻断（证据约束）

核心原则：当证据不足或冲突时，禁止输出确定结论。

- 系统识别输出必须遵循：`selected/maybe/questions`
- 严肃模式下，`selected` 的判定需结合证据等级规则（E0~E3）；不满足时降级为 `maybe + questions`
- 所有 A/B/I/C 的召回结果只能“提示复核”，禁止自动扩范围

#### 4.3.3 多专家多轮盲评与偏离度（Delphi-like）

- 支持多专家独立盲评，多轮收敛（最多 3 轮）
- 偏离度计算：以 `feature_id` 聚合专家均值，与 AI 估算（或区间中值）计算偏离度百分比
- 偏离度阈值：默认 `20%`（见 `backend/api/routes.py` 的 `DEVIATION_THRESHOLD`）
- 第二/三轮仅对“高偏离功能点”发起评估，降低专家负担

#### 4.3.4 证据预览（不下载 + 任务范围收敛）

- expert 预览必须携带 `task_id`，并满足：
  - expert 参与该任务（assignment 关系）
  - `doc_id` 属于任务关联证据清单
- 返回策略：
  - PDF：`Content-Disposition: inline`，不提供 attachment 下载
  - 其它格式：返回抽取文本预览（JSON）

#### 4.3.5 关键失败路径（至少 3 个）

1. 模型 Key 缺失或模型服务不可用：AI 处理失败，必须明确提示并允许人工推进。
2. 文件读写/锁异常：任务保存失败或数据损坏风险，需记录错误、返回可理解提示，并提供备份/恢复策略。
3. expert 越权预览证据：必须 403 阻断且不返回任何文件内容。
4. 外部回调 SSRF：callback_url 校验失败必须 400，并记录审计线索。

## 5. 可靠性与可观测性

### 5.1 超时/重试/降级策略

- LLM/任务超时：`LLM_TIMEOUT`、`TASK_TIMEOUT` 可配置；失败必须落日志并可人工推进。
- Embedding 降级：`DASHSCOPE_API_KEY` 未配置且 `DEBUG=true`（或 `EMBEDDING_FALLBACK=1`）时启用本地确定性哈希 embedding；用于保证系统可运行与可演示。
- 向量检索降级：本地向量库不可用时允许返回空结果并提示（避免阻断主链路）。
- 证据预览降级：原件不可读/渲染失败时返回抽取文本预览或明确错误（不降级为下载）。

### 5.2 日志、审计与追踪

- 运行日志：stdout + `logs/app.log`
- 审计日志：规则更新、任务内覆盖修正、关键操作必须写 `audit_logs.json`
- 建议增强：request_id、结构化日志、统一异常处理中间件（便于定位 502/权限/模型失败）

### 5.3 健康检查与核心可用性

- `GET /api/v1/health`：基础健康检查
- 最小可用链路：登录 → 任务列表 → 上传 → 进入编辑页（AI失败时可人工继续）

### 5.4 指标与告警（建议）

- 任务成功率/耗时/失败原因分布

## 6. 安全设计

### 6.1 认证与授权

- JWT + 角色校验（admin/manager/expert）
- 管理写接口二级保护（`ADMIN_API_KEY` 或管理员 token，视实现）

### 6.2 输入安全与文件安全

- 上传限制：扩展名/大小限制；文件名清洗（防路径穿越、空字节等）
- 证据预览：仅 inline / 文本预览，不提供下载；expert 预览必须校验任务范围

### 6.3 SSRF 防护

- DevOps 回调：callback_url 必须为内网/本机（私网/loopback/link-local），否则拒绝

### 6.4 数据安全与合规

- 证据原件属于审计关键材料：必须纳入备份与访问控制
- 任务/审计日志需可追溯：关键操作强制备注并留存（覆盖修正、规则变更等）

威胁与缓解（简表）：

| 威胁/攻击面 | 风险 | 缓解措施 |
|---|---|---|
| 证据预览越权 | 高 | 参与关系校验 + doc_id 关联校验；403/404 |
| SSRF（callback_url） | 高 | 仅允许内网/本机地址；不通过则 400 |
| 路径遍历（预览/上传） | 高 | 路径白名单/允许目录校验；文件名清洗 |
| LLM 幻觉导致扩范围 | 高 | 证据约束 + 阻断机制 + “提示复核不自动新增” |

## 7. 性能与容量

### 7.1 现状与瓶颈

- 单机小规模可用；LocalVectorStore 全量扫描相似度计算，数据量增大性能下降
- 报告生成：大任务可能耗时（建议后续异步化）
- 证据切块参数（chunk_size/overlap/max_chunks）影响导入耗时与检索质量

### 7.2 优化与扩展路径

- 向量检索：切换 Milvus + metadata filter（权限/系统/日期等过滤）
- 存储：任务/评估/审计落库；证据原件上对象存储
- 作业系统：导入/扫描/报告异步化（队列 + worker）

### 7.3 容量建议（经验口径）

- 单机：证据块数 < 10 万时可接受；超过建议启用 Milvus 并加强过滤
