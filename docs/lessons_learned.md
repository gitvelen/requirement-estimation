# Lessons Learned（经验教训沉淀）
> 目的：把"踩坑/返工/缺陷/事故/认知偏差"沉淀为可复用的规则与验证方式，减少返工与风险。
> 维护：每次出现返工/缺陷/事故/认知偏差后，由 AI 追加 1 条；高频/高风险项需要"升级"为规则/清单/自动化检查。

## 快速索引（硬规则 Top）
> 只保留最关键的 5–10 条，优先写"可执行、可验收"的规则，便于跨项目复用。

- **R1**（需求）：提案必须包含所有用户确认的决策点，不能遗漏
- **R2**（流程）：文档更新后必须自查"是否包含所有讨论内容"
- **R3**（需求）：用户对问题的回答必须逐条体现在输出文档中
- **R4**（流程）：使用AskUserQuestion等工具交互后，必须将结果结构化写入文档
- **R5**（需求）：Requirements 必须覆盖提案 In Scope 清单（每项落到 REQ/API/验收；若 Defer 必须回写提案并重新确认）
- **R6**（追溯）：任何 REQ 编号调整后，必须同步更新 `plan.md` 的"任务关联 REQ/覆盖矩阵"，并做一次"引用存在性"自检（避免出现 REQ-021 这类漂移）
- **R7**（实现）：用户指出"文档有提"时，必须先完整阅读相关文档章节，列出原文，再讨论差异；不得凭搜索结果质疑用户
- **R8**（实现）：开始任何实现前，必须建立TodoList，逐项勾选完成；完成后更新TodoList状态
- **R9**（文档）：proposal 与 requirements 存在差异时，必须列出两处原文请用户确认，不得自行假设优先级
- **R10**（文档/指标）：关键指标口径必须“术语表 + 指标口径”一致；提交前执行 `rg -n "PM修正率|命中率|MAE|Bias|完整度评分" docs/<版本号>/requirements.md` 做一致性自检

### 2026-02-06｜提案阶段遗漏用户反馈（文档/流程）
- **标签**：文档、流程、需求
- **触发（事实）**：在v2.0提案编写过程中，用户通过多轮交互（AskUserQuestion）提供了大量决策信息，但初始提案未包含这些内容，用户指出"你问的问题和我的答复，你分析后放到提案里了吗？"
- **根因**：
  1. 过早将初始草稿视为"完成"，未将交互过程的结果系统化整理
  2. 缺少"交互→整理→验证"的闭环意识
  3. 没有在提交前自查"是否包含所有用户确认的决策"
- **影响**：
  - 文档不完整，用户需要反复提醒
  - 降低用户信任
  - 增加反复修改的时间成本
- **改进行动（可执行）**：
  1. **交互记录清单**：使用AskUserQuestion后，立即将问答结果整理成清单
  2. **完整性自查**：提交文档前，逐项检查是否包含所有用户确认的内容
  3. **变更版本记录**：每次补充内容后更新版本号（如v0.1→v0.2），并列出变更点
- **验证方式（可复现）**：
  - 提交任何文档前，问自己："用户在这次会话中确认的每一项，都在文档里了吗？"
  - 对比检查：列出用户的所有回答，逐一确认是否在文档中体现
- **升级（规则/清单/自动化）**：
  - 是：升级为 `CLAUDE.md` 核心原则："证据驱动、验证可复现"的具体执行细则
- **证据/关联**：v2.0提案从v0.1到v0.2的多次补充过程

### 2026-02-06｜需求细节遗漏（文档）
- **标签**：文档、需求
- **触发（事实）**：用户指出ESB字段、COSMIC混合方式等具体决策未在提案中详细说明
- **根因**：
  1. 只记录了"做什么"，没记录"具体怎么做"

- **影响**：提案不够具体，无法作为后续设计的依据
- **改进行动（可执行）**：
  1. 用户回答包含具体字段/参数/格式时，必须完整记录，不能简化
  2. 对于用户提出的"混合方式"等决策，需解释具体含义
- **验证方式（可复现）**：
  - 检查：是否所有"XXX包含哪些字段"类问题都有完整回答
- **升级（规则/清单/自动化）**：是

### 2026-02-06｜需求口径不一致引发验收/实现风险（指标/降级/权限/输入边界）
- **标签**：文档、需求、验收、安全
- **触发（事实）**：Requirements 审查发现同一概念在不同章节口径不一致（如 PM修正率分母冲突、embedding 服务不可用降级策略矛盾、本地 repo_path 安全边界缺失、资源级权限未落到 API 契约）。
- **根因**：
  1. 术语表/需求明细/指标/API 契约分散编写，缺少一致性校验
  2. 开放问题未在进入下一阶段前关闭，导致“默认行为”被各段落自行假设
- **影响**：
  - 实现与测试口径不一致，验收容易争议
  - 安全边界缺失可能引入越权/路径风险
- **改进行动（可执行）**：
  1. 指标类需求必须同页给出：公式、分母/分子定义、统计周期、取值范围、目标值
  2. 外部依赖/降级策略必须在需求阶段“一处定、处处用”（需求+API+错误码+验收）
  3. 高风险输入（如本地路径）必须明确输入边界（allowlist/校验/错误码）并给出备选路径（如上传包）
  4. 在进入 Design 前固定执行一致性检查命令（如 `rg -n "PM修正率|embedding服务不可用|repo_path|资源级权限" docs/<版本号>/requirements.md`）
- **验证方式（可复现）**：
  - 以一致性 `rg` 清单作为门禁：同一关键词在文件内出现多处时，逐处检查定义是否一致
- **升级（规则/清单/自动化）**：是（纳入 Requirements 阶段“自查清单”）

### 2026-02-06｜需求阶段遗漏提案已承诺内容（Proposal→Requirements 覆盖缺失）
- **标签**：需求、文档、流程、范围管理、验收
- **触发（事实）**：用户指出 `docs/v2.0/proposal.md` 已写入 In Scope/体验优化的内容（如 COSMIC 配置简化、历史评估文档导入 L0、专家下载报告、AI效果报告导航优化、系统画像可视化/AI使用提示、系统清单 Bug 修复等），但在 `docs/v2.0/requirements.md` 中未落为可验收的 REQ/API（仅零散出现在术语/角色描述或未出现）。
- **根因**：
  1. 需求编写时未把提案的 “In Scope ✅清单” 作为输入基线，缺少逐项落地的覆盖核对
  2. 缺少“提案→需求→API→验收”追溯门禁，导致体验优化类条目被“描述性提及”替代“可验收定义”
  3. 没有在提交前执行差异检查（只做了局部一致性检查，未做覆盖性检查）
- **影响**：
  - 范围承诺与需求单一真相源不一致，设计/实现/测试将产生理解偏差与返工
  - 用户信任下降，需求阶段质量不可控
- **改进行动（可执行）**：
  1. 在 Requirements 阶段建立并维护覆盖矩阵：`Proposal In Scope` → `REQ/REQ-NF` → `API` → `验收(GWT)`（至少保证每条 ✅ 有落点或明确 Defer）
  2. 若某项不进入本期：必须回写 `proposal.md` 的 In Scope/Non-goals，并同步更新 `status.md` 与验收范围
  3. 将“覆盖性检查”加入阶段门禁：每次更新 `requirements.md` 后都要跑一次
- **验证方式（可复现）**：
  - 提取提案范围清单：`rg -n \"^- ✅\" docs/<版本号>/proposal.md`
  - 提取需求落地清单：`rg -n \"^#### REQ-|^#### REQ-NF-|^#### API-\" docs/<版本号>/requirements.md`
  - 人工对齐：逐条 ✅ 必须能映射到 REQ/API/验收；否则判定为遗漏
- **升级（规则/清单/自动化）**：是（升级为 Requirements 阶段必做门禁）
- **证据/关联**：本次对齐检查（Proposal 存在但 Requirements 缺失）及用户反馈

### 2026-02-07｜Planning 追溯矩阵与需求编号漂移（文档/追溯）
- **标签**：文档、追溯、流程、验收
- **触发（事实）**：在 `requirements.md` 已完成 REQ-017~020 编号统一后，`plan.md` 仍残留 `REQ-021` 且多处任务关联 REQ 错配（例如把“报告下载”关联到知识导入），导致追溯矩阵失真。
- **根因**：
  1. 需求编号调整后缺少“全局引用存在性检查”，未同步更新计划文档的任务关联与覆盖矩阵
  2. 依赖人工浏览而非命令级校验，容易漏改
  3. 阶段门禁未把“Plan 引用的 REQ 是否都存在”作为必检项
- **影响**：
  - Implementation 执行入口（plan）与验收口径脱节，可能出现“做了但不算/没做但以为做了”的争议
  - 返工与沟通成本显著上升
- **改进行动（可执行）**：
  1. REQ 编号调整后，必须执行“引用存在性自检”：在 `plan.md/design.md` 中搜索所有 `REQ-` 引用并对照 `requirements.md 3.1` 列表确认存在
  2. Planning 阶段强制维护覆盖矩阵：`REQ → Tasks` 只允许引用当前 `requirements.md` 中存在的 REQ-ID
  3. 版本与变更记录同步：任何修复必须更新文档版本号与变更记录（避免“内容已改但版本未变”）
- **验证方式（可复现）**：
  - 计划引用检查：`rg -n \"REQ-0\" docs/<版本号>/plan.md`
  - 需求列表核对：`rg -n \"^\\| .*\\| REQ-\" docs/<版本号>/requirements.md`
  - 检查不存在的 REQ 引用：人工核对 `plan.md` 中出现但 `requirements.md` 不存在的 REQ-ID（例如 `REQ-021`）
- **升级（规则/清单/自动化）**：是（升级为 `R6`，并纳入 Planning 阶段门禁清单）
- **证据/关联**：`docs/v2.0/plan.md` v0.5→v0.6 修复记录；`docs/v2.0/requirements.md` v1.10/v1.12 变更记录

### 2026-02-07｜弃用告警修复引发兼容性回归（代码/验证）
- **标签**：代码、测试、兼容性、流程
- **触发（事实）**：在治理弃用告警时，将 JWT 过期时间写法从 `datetime.utcnow()` 改为 `datetime.now(datetime.UTC)` 后，触发全量回归失败（53 failed），根因是运行时环境不支持 `datetime.UTC` 属性。
- **根因**：
  1. 改造时假设运行时一定支持 `datetime.UTC`，未先做版本兼容核验
  2. 缺少“改造后立即全量回归”的强制门禁，导致问题未在第一时间暴露
  3. 对“语法可用 ≠ 运行时全环境可用”的风险认识不足
- **影响**：
  - 短时间内造成认证链路失效，连带影响大量 API 测试
  - 增加一次返工与复测成本
- **改进行动（可执行）**：
  1. 时区 UTC 写法统一采用跨版本更稳妥的 `datetime.now(timezone.utc)`
  2. 涉及鉴权/时间/路由基础能力的改动，必须执行后端全量回归 `pytest -q`
  3. 关键兼容改造优先使用“最小差异 + 全量复测”策略，避免一次引入多处语义变化
- **验证方式（可复现）**：
  - 全量回归：`.venv/bin/pytest -q`
  - 关键字检查：`rg -n "datetime\.UTC|timezone\.utc|utcnow\(" backend/api/auth.py`
- **升级（规则/清单/自动化）**：是（纳入 Implementation/Testing 阶段门禁：基础能力改动后强制全量回归）
- **证据/关联**：`backend/api/auth.py` 修复记录；`docs/v2.0/review_deployment.md` 复审证据

### 2026-02-07｜第三方告警治理需“升级优先、白名单兜底”双阶段策略（测试/依赖）
- **标签**：测试、依赖、稳定性、流程
- **触发（事实）**：部署复审阶段出现大量第三方告警（`pymilvus` 与 `langchain_core`），仅靠升级补丁版本仍无法完全消除。
- **根因**：
  1. 告警来自上游库对 Python 3.14 的兼容路径，短期无项目侧彻底修复空间
  2. 过往未建立“第三方告警分层治理”规则（可修复/不可修复分流）
- **影响**：
  - 测试输出噪声高，新增告警易被淹没
  - 影响回归判读效率与审查体验
- **改进行动（可执行）**：
  1. 先执行依赖升级到当前可用最新补丁版（减少已修复问题）
  2. 对仍无法消除的第三方告警，使用最小范围 `pytest filterwarnings` 白名单（精确到模块与类型）
  3. 明确规则：仅允许白名单第三方告警，项目自有代码告警不得屏蔽
- **验证方式（可复现）**：
  - 后端回归：`.venv/bin/pytest -q`（应无 warning 噪声）
  - 配置检查：`rg -n "\[tool.pytest.ini_options\]|filterwarnings|langchain_core|pymilvus" pyproject.toml`
- **升级（规则/清单/自动化）**：是（纳入 Testing 阶段门禁）
- **证据/关联**：`pyproject.toml` 告警白名单配置；`docs/v2.0/review_deployment.md` 追加记录

### 2026-02-07｜实现阶段未充分阅读文档导致返工（实现/流程）
- **标签**：实现、流程、文档、沟通
- **触发（事实）**：用户反馈多个实现问题：1)效能看板报404 2)系统清单有多余Tab 3)规则管理缺少快速设置 4)效能看板布局不对 5)菜单结构调整。AI在部分问题上质疑用户"文档有说吗"，但实际上proposal/requirements确实有相关描述。
- **根因**：
  1. **依赖搜索而非完整阅读**：用Grep搜索关键词（如"流程健康"）搜不到，就认为"文档没写"，但实际上proposal第188行明确写了
  2. **文档理解不系统**：proposal是高层概要，requirements是详细需求，两者存在gap。AI没有建立"proposal→requirements"的映射意识
  3. **质疑而非验证**：用户说"明明在proposal有提"时，AI的第一反应是"让我搜索"，而不是"让我完整阅读那一部分"
  4. **缺少TodoList追踪**：多问题并行时，没有系统化追踪完成状态，导致遗漏
- **影响**：
  - 增加返工次数，降低用户信任
  - 本可以一次性完成的工作变成多轮交互
  - 用户需要反复指出"文档里有写"
- **改进行动（可执行）**：
  1. **建立文档导航**：在proposal开头明确说明"本文档定位为高层概要，详细需求见requirements.md"，并建立问题→文档的索引映射
  2. **关键词变体搜索**：用户说"流程健康"时，搜"流程健康"OR"flow"OR"流程"；中英文都要搜
  3. **先读后做**：用户提到文档有写时，先Read相关章节，列出原文和行号，再讨论差异
  4. **TodoList强制**：任何多任务实现必须用TodoWrite工具建立任务列表，完成后勾选
  5. **差异确认流程**：发现proposal与requirements不一致时，列出两处原文请用户确认，不得自行假设
- **验证方式（可复现）**：
  - 文档阅读清单：实现前检查"是否已读proposal相关章节 + requirements对应REQ"
  - 差异核对：列出proposal.md行号 + requirements.md行号，请用户选择按哪个执行
  - TodoList完整性：所有用户提出的问题都列入TodoList，完成后全部勾选
- **升级（规则/清单/自动化）**：是（升级为R7/R8/R9）
- **证据/关联**：本次交互中关于效能看板、系统清单、规则管理、菜单结构的返工记录

### 2026-02-07｜指标与权限口径多处定义未同步（文档/指标/权限）
- **标签**：文档、指标、权限、追溯
- **触发（事实）**：Proposal/Requirements 在多个位置对同一口径描述不一致（如 PM修正率是否包含新增、任务分组口径不一致、Viewer 权限在“角色/权限”章节缺失），且 proposal 存在 Markdown 表格列数错误导致渲染异常。
- **根因**：
  1. 口径在“术语表/指标口径/场景说明/权限口径”多处重复出现，但缺少一致性自检门禁
  2. 修订只改了其中一处描述，未同步其他引用点
  3. 缺少基础的 Markdown 表格结构校验
- **影响**：
  - 文档可读性下降，容易误导实现与验收口径
  - 权限边界不清晰，存在越权/遗漏测试风险
- **改进行动（可执行）**：
  1. 指标/公式变更必须同步更新：术语表 + 6.3 指标口径 + 看板指标描述（出现一次定义，多处引用）
  2. 权限变更必须同步更新：角色定义 + 权限矩阵 + REQ-NF-004 + 相关 API 调用方/权限
  3. 文档提交前执行一致性自检：`rg -n "PM修正率|Viewer|待处理/进行中/已完成" docs/<版本号>/*.md`
- **验证方式（可复现）**：
  - 自检命令无“同词不同义”冲突，且 Markdown 表格渲染正常（至少在 GitHub/VSCode 预览确认）
- **升级（规则/清单/自动化）**：是（已补充 Quick Index `R10`）
