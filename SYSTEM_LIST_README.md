# 标准系统列表维护指南

## 概述

`system_list.md` 是业务需求评估系统中用于系统识别和匹配的核心配置文件。它定义了所有标准系统名称，用于智能识别需求文档中涉及的系统。

## 文件格式

### Markdown表格格式

使用 `|` 分隔的文本格式，每行包含4个字段：

```
系统名称 | 英文简称 | 系统状态 | 系统分类
```

**字段说明：**
- **系统名称**：标准化的中文名称（必需）
- **英文简称**：系统的英文缩写（可选，但建议填写）
- **系统状态**：运行中/实施中/待整合（可选）
- **系统分类**：所属的业务分类（可选，用于组织管理）

### 示例

```markdown
# 标准系统列表

> 本文件定义了所有标准系统名称，用于系统识别和匹配
> 格式：`系统名称 | 英文简称 | 系统状态 | 系统分类`
> 最后更新：2026-01-05

## 核心业务系统

新一代核心 | CBS | 运行中 | 核心系统
交易级总账 | TGL | 运行中 | 核心系统
综合信贷 | CCS | 运行中 | 核心系统

## 渠道系统

企业网银 | CIB | 运行中 | 渠道系统
新移动银行 | MBANK | 运行中 | 渠道系统
```

## 规范说明

### 命名规范

1. **系统名称**：使用完整的中文名称
   - ✅ 正确：`新一代核心`、`企业网银`
   - ❌ 错误：`核心系统`、`网银`

2. **英文名称**：使用大写缩写
   - ✅ 正确：`CBS`、`CIB`
   - ❌ 错误：`cbs`、`Cib`

3. **分类标签**：使用统一的分类体系
   - 核心系统、渠道系统、信贷系统、支付系统等

### 组织结构

- 使用 `##` 二级标题进行分组（可选，便于阅读）
- 系统按业务域或技术域分组
- 每个分组内按重要性排序

### 注释说明

- 使用 Markdown 标题 `#` 添加文件说明
- 使用引用块 `>` 添加元信息（格式说明、更新时间等）

## 使用方式

### 系统加载优先级

程序会按以下顺序尝试加载：

1. **system_list.md** （推荐，Markdown格式）
2. **system_list.csv** （向后兼容，CSV格式）

如果MD文件存在且格式正确，将优先使用MD文件。

### 向后兼容性

如果删除MD文件，系统会自动回退到CSV格式加载。两者可以共存，MD优先。

## 维护指南

### 添加新系统

1. 确定系统名称（参考现有的命名规范）
2. 确定系统分类
3. 在对应的分类分组下添加新行

```markdown
# 新增示例
新业务系统 | NBS | 运行中 | 业务系统
```

### 修改系统名称

如果要修改已有系统的名称：

1. 确保新名称符合命名规范
2. 修改时考虑历史数据的兼容性
3. 建议保留旧名称作为别名（在关键词映射中添加）

### 禁用/删除系统

直接删除对应行即可。如果需要保留记录，可以在行首添加 `#` 注释掉：

```markdown
# 已废弃系统 | OLD | 已停用 | 旧系统
```

## 系统匹配原理

程序使用以下逻辑进行系统匹配：

### 1. 精确匹配
需求中的系统名称与标准列表完全一致

### 2. 模糊匹配
需求中的名称包含标准名称，或标准名称包含需求中的名称

### 3. 关键词映射
预定义的关键词映射表（如：网联→统一支付、云闪付→统一支付）

### 4. 保留原值
如果无法匹配，保留需求中的原始系统名称并标记为"外部系统"

A: 可能的原因：

## 技术细节

### 加载代码位置

文件路径：`backend/agent/system_identification_agent.py`
方法名：`_load_system_list()`

### 解析规则

1. 跳过空行和以 `#` 或 `>` 开头的行
2. 跳过包含 "系统名称" 和 "英文简称" 的标题行
3. 解析包含 `|` 的数据行，提取第一个字段作为系统名称

### 性能考虑

- 文件仅在系统初始化时加载一次
- 建议系统数量不超过500个（性能影响可忽略）
- 使用字符串数组存储，匹配使用线性搜索（性能足够）
